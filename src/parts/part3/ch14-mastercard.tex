\section{Сетевой периметр Mastercard}\label{sec:mc-banknet}

Авторизация, клиринг и~расчёт разведены во времени и~в~требованиях
к~латентности (см.~\ref{ch:tx-lifecycle}), и~у Mastercard их обслуживают
три специализированные подсистемы внутри единого сетевого домена.
Switching Services описывают этот стек так: за авторизацию отвечает
Authorization Platform, за клиринг -- Global Clearing Management
System, GCMS, за расчёт -- Settlement Account Management,
SAM~\cite{mc-switching-services}. Для участника это означает один
сетевой адрес, но три разных интеграции -- запрос-ответ
с~миллисекундными SLA, пакетная клиринговая загрузка и~расчётные
окна по нетто-позициям; сертификация и~тестовые среды у~каждой
подсистемы свои~\cite{mc-rules}.

Failover внутри периметра асимметричен -- недоступность Authorization
Platform не перенаправляет трафик на клиринг и~наоборот, а~отказ
авторизации означает \textit{stand-in} или отказ, а~не переключение
на альтернативную сеть~\cite{mc-switching-services}.

\section{Модель dual message в Mastercard: почему IPM не равен TC33}\label{sec:mc-dmsa}

Общая логика dual message разобрана в~\ref{sec:tx-dual-single}, здесь --
Mastercard-специфика. Авторизационный диалог строится вокруг
MTI~0100/MTI~0110, но на клиринговом этапе эквайер формирует
для GCMS интегрированные продуктовые сообщения IPM~1240
(Integrated Product Messages), а~не TC33, как у Visa~\cite{mc-chargeback-guide}.

На практике это означает разные наборы полей в~авторизации и~в~клиринге.
DE25 (POS Condition Code) различается по допустимым значениям, DE48
в~авторизации несёт один набор подэлементов, а~в IPM расширяется
дополнительными subelement. Если инженер при отладке клирингового
отказа смотрит только в~авторизационный лог, реальная причина может
быть в~том, какие данные попали в~first presentment~\cite{mc-chargeback-guide}.

\importantnote{%
  Dual message у Visa и Mastercard похож по идее, но инженерная единица
  клиринга разная: у Visa BASE II/TC33, у Mastercard GCMS/IPM. Ошибка
  начинается, когда IPM трактуют как переименованный TC33; клиринговый
  конвейер Mastercard требует отдельной логики формирования и разбора
  ответов.
}

\section{IPM и GCMS}\label{sec:mc-ipm}

GCMS принимает и маршрутизирует данные клиринга в~формате IPM --
интегрированных продуктовых сообщениях для клиринга и связанных
с~ним операционных процессов~\cite{mc-switching-services}.

Два инварианта. Во-первых, IPM остаётся структурированным
сетевым форматом с~типами сообщений и нотацией элементов данных~\cite{mc-chargeback-guide}.
Во-вторых, подробная карта записей, подэлементов и служебных
заголовков вынесена в~специализированную документацию Mastercard.
Публичные материалы сети подтверждают формат IPM и~роль GCMS,
но не дают полного атласа полей для построчного разбора.

IPM-клиринговое сообщение -- это не авторизационный ISO 8583, и~его
first presentment использует MTI~1240 вместо привычного 0100.
IPM работает пакетно: получатель не отправляет ответ 1250 на каждую
запись, а принимает файл целиком и реагирует через отдельный процесс --
возвратный платёж без согласия продавца, MTI~1442, или акцепт
с~переходом к~расчёту. Набор элементов данных другой: DE25 несёт
POS Condition Code, DE48 -- расширенные данные со множеством
подэлементов, включая токен, дополнительные коды транзакции
и уже упомянутый Merchant Advice Code (DE48~SE84), DE63 -- сетевые
данные для маршрутизации и флагов транзакции внутри
GCMS~\cite{mc-chargeback-guide}. Разработчик, переносящий логику
разбора с~авторизации на клиринг, не может переиспользовать
авторизационный парсер -- структуры слишком разные.

До марта 2022~г., когда Mastercard обслуживала внутрироссийские
транзакции, клиринговая запись для покупки 5\,000~руб. через чиповый
терминал выглядела так. В~клиринговом файле эквайер формировал
запись MTI~1240, \textit{first presentment}, заголовок которой содержал
Member Message Text и процедурные поля, идентифицирующие участника
и цикл обработки.
Далее шли data elements -- DE~2 с~PAN, DE~4 с~суммой в~формате
\texttt{000000500000}, DE~49 с~кодом валюты \texttt{643},
DE~22 с~POS Entry Mode, указывающим на чип, DE~48 с~подэлементами
SE~22 Multi-Purpose Merchant Indicator, SE~37 Additional Merchant
Data и SE~84 Merchant Advice Code, если авторизация была
с~рекомендацией по повторам. DE~63 нёс Network Data --
флаги маршрутизации внутри GCMS. Отдельно появлялись
записи комиссий: interchange fee в~валюте расчёта
и assessment fee сети~\cite{mc-chargeback-guide}.

Если эмитент оспаривает транзакцию, он создаёт \textit{first chargeback}
IPM~1442, ссылающийся на тот же DE~63 Network Data
и Acquirer Reference Number. Эквайер видит возвратный платёж
в~следующем клиринговом файле; для \textit{second presentment}
он формирует IPM~1240 с~соответствующим function code 205 или 282
и прикладывает доказательства через отдельный протокол Mastercard
Connect~\cite{mc-chargeback-guide}.

Код рекомендации для ТСП (Merchant Advice Code, MAC) -- совет, который
сеть передаёт в~авторизационном ответе в DE~48, subelement 84
для поддерживаемых сценариев~\cite{mc-tpr}. По нему продавец или шлюз
решает, повторять ли попытку, обновлять ли реквизиты или прекращать
автоматические списания. Без MAC логика повторов слепа.

На диагностике различаются две плоскости: IPM и GCMS отвечают
за клиринговый обмен и последующую обработку, а рекомендации
по повторам приходят раньше, в~авторизационном ответе. Дальше сеть
использует результаты клиринга для расчёта нетто-позиций и~передачи
их в~расчётный процесс; конкретные окна и~порядок урегулирования зависят
от правил региона и~роли участника~\cite{mc-rules}.

\section{Собственные реализации: M/Chip и MDES}\label{sec:mc-proprietary}

Над общим уровнем стандартов EMVCo Mastercard задаёт два
семейства собственных требований: M/Chip покрывает чиповое
приложение на карте и терминале, а~MDES -- токенизацию
и удалённую оплату.

Общего знания EMV (глава~\ref{ch:emv}) недостаточно, когда устройство,
карта или процессинг должны поддерживать именно Mastercard.
Семейство требований M/Chip~\cite{mc-mchip} поверх стандарта EMV
задаёт собственные требования к~поведению приложения и сертификации,
причём в~двух профилях -- контактном и бесконтактном M/Chip Contactless,
разобранном в~главе~\ref{ch:contactless}. Каждый профиль добавляет
свой набор тегов, правил офлайн-поведения и тестовых сценариев.
Утверждение \enquote{терминал уже поддерживает EMV, значит Mastercard
заработает автоматически} слишком оптимистично -- нужны
специфичные для схемы требования и их проверка. Точную и актуальную
матрицу профилей M/Chip сверяют через Mastercard Connect, в~открытом
доступе полной матрицы нет. Типичная ошибка при сертификации --
терминал проходит EMVCo Level~2, но не проходит тесты M/Chip
из-за расхождения в~обработке конкретных тегов или
CVM-логике~\cite{mc-mchip}.

Общая модель сетевого токена и~MDES как Token Service Provider --
в~главе~\ref{ch:tokenization}. Mastercard-специфика в~MDES -- встроенный
механизм защищённой удалённой оплаты DSRP, Digital Secure Remote
Payments, для приёма интернет-платежей и~платежей в~приложениях
с~опорой на EMV-криптографию~\cite{mc-tpr}. DSRP формирует криптограмму
для каждой CNP-операции и~даёт эмитенту более сильный сигнал
подлинности, чем обычный сценарий сохранённых
реквизитов~\cite{mc-mdes-token-connect}. Какой именно набор криптограмм
и полей пришлёт клиентский кошелёк, зависит от его реализации;
Mastercard публично не фиксирует единый профиль для всех Token
Requestors.

\section{Mastercard Send}\label{sec:mc-send}

Mastercard Send -- выплаты и~переводы, где средства
должны уйти получателю на карту, банковский счёт, цифровой кошелёк
или другой канал, который поддерживает конкретная программа~\cite{mc-send-aptpay}.

У сервиса два продуктовых направления. Disbursements -- выплаты
физлицам от бизнеса: зарплаты, страховые возмещения, выплаты
продавцам маркетплейса и гиг-платформам, а~P2P-переводы идут между
физическими лицами в~режимах domestic и cross-border. В~карточном
сценарии платформа сначала инициирует финансирующую Funding
Transaction для резервирования средств отправителя, а~затем
запускает Payment Transaction в~пользу получателя~\cite{mc-tpr}.

Технически Mastercard Send требует от участника поддержки
Send Transaction в~поле DE48. Авторизационный запрос на выплату
несёт в~DE3 Transaction Processing Code, отличный от обычной
покупки, -- транзакция направляется получателю, а~не списывается с~него.
Процессинговый парсер, настроенный только на покупки и~рефанды,
транзакцию \textit{Send} не обработает -- нужна отдельная ветка логики.

Сравнительный аналог у Visa -- Visa Direct, архитектурно схожая
push-модель с~той же парой Funding/Payment Transaction, но с~разными
полями и процедурами участника. Отличия видны в~funding source
и~в~нормах finality: ряд коридоров Mastercard Send доставляет
выплаты near real-time, за секунды или минуты, тогда как фактическая
доступность средств у~получателя зависит от банка-получателя.

Обещание \enquote{почти мгновенной выплаты} описывает возможность
сети, но не заменяет проверку конкретного платёжного коридора, банка
или провайдера-получателя и режима их работы~\cite{mc-send-aptpay}.
SLA под каждый рынок сверяется отдельно, даже если бренд сервиса
один и~тот же.

\practicenote{%
  При построении платформы выплат через Mastercard Send заранее
  проверяйте наличие у~сети нужного платёжного коридора и готовность
  банка или провайдера-получателя принимать такие переводы в~нужном режиме.
  Режим near real-time в~маркетинговом описании сервиса и фактическая
  доступность средств получателю не одно и~то же. Для платформ
  выплат это различие существенно -- именно на нём ломаются обещания
  \enquote{деньги за минуту}.

}
\section{ABU: автоматическое обновление данных карты}\label{sec:mc-abu}

Общая модель обновления реквизитов после перевыпуска карты, оба
режима -- batch и~real-time API, -- сетевые токены как альтернатива
и~аналог Visa Account Updater разобраны в~главе~\ref{ch:subscriptions}
(\S\ref{sec:subscriptions-account-updater}). Mastercard-специфика
ABU -- в~ограничениях охвата. Сервис покрывает не все типы карт,
предоплаченные и~часть региональных продуктов могут оставаться
вне обновления, а~участие эмитента в ABU обязательным
не является~\cite{mc-abu,mc-tpr}.

\section{Interchange Mastercard}\label{sec:mc-interchange}

Interchange у Mastercard, как и~у Visa, определяется сочетанием региона,
продукта, MCC, канала приёма и квалифицирующих условий транзакции~\cite{mc-interchange-us}.

Логика квалификации трёхступенчатая. Сначала выбирается региональная
таблица, потом тип продукта и категория ТСП, затем проверяются
признаки, от которых зависит квалификация транзакции: какие данные
переданы, была ли операция электронной, использовались ли корректные
карточные или токенные реквизиты. Здесь возникают понижения
категории, downgrade, и связанные с~ними споры между продавцом,
эквайером и процессором.

Downgrade возникает, когда транзакция не набирает достаточно
квалифицирующих признаков для целевой категории. Типичные причины --
отсутствие electronic commerce indicator, неполные данные в DE48,
пропущенный Address Verification Service или позднее предъявление
(\textit{late presentment}) в~клиринг. Каждый пропущенный признак сдвигает транзакцию на ступень
ниже, увеличивая interchange для эквайера.

Для аудита downgrade нужен доступ к~клиринговым данным -- в IPM-файле
видно, какую категорию присвоила сеть и~какие признаки отсутствовали.
Авторизационный лог для этого недостаточен, квалификация происходит
на этапе клиринга~\cite{mc-interchange-us}.

\begin{figure}[tbp]
  \centering
  \includefiguresvg[width=.8\linewidth]{assets/figures/ch14-mastercard/interchange-qualification}
  \caption{Дерево квалификации interchange: от региона до даунгрейда.}\label{fig:interchange-qualification}
\end{figure}

Без привязки к~конкретному рынку, дате публикации и публичной
региональной таблице нельзя надёжно утверждать, что тот или иной
премиальный продукт Mastercard \emph{всегда} дороже сопоставимого
продукта Visa. Такие сравнения быстро превращаются в~смесь локальных
правил, частных договорённостей и устаревших тарифных редакций.
Для рынка США Mastercard публикует отдельные таблицы ставок
и критериев interchange; любой численный пример привязывается
именно к~такой конкретной региональной
таблице~\cite{mc-interchange-us}.

\practicenote{%
  При сравнении interchange Mastercard и Visa для конкретного
  ТСП определяющим параметром остаётся фактический микс продуктов и
  качество данных в~транзакции.
  Диагностику лучше вести в~таком порядке. Сначала выбрать
  региональную таблицу, определить продукт карты, проверить MCC
  и~затем сверить, выполнила ли транзакция квалифицирующие условия.
  Если спорить со ставкой раньше этих четырёх шагов, разговор почти
  всегда уходит в~догадки.

}

Существеннее всего различия в~клиринговой модели и~формате обмена.
Разработчик, переносящий систему обработки клиринга с~одной схемы
на другую, не может переименовать поля: структуры IPM и TC33
различаются настолько, что требуют отдельного парсинга и отдельной
операционной диагностики. Различия в~токенизации важны для
разработчиков кошельков и интернет-платёжных интеграций. Различия
в ABU и VAU важны для платёжных платформ с~рекуррентными платежами.

Есть и~различия другого рода, когда одни и~те же отраслевые слова
у~схем обозначают не совсем одно и~то же. Термин \enquote{dual
message} применяют и к Visa, и~к Mastercard, но за ним стоят разные
технические реализации, и~нейтральная формула \enquote{авторизация
и клиринг разделены во времени} верна для обеих схем -- только
детали этого разделения различаются.

\practicenote{%
  При работе с IPM-сообщениями Mastercard для диагностики отказов
  нужно смотреть на сам факт отказа и~на Merchant Advice Code
  в~авторизационном ответе -- именно он подсказывает, допустимы
  ли повторы и~нужно ли обновлять сохранённые реквизиты.
  IPM нужен для клиринга и последующей обработки, а оперативная
  логика повторов определяется ещё на авторизационном шаге~\cite{mc-tpr}.
