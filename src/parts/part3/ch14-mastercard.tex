\chapter{Mastercard}

\begin{kaobox}[title=Что вы узнаете из этой главы]
\scheme{Mastercard} часто воспринимается как~зеркало \scheme{Visa}:
та~же четырёхсторонняя модель, тот~же стандарт EMV, те~же
interchange-таблицы. В~этой главе мы~разберём, где~сходство

структурой subelements. M/Chip~--- собственная EMV-реализация.
Account Based Updater~--- автоматическое обновление данных перевыпущенных
карт. И~таблица в~конце~--- восемь параметров, по~которым Visa
и~Mastercard действительно различаются.
\end{kaobox}


\section{Banknet: глобальная авторизационная сеть}
\label{sec:mc-banknet}

Если~VisaNet состоит из~двух отдельных сетей (BASE~I для~авторизации
и~BASE~II для~клиринга), то~\scheme{Mastercard} с~самого начала строила
единую инфраструктуру~--- Banknet. Это~не~означает, что~авторизация
и~клиринг обрабатываются одинаково; это~означает, что~обе задачи
решаются в~рамках одной сетевой архитектуры с~общими узлами маршрутизации~\sidecite{mc-rules}.

Топология Banknet строится на~региональных процессинговых центрах,
объединённых глобальным кольцом. Каждый региональный хаб принимает
транзакции от~участников своего региона и~маршрутизирует запрос либо
к~эмитенту внутри региона, либо через глобальное кольцо в~другой
регион. Двойное кольцо обеспечивает отказоустойчивость: отказ одного
сегмента вызывает автоматическое переключение на~резервный путь
за~десятки миллисекунд. SLA авторизации на~стороне сети~--- менее
130~мс от~эквайера до~эмитента и~обратно~\sidecite{mc-rules}.

Важный практический момент: в~отличие от~VisaNet, где~клиринговый
файл TC33 передаётся по~отдельному каналу BASE~II, в~Banknet
клиринговые сообщения (IPM-файлы) маршрутизируются через ту~же
инфраструктуру. Это~упрощает операционную модель для~банков-участников:
одно сетевое подключение к~Banknet покрывает оба потока.


\section{DMSA: почему клиринг Mastercard устроен иначе}
\label{sec:mc-dmsa}

Многие, впервые знакомясь с~операционными деталями \scheme{Mastercard},
обнаруживают удивительную вещь: у~\scheme{Mastercard} нет отдельного
клирингового сообщения, аналогичного TC33 у~\scheme{Visa}. Вместо него
используется схема DMSA~--- Dual Message System for Authorization~---
где~второе сообщение является частью файла IPM (Interchange Process
Message), а~не~отдельным протокольным обменом~\sidecite{mc-rules}.

Разберём детали. При~транзакции по~\scheme{Mastercard} первое сообщение~---
авторизация, \mti{0100} от~эквайера и~\mti{0110} в~ответ. Это~стандартный
диалог, аналогичный \scheme{Visa}. После завершения транзакции
эквайер помещает её~запись в~IPM-файл и~передаёт его~в~GCMS (Global
Clearing Management System). Ключевой нюанс в~том, что~IPM-запись
\emph{встроена} в~клиринговый файл: это~не~отдельное транзакционное
сообщение по~протоколу \iso{8583}, а~структурированная запись
в~пакетном файле~\sidecite{mc-rules}.

\begin{important}
DMSA не~равнозначна dual message \scheme{Visa}~--- это~принципиально
разные flow при~клиринге. Visa Base~II: эквайер отправляет TC33-файл
отдельным процессом после авторизации. Mastercard GCMS: эквайер
помещает транзакцию в~IPM-файл, который является единицей
обмена с~системой клиринга. Путаница между этими моделями~---
один из~наиболее частых источников ошибок при~интеграции систем,
поддерживающих обе схемы.
\end{important}

С~точки зрения разработчика это~означает, что~система обработки
транзакций \scheme{Mastercard} должна уметь формировать IPM-файлы
согласно спецификации~--- со~своими заголовками, структурой записей
и~полями, которые не~совпадают со~структурой TC33. Попытка использовать
TC33-логику для~генерации IPM-файлов \scheme{Mastercard}~--- верный
путь к~проблемам при~сверке~\sidecite{mc-rules}.

Ссылка на~общую концепцию dual message системы, где~авторизация
и~клиринг разделены во~времени~--- глава~\ref{ch:tx-lifecycle}.


\section{IPM-файлы и GCMS}
\label{sec:mc-ipm}

IPM (Interchange Process Message)~--- формат клирингового файла
\scheme{Mastercard}, основанный на~расширенной версии \iso{8583}.
GCMS (Global Clearing Management System)~--- система приёма и~маршрутизации
этих файлов на~стороне \scheme{Mastercard}. Понимание структуры IPM
необходимо любому, кто~строит или~отлаживает клиринговый pipeline
для~\scheme{Mastercard}~\sidecite{mc-rules}.

Каждый IPM-файл начинается с~заголовка, содержащего идентификатор
файла, дату, номер цикла клиринга и~идентификаторы участников. Тело
файла~--- последовательность записей (records), каждая из~которых
соответствует одной транзакции. Структура записи определяется bitmap-ом
как~в~\iso{8583}: активные биты указывают, какие поля данных
(DE, Data Element) присутствуют в~данной записи~\sidecite{mc-rules}.

Для~отладки клиринга наиболее критичны несколько полей. \de{25}~---
Point of Service Condition Code: характеризует условия транзакции
(карта присутствует, карта отсутствует, автоматическое устройство и~т.\,д.).
\de{48}~--- Additional Data: поле переменной длины с~subelements, где
\scheme{Mastercard} размещает дополнительную информацию, не~укладывающуюся
в~стандартную структуру \iso{8583}.

% \label{fig:mc-ipm-structure}

DE~48 subelement~26 (Merchant Advice Code, MAC)~--- особенно важен
для~работы с~decline. Этот sublement передаёт эмитент в~ответе
и~содержит рекомендацию мерчанту: как~именно следует поступить
при~данном отказе. MAC~01 означает \enquote{новую карту обновите через
ABU}; MAC~03~--- \enquote{не~повторяйте транзакцию}. Корректная
интерпретация MAC критична для~построения логики повторных попыток~---
тема, которую мы~подробнее рассмотрим в~главе о~decline management.

Когда GCMS получает IPM-файл, он~проверяет его~структуру, разбирает
записи по~эмитентам и~рассчитывает нетто-позиции для~урегулирования.
Settlement происходит через банк урегулирования \scheme{Mastercard}~---
как~правило, на~следующий рабочий день после клирингового цикла.


\section{MDES и токенизация}
\label{sec:mc-mdes}

Mastercard Digital Enablement Service (MDES)~--- это~реализация \scheme{Mastercard}
стандарта EMVCo Payment Tokenisation~\sidecite{mc-mdes}. Основная
архитектура та~же, что~у~Visa Token Service, описанного
в~главе~\ref{ch:tokenization}: FPAN заменяется DPAN, MDES хранит
мэппинг и~подставляет FPAN при~авторизации.

Специфичным для~MDES является DSRP (Digital Secure Remote Payments)~---
технология, позволяющая браузерным и~in-app платежам использовать
EMV-криптограммы вместо статических карточных данных. При~DSRP
мобильное приложение генерирует ARQC (Application Request Cryptogram)
точно так~же, как~это~делает чип на~физической карте при~контактной
транзакции. В~результате card-not-present транзакция приобретает
уровень защиты, сравнимый с~chip-and-PIN: эмитент проверяет криптограмму
и~не~принимает её~повторное использование. DSRP~--- ключевое отличие
MDES от~классической токенизации на~уровне хранения PAN~\sidecite{mc-mdes}.


\section{M/Chip: семейство EMV-реализаций}
\label{sec:mc-mchip}

Общий стандарт EMV~--- тема главы~\ref{ch:emv}. Здесь~--- о~том,
что~является специфичным для~\scheme{Mastercard}. Семейство M/Chip включает
несколько профилей, рассчитанных на~разные сценарии использования~\sidecite{mc-mchip}.

\textbf{M/Chip Advance}~--- основной профиль для~контактных транзакций:
поддерживает полный набор CVM (online PIN, offline PIN, signature,
no-CVM), Dynamic Data Authentication (DDA), Combined DDA/Application
Cryptogram (CDA). Именно этот профиль используется в~большинстве
кредитных и~дебетовых карт \scheme{Mastercard} в~Европе и~на~многих
других рынках~\sidecite{mc-mchip}.

\textbf{M/Chip Select}~--- профиль для~рынков с~особыми требованиями
к~гибкости: поддерживает несколько приложений на~одной карте и~позволяет
держателю выбрать приложение при~оплате. Используется в~кобейджинговых
картах, где~на~одном чипе могут одновременно присутствовать приложения
\scheme{Mastercard} и~национальной платёжной системы.

\textbf{M/Chip Lite}~--- упрощённый профиль для~ограниченных сред:
менее ресурсоёмкий чип, упрощённый набор CVM (как~правило,
no-CVM или~только online PIN). Исторически использовался в~предоплаченных
картах и~картах лоялти.

Специфика \scheme{Mastercard} в~бесконтактных CVM~--- наличие поддержки
DCVV (Dynamic Card Verification Value). При~бесконтактной транзакции
чип генерирует DCVV~--- динамическое значение, меняющееся при~каждой
транзакции. В~отличие от~статического CVV2 на~обратной стороне карты,
скомпрометированный DCVV бесполезен: он~привязан к~конкретной транзакции~\sidecite{mc-mchip}.


\section{Mastercard Send}
\label{sec:mc-send}

Mastercard Send~--- программа push-выплат, аналогичная Visa Direct
по~назначению, но~с~собственной технической реализацией. Отправитель
инициирует Account Funding Transaction (AFT): средства списываются
с~его~счёта или~карты. Затем система \scheme{Mastercard} направляет
Payment Transaction (PT) на~карту или~счёт получателя. Эмитент получателя
зачисляет деньги; типичное время~--- несколько минут~\sidecite{mc-rules}.

Ключевые отличия от~Visa Direct проявляются в~деталях. Mastercard
Send поддерживает большее число funding sources: помимо карт, отправитель
может финансировать перевод напрямую с~банковского счёта через ACH-линку.
Доступность в~реальном времени зависит от~готовности банка-эмитента
получателя~--- не~все эмитенты обрабатывают PT в~режиме реального
времени круглосуточно; некоторые делают это~только в~рабочие часы.


\section{ABU: автоматическое обновление данных карты}
\label{sec:mc-abu}

Account Based Updater (ABU)~--- сервис \scheme{Mastercard}, который решает
одну из~наиболее болезненных проблем для~мерчантов с~рекуррентными
платежами: карту клиента перевыпустили (изменился номер или~срок
действия), и~все запланированные списания начинают падать с~кодом
decline~\sidecite{mc-rules}.

ABU работает следующим образом. Эмитент при~перевыпуске карты сообщает
ABU о~новых данных. Эквайер (или~платёжный шлюз) с~определённой
периодичностью запрашивает ABU, предоставляя список PAN, которые
хранит для~рекуррентных платежей. ABU возвращает обновлённые данные~---
новый PAN, новый срок действия или~статус (карта закрыта, счёт закрыт).
Эквайер обновляет свои данные и~продолжает списания уже по~актуальным
реквизитам~\sidecite{mc-rules}.

Аналог у~\scheme{Visa}~--- Visa Account Updater (VAU): архитектурно
схожий сервис с~теми~же целями. Принципиальное различие~--- техническое:
ABU и~VAU имеют разные API, разные форматы запросов и~разные регуляторные
требования к~участию. Эквайер, поддерживающий рекуррентные платежи
на~обеих сетях, обязан поддерживать оба сервиса. Тема связи ABU
с~управлением подписками~--- в~главе~\ref{ch:subscriptions} (Часть~IV).


\section{Interchange Mastercard}
\label{sec:mc-interchange}

Структура interchange \scheme{Mastercard} в~целом параллельна \scheme{Visa}:
программы определяются сочетанием типа карты, MCC и~канала транзакции.
Отдельная публикация~--- Mastercard Interchange Rates~--- обновляется
дважды в~год синхронно с~Mastercard Rules~\sidecite{mc-rules}.

Принципиальное различие~--- в~программах для~карт с~расширенными
преимуществами. \scheme{Mastercard} World и~World Elite~--- категории
premium-карт с~повышенными ставками interchange, отражающими более высокие
программы rewards для~держателей. Мерчант, принимающий большую долю
таких карт (туристические, бизнес-расходы), обнаружит, что~его~effective
interchange rate выше среднего по~рынку~--- именно из-за~структуры
программ rewards~\sidecite{mc-rules}.


\section{Архитектурные отличия от Visa}
\label{sec:mc-vs-visa}

Подведём сравнение. Многие различия между \scheme{Mastercard} и~\scheme{Visa}~---
терминологические или~исторические; но~несколько из~них существенны
с~инженерной точки зрения.

Наиболее существенны различия в~клиринговой модели и~форматах файлов.
Разработчик, переносящий систему обработки клиринга с~одной схемы
на~другую, не~может просто переименовать поля~--- структура IPM
и~TC33 достаточно различны, чтобы потребовать переработки парсинга
и~маппинга. Различия в~токенизации (DSRP vs~VTS domain restriction)
важны для~разработчиков мобильных кошельков. Различия в~ABU/VAU
важны для~платёжных платформ с~рекуррентными платежами.

Там, где~различия терминологические: оба термина \enquote{dual message}
применяются к~обеим схемам, но~описывают не~совпадающие детали
реализации~--- что~регулярно вызывает путаницу в~технической
документации. Нейтральное описание \enquote{авторизация и~клиринг
разделены во~времени}~--- правда для~обеих схем; технические детали
разделения~--- разные.

\begin{practice}
При~работе с~IPM-файлами \scheme{Mastercard} для~диагностики отказов
первым делом стоит смотреть на~DE~48 sublement~26 (Merchant Advice Code):
именно там эмитент сообщает \emph{почему} отклонён платёж
и~что~делать дальше. MAC~01 означает: обновите данные карты через ABU.
MAC~03 означает: не~повторяйте~--- карта или~счёт закрыты. Игнорирование
MAC и~слепые повторные попытки не~только бесполезны, но~и~нарушают
правила \scheme{Mastercard} по~управлению повторными транзакциями.
\end{practice}


\section*{Итоги главы}

\begin{itemize}
  \item Banknet~--- единая сеть, в~отличие от~раздельных BASE~I/BASE~II
        у~Visa; SLA авторизации~--- менее 130~мс.
  \item DMSA~--- клиринговые данные встроены в~IPM-файл, а~не~передаются
        отдельным транзакционным сообщением; путать с~dual message
        Visa нельзя.
  \item IPM-файлы обрабатываются GCMS; DE~48 с~subelements (в~частности,
        sublement~26~--- Merchant Advice Code) критичен для~диагностики
        decline и~логики повторных попыток.
  \item MDES добавляет к~стандарту EMVCo токенизации технологию
        DSRP: card-not-present транзакции с~EMV-криптограммой.
  \item M/Chip Advance/Select/Lite~--- три профиля под~разные сценарии;
        DCVV в~бесконтактных транзакциях обеспечивает динамическую
        верификацию.
  \item ABU автоматически обновляет данные перевыпущенных карт
        для~рекуррентных платежей; аналог Visa~--- VAU; оба необходимы
        для~платформ с~подписочной моделью.
  \item Ключевые архитектурные различия с~Visa~--- клиринговая модель
        (IPM vs~TC33) и~токенизация (DSRP vs~VTS domain restriction);
        остальное~--- в~значительной мере терминология.
\end{itemize}
