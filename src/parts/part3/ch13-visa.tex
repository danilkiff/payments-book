\section{VisaNet: двухсетевая архитектура}
\label{sec:visa-visanet}

Когда говорят \enquote{платёж прошёл через Visa}, за~этим скрываются два
совершенно разных события, которые происходят в~разное время и~по~разным
каналам. Первое~--- авторизация: терминал эквайера отправляет запрос
эмитенту и~в~течение секунды получает ответ. Второе~--- клиринг:
через несколько часов или~на~следующий день эквайер передаёт данные
завершённой транзакции, чтобы инициировать фактическое движение
денег~\sidecite{visa-core-rules}.

Это~разделение по~времени и~требованиям к~надёжности определяет
двухсетевую архитектуру VisaNet. \textbf{BASE~I} (Base Settlement
Interchange Network~I) обслуживает авторизацию: это~сеть реального
времени с~жёсткими SLA, рассчитанная на~обработку тысяч авторизационных
запросов в~секунду с~p99-латентностью менее~1~с до~эмитента и~обратно.
BASE~I работает по~собственному протоколу поверх двойного кольца
частных сетей с~переключением на~резервный канал за~миллисекунды.
\textbf{BASE~II} (Base Settlement Interchange Network~II) обслуживает
клиринг: это~пакетная сеть, по~которой эквайеры передают файлы TC33
с~записями завершённых транзакций. BASE~II не~требует ответа за~секунду~---
она должна гарантированно обработать файл в~заданное расчётное окно,
обычно к~следующему рабочему дню~\sidecite{visa-core-rules}.

Зачем разделять? Потому что требования к~двум задачам диаметрально
противоположны. Авторизация~--- онлайн-диалог с~жёстким таймаутом:
если ответ не~пришёл за~установленное время, терминал вынужден
принять решение (approve или decline) самостоятельно или~отклонить
транзакцию. Клиринг~--- пакетная передача данных, где~важна не~скорость
каждого сообщения, а~целостность и~верность пересылки всего файла.
Объединить эти задачи в~одной сети означало бы либо тормозить
авторизацию тяжёлыми пакетными операциями, либо завышать требования
к~пропускной способности клиринговой инфраструктуры. Архитектурное
разделение позволяет масштабировать каждую часть независимо.

Для~разработчика платёжного шлюза практическое следствие
таково: авторизационный запрос уходит по~ISO~8583 (или~его~Visa-вариации)
в~режиме реального времени, а~клиринговый файл TC33 отправляется
отдельным процессом по~расписанию. Сбой в~одном канале не~останавливает
другой: если BASE~II временно недоступна, авторизации продолжаются;
если BASE~I загружена, клиринговый pipeline не~пострадает.


\section{Core Rules: как читать публичный документ}
\label{sec:visa-core-rules}

\scheme{Visa} публикует свой основной регуляторный документ~--- Visa Core
Rules and Visa Product and Service Rules~--- в~виде публично доступного PDF.
Это~редкий случай в~платёжной индустрии: большинство операционных
деталей карточных сетей существуют только под~NDA. Однако публичность
не~означает простоту: документ насчитывает сотни страниц
и~требует понимания его~внутренней логики, чтобы не~утонуть
в~перекрёстных ссылках~\sidecite{visa-core-rules}.

Документ организован в~три слоя. \textbf{Product \& Service Rules}
описывают условия участия в~каждом продукте Visa: какие карты можно
эмитировать, какие терминалы принимать, кто несёт ответственность
в~спорных ситуациях. \textbf{Technical Standards} задают технические
требования~--- форматы сообщений, требования к~криптограммам,
параметры EMV-ядра. \textbf{Operating Regulations} регулируют
операционные процедуры: сроки клиринга, порядок диспутов, тарифы.
Большинство операционных требований к~эквайерам и~эмитентам живёт
именно в~этих трёх слоях~\sidecite{visa-core-rules}.

Что~остаётся под NDA? Прежде всего, членские соглашения с~конкретными
банками, специальные программы и~кастомизированные ставки interchange.
Публичная редакция Core Rules~--- базовый уровень; реальные коммерческие
условия конкретного банка-участника содержатся в~приложениях к~его~членскому
соглашению с~\scheme{Visa}.

\scheme{Visa} обновляет правила дважды в~год~--- в~апреле и~в~октябре.
Каждый цикл сопровождается \enquote{bulletins}: короткими официальными
уведомлениями об~изменениях, которые рассылаются участникам заблаговременно.
Апрельские изменения обычно вступают в~силу 14~апреля, октябрьские~---
в~середине октября. Эквайерам и~процессорам важно следить именно
за~bulletins: принципиальные изменения (новые требования к~сертификации,
изменение interchange-программ, новые fraud rules) анонсируются там
за~6--12~месяцев~\sidecite{visa-core-rules}.

\begin{practice}
Чтобы не~пропустить критические изменения, удобно завести календарную
пометку на~январь и~июль~--- именно тогда Visa обычно публикует
черновики апрельских и~октябрьских bulletins. При~работе с~Core Rules
стоит всегда начинать с~определения раздела: нас~интересует эквайерная
тема или~эмитентная? Раздел «Acquirer» описывает обязательства эквайера;
«Issuer»~--- эмитента. Большинство разногласий между банками возникает
именно потому, что одна сторона читает только \enquote{свой} раздел,
не~замечая зеркальных обязательств контрагента.
\end{practice}


\section{TADG и сертификация терминалов}
\label{sec:visa-tadg}

Терминал, принимающий карты \scheme{Visa}, проходит сертификацию
в~три этапа, каждый из~которых проверяет отдельный уровень стека.
Понимание этих уровней важно не~только производителям терминалов,
но~и~эквайерам: именно эквайер несёт ответственность перед \scheme{Visa}
за~то, что~в~его~сети работают только сертифицированные устройства.

\textbf{Уровень~1 (L1)}~--- физический и~электрический: терминал
должен соответствовать требованиям к~считывателю чипа (EMV contact),
бесконтактному интерфейсу (ISO/IEC~14443) и~считывателю магнитной полосы.
L1-сертификацию проводит EMVCo или~её~аккредитованные лаборатории.
По~сути, это~проверка железа: корректность считывания данных карты
на~физическом уровне~\sidecite{visa-tadg}.

\textbf{Уровень~2 (L2)}~--- программный, EMV-ядро: терминал должен
правильно реализовывать логику EMV-транзакции (выбор приложения,
чтение данных, обработку CVR/TVR, генерацию криптограмм). Для~Visa
это~означает соответствие спецификации Visa Contactless Payment Specification
(VCPS) и~требованиям TADG к~поведению ядра. L2-сертификацию
выдаёт \scheme{Visa} или~её~уполномоченные лаборатории. Именно на~этом
уровне проверяется, правильно~ли терминал общается
с~картой~\sidecite{visa-tadg}.

\textbf{Уровень~3 (L3)}~--- интеграционный: проверяет связку
EMV-ядра с~конкретной системой управления терминалом (TMS) эквайера
и~его~хостом авторизации. Эквайер или~процессор проводит L3-тестирование,
часто совместно с~поставщиком терминала, и~представляет результаты
Visa для~одобрения. Именно здесь выясняется, правильно~ли терминал
формирует авторизационное сообщение, корректно~ли передаёт \de{55}
с~EMV-данными~\sidecite{visa-tadg}.

Сертификация важна эквайеру по~коммерческой причине: правила \scheme{Visa}
перекладывают ответственность (liability) за~мошенническую транзакцию
на~ту~сторону, которая обеспечила более низкий уровень защиты.
Эквайер с~несертифицированным терминалом или~терминалом без~поддержки
чипа автоматически несёт ответственность за~fraud при~использовании
EMV-карты~--- даже если эмитент одобрил транзакцию. Это~именно тот~механизм
liability shift, с~которым читатель встречался в~главе~\ref{ch:emv}.

\begin{termdef}{TADG}
TADG (Terminal and Application Design Guide)~--- технический документ
\scheme{Visa}, описывающий требования к~пользовательскому интерфейсу
терминала, порядку выбора CVM (Cardholder Verification Method)
и~поведению при~различных статусах транзакции. TADG дополняет
технические спецификации EMVCo требованиями, специфичными для~сети Visa.
\end{termdef}
\sidecite{visa-tadg}


\section{Visa Token Service}
\label{sec:visa-vts}

Visa Token Service (VTS) является реализацией \scheme{Visa} стандарта
EMVCo Payment Tokenisation, который мы~подробно разбирали
в~главе~\ref{ch:tokenization}. Напомним основную идею: вместо
настоящего PAN (Funding PAN, FPAN) в~транзакции используется
суррогатный токен (Digital PAN, DPAN), и~только Visa Token
Service умеет заменить DPAN обратно на~FPAN перед отправкой
запроса эмитенту~\sidecite{visa-vts}.

Специфичным для~VTS является понятие Token Assurance Level (TAL).
TAL~--- это~числовой показатель уверенности системы в~том, что~токен
используется законным держателем карты. Он~зависит от~метода
верификации, применённого при~привязке карты к~кошельку (для~Apple Pay,
Google Pay и~т.\,д.): биометрия владельца устройства даёт более высокий
TAL, чем~простое SMS-OTP. Эмитент получает TAL в~авторизационном запросе
и~может использовать его~при~принятии решения~\sidecite{visa-vts}.

Domain restriction~--- второй специфичный механизм VTS: токен
привязывается к~конкретной области применения (домену), например
к~определённому устройству или~мерчанту. VTS проверяет при~авторизации,
что~транзакция поступает именно из~разрешённого домена, и~отклоняет
попытки использовать токен за~пределами своего домена. Именно этот
механизм делает токен бесполезным при~компрометации: даже если
злоумышленник получит DPAN, он~не~сможет совершить транзакцию
с~другого устройства или~у~другого мерчанта~\sidecite{visa-vts}.


\section{Visa Direct: кредит на~карту в~реальном времени}
\label{sec:visa-direct}

Стандартная транзакция по~карте движется по~модели \enquote{дебет
у~плательщика}: авторизация резервирует деньги на~счёте держателя
карты, клиринг списывает их~и~переводит мерчанту. Visa Direct работает
в~обратной модели~--- деньги движутся \emph{к}~держателю карты,
а~не~от~него. Технически это~называется OCT (Original Credit Transaction):
запрос с~кредитом на~счёт получателя, обозначенный
как~\mti{0200}~с~функциональным кодом, указывающим на~кредитовую
операцию~\sidecite{visa-core-rules}.

Практические сценарии Visa Direct разнообразны. Платформы
гиг-экономики (Uber, Lyft и~их~аналоги) используют её~для~мгновенных
выплат водителям~--- деньги поступают на~карту через несколько минут
после окончания поездки, а~не~через банковский рабочий день.
Страховые компании выплачивают небольшие компенсации без~оформления
банковского перевода. Сервисы P2P-переводов позволяют отправить
деньги другу, зная только номер его~карты.

Принципиальное отличие OCT от~обычной авторизации~--- отсутствие
холда. При~стандартной покупке авторизация блокирует сумму на~счёте
плательщика, и~эти деньги как~бы~\enquote{зарезервированы} до~клиринга.
При~OCT никакого холда нет: деньги кредитуются непосредственно
на~счёт получателя, и~банк-эмитент получателя обязан сделать их~доступными
в~установленные сроки~--- как~правило, в~течение нескольких минут.

Полный flow Visa Direct включает две связанные транзакции. Отправитель
совершает AFT (Account Funding Transaction): деньги списываются
с~его~карты или~счёта через обычную авторизацию. Затем платформа
инициирует OCT на~карту получателя. \scheme{Visa} маршрутизирует OCT
через VisaNet, эмитент получателя зачисляет средства.

\section{Visa Advanced Authorization}
\label{sec:visa-vaa}

Традиционно оценку риска транзакции выполняет эмитент: именно он~знает
историю покупок клиента, его~типичное поведение и~текущий баланс.
Но~у~эмитента есть только те~данные, которые он~сам накопил о~своих
клиентах. \scheme{Visa} как~сеть видит \emph{все} транзакции
всех~участников~--- и~именно это~дополнительное знание лежит
в~основе Visa Advanced Authorization (VAA)~\sidecite{visa-core-rules}.

VAA~--- это~система оценки рисков на~уровне сети: перед тем как
авторизационный запрос достигнет эмитента, \scheme{Visa} вычисляет
для~него числовой risk score на~основе транзакционной истории
по~всей сети. Этот score передаётся эмитенту в~авторизационном запросе
в~специальном поле (DE~44, subfield~\todocite{точный DE и subfield VAA
risk score в~авторизационном сообщении Visa}). Эмитент может
использовать его~в~своих правилах принятия решений: например,
снизить порог для~дополнительной проверки, если~VAA сигнализирует
о~высоком риске~\sidecite{visa-core-rules}.

Связь VAA с~системой 3-D Secure не~случайна. Когда 3DS запускает
Risk-Based Authentication (RBA)~--- механизм, описанный в~главе~\ref{ch:3ds}~---
один из~факторов, которые эмитент учитывает при~решении о~friction-less
flow или~challenge, это~именно network-level risk score от~VAA.
Высокий VAA score может принудить к~challenge даже транзакцию, которая
по~всем остальным параметрам выглядела бы безопасной для~конкретного
эмитента. Сочетание сетевых данных \scheme{Visa} и~данных эмитента
даёт более точную оценку, чем~любой из~источников в~отдельности.


\section{Interchange: структура таблиц}
\label{sec:visa-interchange}

Interchange~--- комиссия, которую банк-эквайер платит банку-эмитенту
за~каждую транзакцию. Размер этой комиссии определяется \scheme{Visa}
через таблицу interchange rates, которая обновляется дважды в~год
вместе с~Core Rules. Понять логику таблицы важно: именно от~неё
зависит то, что~мерчант видит в~своей выписке как~\enquote{processing fee}~\sidecite{visa-core-rules}.

Таблица организована по~программам (interchange programs): каждая
программа соответствует определённому сочетанию типа карты (дебетовая,
кредитная, corporate, premium rewards), категории мерчанта по~MCC
(Merchant Category Code) и~канала транзакции (card-present, card-not-present,
contactless). Мерчант с~MCC~5411 (продуктовые магазины) получит
одну ставку для~дебетовых карт, другую~--- для~кредитных, третью~---
для~корпоративных, и~каждая из~них ещё~раз разбивается по~наличию
EMV-данных в~транзакции~\sidecite{visa-core-rules}.

Понятие qual / non-qual / downgrade возникает именно здесь. Если~транзакция
соответствует всем критериям программы~--- передан \de{55} с~EMV-данными,
заполнены нужные поля авторизации, тип карты совпадает с~ожидаемым~---
транзакция \enquote{квалифицируется} по~программе и~получает
соответствующую (обычно более низкую) ставку. Если~хоть одно условие
не~выполнено~--- например, терминал не~передал chip data или~мерчант
забыл включить AVS (Address Verification Service) для~card-not-present~---
транзакция \enquote{падает} на~более высокую ставку. Это~и~называется
downgrade.

\begin{important}
Fallback-транзакция~--- когда терминал не~смог прочитать чип
и~прошёл по~магнитной полосе (разобрано в~главе~\ref{ch:emv})~---
почти гарантированно приводит к~downgrade interchange. Помимо того,
что~терминал принимает на~себя liability за~возможный fraud,
эквайер платит за~каждую такую транзакцию повышенную ставку, а~сам
мерчант, как~правило, видит это~в~виде \enquote{неожиданно высоких}
processing fees в~конце месяца.
\end{important}

Существует два основных ценовых формата, которые предлагают эквайеры
мерчантам. \textbf{Interchange++}~--- прозрачная модель: мерчант видит
фактическую ставку interchange для~каждой транзакции плюс комиссию
схемы плюс маржу эквайера. Мерчант точно знает, что~именно он~платит
и~за~что. \textbf{Blended pricing} (смешанное ценообразование)~--- упрощённая
модель: единая ставка для~всех транзакций, независимо от~типа карты
и~канала. Удобно для~малого бизнеса, который не~хочет разбираться
в~программах interchange, но~экономически невыгодно для~крупного
мерчанта с~большой долей дешёвых (дебетовых, в~том~числе PIN-debit)
транзакций: он~переплачивает за~дешёвые карты, субсидируя дорогие~\sidecite{visa-core-rules}.

\begin{practice}
Чтобы найти interchange-программу для~конкретной транзакции в~документе
Visa Interchange Rates, удобно двигаться от~общего к~частному. Первый
шаг~--- определить тип карты: потребительская или~коммерческая (commercial),
дебетовая или~кредитная, и~тир (Visa Traditional, Visa Signature, Visa
Infinite). Второй шаг~--- найти раздел для~вашего MCC: таблица объединяет
коды в~группы (Supermarkets, Travel, Restaurants и~т.\,д.); MCC вне
именованной группы получает базовую ставку данного типа карты. Третий
шаг~--- проверить qualifier conditions рядом со~ставкой: типичные условия
включают наличие~\de{55} с~chip data или~токеном, заполненность~\de{18}
(MCC) и~AVS-response для~card-not-present транзакций. Несоответствие
хотя~бы одному условию ведёт к~downgrade~--- транзакция перекидывается
на~ставку из~колонки non-qualified или~standard rate. Четвёртый шаг~---
сравнить фактическую ставку в~выписке с~рассчитанной: расхождение
указывает либо на~downgrade из-за~отсутствующих данных, либо на~кастомизированные
условия членского соглашения эквайера.
\end{practice}

\section*{Итоги главы}

\begin{itemize}
  \item VisaNet разделена на~BASE~I (авторизация реального времени,
        p99 < 1~с) и~BASE~II (пакетный клиринг, файлы TC33);
        разделение позволяет масштабировать каждую часть независимо.
  \item Core Rules~--- публичный документ, обновляется в~апреле
        и~октябре; изменения анонсируются через bulletins за~6--12~месяцев.
  \item Сертификация терминалов трёхуровневая: L1 (физика), L2
        (EMV-ядро), L3 (интеграция с~эквайером); ответственность
        за~наличие сертифицированных терминалов лежит на~эквайере.
  \item Visa Token Service добавляет к~стандарту EMVCo Token Assurance
        Level и~domain restriction; TOKEN бесполезен за~пределами
        своего домена.
  \item Visa Direct (OCT) кредитует карту получателя в~реальном
        времени без~холда; полный flow: AFT (списание у~отправителя)
        → OCT (кредит получателю).
  \item VAA оценивает риск на~уровне сети до~эмитента и~передаёт
        score в~авторизационном запросе; используется эмитентом
        в~том~числе при~RBA в~3-D Secure.
  \item Interchange определяется программой (тип карты × MCC ×
        канал); несоответствие критериям программы ведёт к~downgrade;
        interchange++ прозрачнее blended pricing для~крупных мерчантов.
\end{itemize}
