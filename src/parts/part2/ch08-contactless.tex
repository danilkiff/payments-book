\section{NFC и ISO 14443}
\label{sec:contactless-nfc}

Бесконтактная платёжная карта общается с~терминалом не~через контактную
площадку чипа, а~через радиоинтерфейс ближнего поля~--- NFC (Near Field
Communication), работающий на~частоте 13,56~МГц с~дальностью связи
около четырёх сантиметров. Физический и~канальный уровни бесконтактного
обмена определены стандартом \iso{14443}, который описывает два типа
карт~--- Type~A (с~амплитудной модуляцией) и~Type~B (с~фазовой),~---
протокол антиколлизии (позволяющий терминалу общаться с~одной из~нескольких
карт в~поле) и~транспортный протокол \iso{14443}-4, поверх которого
работает всё платёжное приложение~\sidecite{iso-14443}.

Важная особенность бесконтактного интерфейса~--- карта не~имеет
собственного источника питания. Электромагнитное поле, создаваемое
антенной терминала, индуцирует ток в~антенне карты, и~этого тока
достаточно для~питания чипа и~выполнения криптографических операций.
Этот~же принцип позволяет использовать для~оплаты устройства без
батареи~--- платёжные кольца, брелоки и~стикеры: если~в~них есть
антенна и~чип, терминал обеспечит энергию.

Терминал, поддерживающий бесконтактные платежи, оснащён NFC-антенной
(обычно встроенной в~верхнюю панель) и~контроллером, который
реализует оба типа \iso{14443} и~протокол антиколлизии. Когда карта
или~телефон оказываются в~поле, контроллер обнаруживает устройство,
устанавливает канал связи и~передаёт управление платёжному программному
обеспечению терминала.

\section{Contactless EMV: тот же протокол, меньше шагов}
\label{sec:contactless-emv}

Бесконтактная EMV-транзакция~--- это не~другой протокол, а~оптимизированная
версия контактного flow, описанного в~главе~\ref{ch:emv}. Сокращение
обусловлено жёстким ограничением по~времени: от~момента, когда карта
оказалась в~поле, до~завершения обмена данными должно пройти не~более
500~миллисекунд~--- иначе пользователь убирает карту, и~связь теряется.
Пятифазный контактный flow в~полсекунды не~укладывается, поэтому
бесконтактные спецификации сокращают число APDU-команд, объединяют шаги
и~допускают менее ресурсоёмкие методы
аутентификации~\sidecite{emvco-contactless-book-a}.

Бесконтактный выбор приложения начинается с~команды SELECT для~PPSE
(\texttt{2PAY.SYS.DDF01})~--- бесконтактного каталога, аналогичного
PSE из~контактного flow. PPSE возвращает список AID бесконтактных
приложений на~карте, и~терминал выбирает подходящее. Затем терминал
отправляет GET PROCESSING OPTIONS (GPO), карта возвращает AIP и~AFL,
терминал считывает необходимые записи~--- и~на~этом диалог может
завершиться генерацией криптограммы. Верификация держателя (PIN,
подпись) для~небольших сумм не~выполняется вовсе, а~аутентификация
данных использует упрощённые методы (fDDA вместо полного DDA).

Ещё одно ключевое понятие бесконтактных платежей~--- CVM limit
(Cardholder Verification Method limit): пороговая сумма, ниже которой
терминал не~требует никакой верификации держателя (No CVM). Порог
устанавливается для~каждой страны отдельно: например, в~России
для~\scheme{Visa}~--- 3000~рублей, для~\scheme{Mastercard}~---
5000~рублей~\todocheck{актуальные CVM limits для~РФ по~Visa/MC/Мир~---
проверить Core Rules и~правила НСПК}. Выше порога терминал требует
онлайн-PIN или~CDCVM (биометрию на~телефоне).

\subsection*{Концепция ядра (kernel)}

EMVCo определяет несколько \emph{ядер} (kernels)~--- спецификаций
бесконтактного протокола, каждая из~которых описывает поведение
конкретной карточной сети. Терминал загружает нужное ядро после
определения AID приложения: Kernel~3 для~\scheme{Visa} (qVSDC),
Kernel~2 для~\scheme{Mastercard} (M/Chip Contactless), а~Мир
использует собственную спецификацию бесконтактного ядра. Различия
между ядрами~--- не~только в~деталях протокола, но~и~в~методах
аутентификации данных, формате криптограммы и~правилах
обработки~\sidecite{emvco-contactless-book-a}.


\section{Три ядра: qVSDC, M/Chip Contactless, Мир бесконтактный}
\label{sec:contactless-kernels}

\subsection*{qVSDC (\scheme{Visa})}

qVSDC (quick Visa Smart Debit/Credit)~--- бесконтактное ядро
\scheme{Visa}. Его~главная особенность~--- fDDA (fast DDA):
упрощённая динамическая аутентификация данных, при~которой карта
подписывает криптограмму и~данные транзакции своим приватным ключом
в~рамках ответа на~GPO, объединяя аутентификацию данных и~генерацию
криптограммы в~один шаг. Это~сокращает число APDU-команд
и~укладывает весь обмен в~лимит времени. qVSDC всегда генерирует
ARQC~--- все бесконтактные транзакции \scheme{Visa} идут онлайн
(офлайн-одобрение для~бесконтактных карт \scheme{Visa} не~предусмотрено
в~стандартном flow)~\sidecite{visa-core-rules}.

\subsection*{M/Chip Contactless (\scheme{Mastercard})}

Бесконтактное ядро \scheme{Mastercard} использует иной подход
к~аутентификации: вместо fDDA применяется CVC3~--- динамический
код верификации, вычисляемый картой с~использованием сессионного
ключа, непредсказуемого числа (UN), генерируемого считывателем,
и~счётчика транзакций (ATC). CVC3 компактнее криптограммы,
что~ускоряет обмен, но~обеспечивает меньший уровень
криптографической защиты по~сравнению с~полной криптограммой ARQC.
Впрочем, современные реализации M/Chip Contactless также поддерживают
полный ARQC-flow~\sidecite{emvco-contactless-book-c2}.

\subsection*{Мир бесконтактный}

Бесконтактная спецификация Мир реализована в~платёжном приложении
MPA и~использует ГОСТ-криптографию: аутентификация данных выполняется
по~ГОСТ~Р~34.10 (ЭЦП на~эллиптических кривых), криптограмма
вычисляется с~использованием ГОСТ-симметричных алгоритмов. ECC-подпись
(ГОСТ~34.10) компактнее RSA и~хорошо подходит для~бесконтактного
flow, где каждый байт на~счету. Ядро Мир поддерживает как~офлайн-
так и~онлайн-режим, а~CVM limit устанавливается по~правилам
НСПК~\sidecite{nspk-mir-contactless}.

\section{Мобильные платежи: Apple Pay, Google Pay, Mir Pay}
\label{sec:contactless-mobile}

Платёжное кольцо или~карта с~чипом~--- это физический объект,
который нельзя скопировать (приватный ключ не~покидает чип).
Телефон~--- устройство общего назначения, на~котором работают
сотни приложений, и~хранить платёжные ключи в~обычной памяти
было~бы небезопасно. Мобильные платёжные системы решают эту
задачу двумя архитектурными подходами.

\subsection*{Secure Element (SE)}

Apple Pay использует Secure Element~--- отдельный чип внутри
телефона (или~часов), физически изолированный от~основного процессора
и~операционной системы. SE функционально аналогичен чипу на~карте:
хранит приватные ключи и~платёжные credentials, выполняет
криптографические операции, не~выпуская ключевой материал наружу.
Даже если операционная система телефона скомпрометирована
(jailbreak, вредоносное ПО), SE остаётся недоступным~\sidecite{apple-pay-security}.

\subsection*{HCE (Host Card Emulation)}

Google Pay (и~до~недавнего времени Mir Pay) используют HCE~---
программную эмуляцию карты, при~которой криптографические
операции выполняются в~облаке, а~на~телефоне хранятся ограниченные
по~числу использований ключи (LUK~--- Limited Use Keys). Каждый
LUK годен для~небольшого числа транзакций или~ограниченного
периода времени; после исчерпания телефон запрашивает новый
набор ключей у~облачного сервера. HCE не~требует специального
оборудования (SE может отсутствовать), но~зависит от~интернет-соединения
для~пополнения ключей и~менее устойчив к~компрометации устройства,
чем~SE.

\subsection*{Provisioning: как телефон становится картой}

Добавление карты в~мобильный кошелёк (provisioning) запускает
цепочку, в~которой участвуют кошелёк (Wallet App), Token Service
Provider (TSP) карточной сети и~эмитент:

\begin{enumerate}
  \item Держатель вводит данные карты в~приложении кошелька (или~делает
        фото карты, или~выбирает карту из~мобильного банка).
  \item Кошелёк отправляет данные TSP (VTS для~\scheme{Visa},
        MDES для~\scheme{Mastercard}, TSP НСПК для~Мир).
  \item TSP проверяет данные у~эмитента и~запрашивает одобрение
        на~токенизацию (подробно в~главе~\ref{ch:tokenization}).
  \item Эмитент может потребовать дополнительную аутентификацию
        (step-up): SMS-код, push в~мобильном банке, звонок.
  \item После одобрения TSP генерирует DPAN (Device PAN)~--- токен,
        привязанный к~конкретному устройству и~домену
        (NFC / in-app / e-commerce).
  \item DPAN и~ключевой материал загружаются в~SE (Apple Pay)
        или~в~облачный профиль HCE (Google Pay).
\end{enumerate}

\subsection*{CDCVM: биометрия вместо PIN}

При~оплате телефоном верификация держателя выполняется
не~на~клавиатуре терминала, а~на~самом устройстве: Face~ID,
Touch~ID, отпечаток пальца или~пароль разблокировки. Этот метод
называется CDCVM (Consumer Device CVM) и~передаётся в~авторизационном
сообщении как~отдельный признак в~CVM Results (тег~\hex{9F34}).
С~точки зрения эмитента CDCVM~--- полноценный метод верификации,
эквивалентный онлайн-PIN; поэтому при~оплате телефоном CVM limit
не~применяется~--- биометрия заменяет PIN при~любой
сумме~\sidecite{emvco-contactless-book-a}.


\section{Что видит эмитент, когда вы платите телефоном}
\label{sec:contactless-issuer-view}

Авторизационное сообщение от~мобильного платежа отличается от~обычного
бесконтактного несколькими ключевыми элементами:

\begin{itemize}
  \item \textbf{\de{2} (PAN)}~--- содержит не~настоящий PAN карты,
        а~DPAN (токен), выданный TSP при~провизионировании. DPAN
        проходит проверку Луна, имеет собственный BIN-диапазон
        и~выглядит как~обычный PAN, но~не~совпадает с~номером
        физической карты.
  \item \textbf{\de{22} (POS Entry Mode)}~--- код~07 (contactless
        M-Chip) или~аналогичный, указывающий на~бесконтактный вход.
        Некоторые реализации добавляют специфический код для~мобильных
        устройств.
  \item \textbf{\de{55} (EMV Data)}~--- содержит EMV-теги, специфичные
        для~мобильного платежа: Device Type (телефон, часы),
        данные о~форм-факторе, криптограмму, вычисленную с~ключами,
        полученными при~провизионировании.
  \item \textbf{Token Requestor ID}~--- идентификатор кошелька
        (Apple Pay, Google Pay), позволяющий эмитенту понять,
        через какой канал пришла транзакция.
\end{itemize}

Карточная сеть (или~TSP) выполняет де-токенизацию до~эмитента:
заменяет DPAN на~настоящий PAN и~добавляет метаданные токена
(assurance level, domain restriction). Эмитент видит и~настоящий
PAN, и~индикатор того, что~транзакция пришла через мобильный
кошелёк~--- и~может использовать эту информацию для~более точного
risk scoring (мобильные платежи статистически менее рискованны,
чем~пластиковые карты).


\section{Wearables: кольцо, часы, браслет}
\label{sec:contactless-wearables}

Тот~же принцип~--- NFC-антенна, чип с~SE, EMV-ядро~--- работает
в~любом форм-факторе, который можно поднести к~терминалу. Apple Watch
используют Secure Element и~тот~же flow провизионирования, что
и~iPhone; платёжные кольца (например, McLear RingPay) содержат
пассивный NFC-чип с~предзагруженным платёжным приложением,
получающий питание от~поля терминала; фитнес-браслеты Garmin Pay
и~аналоги хранят токенизированные credentials в~встроенном
чипе.

Ограничения wearables связаны с~форм-фактором: отсутствие экрана
или~клавиатуры делает невозможным ввод PIN на~самом устройстве;
CDCVM может быть ограничен (часы с~датчиком пульса используют
\enquote{wrist detection} как~суррогат аутентификации). Для~высоких
сумм, требующих полноценной верификации, wearable-устройство может
попросить разблокировать сопряжённый телефон или~ввести PIN
на~терминале.


\section{Атака relay и границы защиты}
\label{sec:contactless-relay}

Бесконтактный интерфейс создаёт вектор атаки, невозможный
для~контактных карт: relay attack (атака ретрансляции). Злоумышленник
размещает одно устройство рядом с~картой жертвы (например,
в~переполненном вагоне метро), а~второе~--- рядом с~терминалом
в~магазине. Два устройства соединены каналом связи (Bluetooth, Wi-Fi,
интернет), и~NFC-обмен между картой и~терминалом транслируется
в~реальном времени: терминал \enquote{думает}, что~карта рядом,
а~карта \enquote{думает}, что~рядом терминал.

Криптограмма при~этом остаётся валидной~--- она вычислена настоящим
чипом, с~настоящим ключом, на~основании реальных данных транзакции.
Привязка криптограммы к~дате, времени и~сумме не~помогает: relay
добавляет лишь миллисекунды задержки, и~все параметры транзакции
подлинные~--- злоумышленник не~подменяет данные, а~лишь
транслирует~диалог.

Теоретическая защита~--- distance bounding protocols: терминал
измеряет время отклика карты с~точностью до~наносекунд и~отклоняет
транзакцию, если задержка превышает ожидаемую для~расстояния в~несколько
сантиметров. Однако на~практике distance bounding не~получил широкого
распространения в~платёжных системах из-за сложности реализации
и~необходимости модификации оборудования.

\begin{important}
Реальная защита от~relay attack в~сегодняшних платёжных системах
обеспечивается не~криптографией, а~экономическими и~поведенческими
мерами: CVM limits ограничивают сумму бесконтактной транзакции
без~верификации; velocity checks блокируют серию мелких покупок
за~короткий период; антифрод-система эмитента анализирует
геолокацию, привычный паттерн расходов и~тип мерчанта. Для~мобильных
платежей relay практически невозможен: телефон требует CDCVM
(биометрию) перед каждой транзакцией, и~без~участия держателя
NFC-интерфейс не~активируется.
\end{important}


\section*{Итоги главы}

\begin{itemize}
  \item Бесконтактный платёж использует NFC (\iso{14443}, 13,56~МГц,
        $\sim$4~см); карта получает питание от~электромагнитного поля
        терминала.
  \item Бесконтактный EMV-flow~--- сокращённая версия контактного:
        меньше APDU-команд, упрощённая аутентификация (fDDA, CVC3),
        время обмена~$<$~500~мс.
  \item Три ядра отличаются методами аутентификации и~криптографией:
        qVSDC (\scheme{Visa})~--- fDDA + ARQC;
        M/Chip Contactless (\scheme{Mastercard})~--- CVC3 или~ARQC;
        Мир~--- ГОСТ~34.10 + ГОСТ-криптограмма.
  \item Мобильные платежи хранят credentials в~Secure Element
        (Apple Pay) или~облаке с~LUK (Google Pay / HCE);
        провизионирование создаёт DPAN (токен), привязанный
        к~устройству и~домену.
  \item CDCVM (биометрия на~телефоне) заменяет PIN при~любой сумме;
        при~оплате телефоном CVM limit не~применяется.
  \item Эмитент получает DPAN вместо PAN, POS Entry Mode~07,
        EMV-данные с~device-специфичными тегами и~Token Requestor~ID;
        сеть де-токенизирует DPAN до~эмитента.
  \item Relay attack транслирует NFC-диалог на~расстояние;
        криптограмма остаётся валидной. Защита~--- CVM limits,
        velocity checks, антифрод эмитента; мобильные платежи
        защищены требованием CDCVM перед каждой транзакцией.
\end{itemize}
