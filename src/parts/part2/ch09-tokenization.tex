\chapter{Токенизация: PAN, который не PAN}
\label{ch:tokenization}

\begin{kaobox}[title=Что вы узнаете из этой главы]
  В~главе~\ref{ch:card-data} мы~увидели, что PAN~--- одновременно
  идентификатор и~квази-секрет, хранящийся повсюду в~платёжной цепочке.
  В~главе~\ref{ch:contactless}~--- что~мобильный кошелёк получает
  не~настоящий PAN, а~DPAN. Здесь мы~разберём механизм, стоящий
  за~этой подменой: сетевую токенизацию. Что~такое DPAN, Token Requestor
  и~Assurance Level; чем~отличаются три провайдера (VTS, MDES, TSP~НСПК);
  почему сетевой токен повышает approval rate; и~кто в~итоге видит
  настоящий PAN.
\end{kaobox}

Представим типичный инцидент: база интернет-мерчанта с~сохранёнными
картами утекла, и~злоумышленник получил миллионы PAN. Токенизация
интересна ровно в~той точке, где нужно превратить такую утечку
из~катастрофы в~ограниченное событие: нас будет интересовать,
что именно заменяется, кто выпускает замену и~почему украденный
токен не~равен украденной карте.


\section{Проблема, которую решает токенизация}
\label{sec:token-problem}

PAN путешествует через десятки систем: мерчант записывает его
при~оплате, PSP хранит для~возвратов, подписочный сервис использует
для~рекуррентных списаний, маркетплейс~--- для~выплат. Каждая копия~---
потенциальная точка утечки: взлом базы мерчанта, перехват логов,
компрометация резервной копии. Масштаб проблемы иллюстрируют
регулярные утечки миллионов карточных номеров~--- каждая из~которых
начинается с~того, что~PAN хранился там, где~в~этом не~было
криптографической необходимости.

Токенизация решает проблему заменой: вместо настоящего PAN
по~платёжной цепочке перемещается токен~--- суррогатное значение,
которое выглядит как~PAN (проходит проверку Луна, имеет собственный
BIN-диапазон), но~бесполезно за~пределами конкретного контекста
использования. Утечка токена не~эквивалентна утечке PAN: токен
ограничен доменом (только NFC, только конкретный мерчант, только
e-commerce), и~попытка использовать его~в~другом контексте будет
отклонена~\sidecite{emvco-payment-tokenisation}.

Идея замены PAN суррогатом не~нова: PSP и~платёжные шлюзы
давно генерировали собственные \enquote{токены}~--- непрозрачные
ссылки на~PAN в~своём хранилище. Принципиальное отличие \emph{сетевой}
токенизации~--- токен выпускается и~распознаётся карточной сетью,
что~превращает его~из~локальной ссылки в~сквозной идентификатор,
валидный на~всём пути от~мерчанта до~эмитента.


Итак, сетевой токен решает не~абстрактную задачу маскировки,
а~конкретную задачу сокращения поверхностей компрометации
и~привязки платёжного идентификатора к~домену использования.
Чтобы понять, почему это работает лучше локального PSP-токена,
нужно разобрать анатомию самого токена.

\section{Анатомия сетевого токена}
\label{sec:token-anatomy}

\begin{termdef}{DPAN}
  DPAN (Device PAN, Digital PAN) --- токен, заменяющий настоящий PAN
  (FPAN, Funding PAN) в~платёжной транзакции. DPAN выпускается Token
  Service Provider (TSP) карточной сети, проходит проверку Луна, имеет
  длину и~формат стандартного PAN, но~принадлежит отдельному
  BIN-диапазону, зарезервированному для~токенов.
\end{termdef}

Каждый сетевой токен характеризуется несколькими атрибутами, понимание
которых необходимо для~грамотного применения токенизации в~платёжных
системах~\sidecite{emvco-payment-tokenisation}.

\textbf{Token Requestor}~--- сущность, запросившая токен: мобильный
кошелёк (Apple Pay, Google Pay), мерчант для~card-on-file, PSP
или~IoT-устройство. Каждый Token Requestor идентифицируется уникальным
TRID (Token Requestor ID), который позволяет сети и~эмитенту понять,
кто~именно владеет токеном. Одна карта может иметь множество токенов
с~разными TRID: Apple Pay, конкретный мерчант подписки и~Google Pay
получат три отдельных токена, каждый со~своим жизненным циклом
и~независимый от~остальных~\sidecite{emvco-payment-tokenisation}.

\textbf{Assurance Level}~--- уровень уверенности TSP в~том, что~токен
привязан к~законному держателю карты, выраженный по~шкале от~0 до~99
(EMVCo задаёт общую рамку; конкретные значения определяются схемой).
Высокий Assurance Level присваивается после строгой аутентификации~---
например, по~итогам 3-D~Secure challenge или~step-up через мобильный банк.
Для~эмитента это~сигнал доверия: токен с~высоким Assurance Level
представляет меньший риск, чем~необработанный PAN, чья история
аутентификации эмитенту неизвестна. Именно это свойство напрямую
улучшает approval rate токенизированных транзакций~\sidecite{emvco-payment-tokenisation}.

\textbf{Domain restriction}~--- контекст, в~котором токен валиден:
NFC (только бесконтактные транзакции на~конкретном устройстве),
e-commerce (только онлайн-покупки у~конкретного мерчанта),
card-on-file (рекуррентные списания у~фиксированного получателя).
Попытка использовать токен за~пределами его~домена~--- например,
передать NFC-токен мобильного кошелька в~онлайн-форму~--- приведёт
к~отклонению на~уровне сети, ещё до~того как~запрос достигнет
эмитента. PSP-токены доменных ограничений не~несут: они~валидны
в~любом контексте, где~их~примет соответствующий PSP.

\textbf{Lifecycle}~--- жизненный цикл токена:
inactive, active, suspended, resumed, deleted.

Эмитент может приостановить конкретный токен при~подозрении
на~компрометацию, не~затрагивая ни~саму карту, ни~другие токены
на~её~основе. Держатель продолжает пользоваться остальными
платёжными инструментами без~перевыпуска пластика.

Именно это делает токенизацию хорошим ответом на~утечку данных:
скомпрометированный токен удаляется одним вызовом API, а~карта
остаётся нетронутой~\sidecite{emvco-payment-tokenisation}.

Между атрибутами существует взаимодействие, которое делает систему
сложнее, но~и~мощнее: высокий Assurance Level в~сочетании с~узким
доменом~--- например, NFC-only на~конкретном устройстве~--- даёт
эмитенту максимальный сигнал доверия. Такой токен подтверждён строгой
аутентификацией и~при~этом физически неотделим от~устройства
держателя~--- злоумышленнику недостаточно знать его~значение, чтобы
провести транзакцию в~другом канале или~с~другого устройства.

\begin{kaobox}[title=Атрибуты сетевого токена (справка)]
  \begin{itemize}
    \item \textbf{Token Requestor / TRID}~--- кто~запросил: кошелёк, мерчант, PSP, IoT
    \item \textbf{Assurance Level} (0--99)~--- уровень уверенности в~держателе
    \item \textbf{Domain restriction}~--- NFC / e-commerce / card-on-file
    \item \textbf{Lifecycle}~--- inactive $\to$ active $\to$ suspended $\to$ deleted
  \end{itemize}
\end{kaobox}


\section{VTS, MDES, TSP НСПК --- три провайдера}
\label{sec:token-providers}

Каждая крупная карточная сеть управляет собственным Token Service
Provider. Хотя архитектурно все три аналогичны~--- Token Vault,
маппинг DPAN--FPAN, интерфейс к~эмитенту для~проверки личности~---
каждый реализует детали enrollment, lifecycle и~криптографии по-своему.
Понимание этих различий важно и~при~интеграции, и~при~разборе инцидентов.

\subsection*{VTS (\scheme{Visa} Token Service)}

С~точки зрения Token Requestor~--- например, Apple при~добавлении
карты \scheme{Visa} в~Apple~Pay~--- интеграция начинается с~Enrollment
API: Requestor передаёт в~VTS данные карты и~идентификатор устройства,
VTS формирует запрос ID\&V (Identity and Verification) к~эмитенту~---
проверку того, что~карта принадлежит законному держателю. Эмитент
может делегировать ID\&V самому VTS, полагаясь на~скоринг по~данным
карты, или~запросить step-up аутентификацию: SMS-код, push
в~мобильном банке, видеозвонок. После одобрения VTS генерирует
DPAN, присваивает TRID запрашивающего кошелька, устанавливает
Assurance Level в~соответствии с~методом аутентификации и~загружает
ключевой материал в~Secure Element или~облачный
профиль~\sidecite{visa-vts}.

Архитектурно VTS построен вокруг централизованного Token Vault~---
хранилища маппинга \mbox{DPAN~$\leftrightarrow$~FPAN} с~полным
набором метаданных. При~авторизации сеть \scheme{Visa} перехватывает
транзакцию, распознаёт DPAN по~зарезервированному BIN-диапазону
и~выполняет lookup в~Token Vault~--- подставляет FPAN в~сообщение
для~эмитента, а~метаданные (Assurance Level, TRID, domain restriction)
добавляет в~специфичные поля. Вся операция де-токенизации происходит
внутри сетевой инфраструктуры \scheme{Visa}, добавляя лишь несколько
миллисекунд к~общему времени обработки; ни~эквайер, ни~PSP
эту задержку не~ощущают~\sidecite{visa-vts}.

Lifecycle-события обрабатываются автоматически. Когда эмитент
уведомляет VTS об~изменении FPAN (перевыпуск карты с~новым номером
или~сроком действия), все привязанные DPAN обновляются в~Token Vault:
мерчант с~card-on-file токеном продолжает выставлять рекуррентные
списания, а~пользователь Apple Pay не~замечает замены пластика.
Эмитент также может через интерфейс управления токенами приостановить
или~удалить отдельный токен~--- например, при~потере телефона,~---
не~затрагивая карту и~другие привязанные
устройства~\sidecite{visa-vts}.

\subsection*{MDES (Mastercard Digital Enablement Service)}

MDES (Mastercard Digital Enablement Service)~--- платформа токенизации
\scheme{Mastercard}, архитектурно близкая к~VTS, но~реализующая
enrollment flow с~рядом отличий. При~добавлении карты Token Requestor
отправляет в~MDES запрос на~токенизацию; MDES оценивает риск
с~помощью собственного скоринга на~основе данных об~устройстве
и~истории карты, а~затем определяет метод ID\&V~--- от~тихого
одобрения на~основе скоринговой оценки до~принудительного step-up.
Результирующий Assurance Level отражает эту оценку: токен,
провизионированный через step-up биометрией в~мобильном банке,
получает более высокий уровень, чем~токен, добавленный только
по~данным карты~\sidecite{mc-mdes}.

Механизм автообновления токенов при~перевыпуске карты~---
Account on File update~--- встроен в~Token Life Cycle Management API.
Когда карта перевыпускается (изменился PAN или~срок действия),
MDES получает уведомление от~эмитента, обновляет маппинг в~Token
Vault и~может передать событие Token Requestor~--- например,
мерчант с~card-on-file получит уведомление об~обновлении учётных
данных карты, не~узнав при~этом сам FPAN. Держатель при~этом
никаких действий не~предпринимает, а~непрерывность рекуррентных
платежей обеспечивается автоматически~\sidecite{mc-mdes}.

MDES также поддерживает специализированный сценарий для~IoT:
устройства без~экрана (умные колонки, подключённые автомобили,
торговые автоматы) получают токен с~жёстко ограниченным доменом~---
только конкретное устройство, только конкретный мерчант или~категория
мерчантов. Enrollment для~IoT предполагает упрощённый цикл без
держателя в~контуре, но~при~явном одобрении эмитента и~строгом
ограничении суммы и~типа транзакций~--- что~снижает риск
злоупотребления даже в~случае компрометации
устройства~\sidecite{mc-mdes}.

\subsection*{TSP НСПК}

Российская суверенная инфраструктура токенизации появилась по~той~же
причине, по~которой НСПК выстраивал всю операционную инфраструктуру
карт Мир: требование обрабатывать транзакции внутри страны. Если~бы
токенизация карт Мир строилась на~VTS или~MDES, де-токенизация
при~каждой авторизации требовала~бы обращения к~зарубежным серверам~---
что~прямо противоречит принципу локальной обработки. TSP НСПК
размещён в~инфраструктуре ОПКЦ НСПК, де-токенизация выполняется
при~обработке внутрироссийских транзакций без~выхода за~пределы
страны~\sidecite{nspk-tsp}.

Для~защиты ключевого материала токенов TSP НСПК применяет
ГОСТ-криптографию~--- ГОСТ~Р~34.10 (ЭЦП на~эллиптических кривых)
и~ГОСТ~Р~34.12 (блочный шифр Кузнечик)~--- в~соответствии
с~требованиями \law{683-П} к~применению сертифицированных СКЗИ.
Ключевой материал, передаваемый в~Mir~Pay или~банковское
мобильное приложение, тем самым защищён российскими
криптографическими стандартами. В~этом принципиальное отличие
от~VTS и~MDES, работающих с~AES и~RSA/ECC~\sidecite{nspk-tsp}.

После 2022~года основным каналом провизионирования стало банковское
мобильное приложение. Роль \enquote{кошелька} выполняет сам банк-эмитент,
а~не~сторонний агрегатор: держатель добавляет карту в~приложении
своего банка, банк инициирует enrollment через НСПК API, TSP НСПК
выпускает DPAN и~загружает ключи. Такая архитектура упрощает
ID\&V~--- банк уже знает клиента, имеет его~биометрические данные
и~историю операций~--- и~ускоряет провизионирование по~сравнению
с~традиционным маршрутом через сторонний
кошелёк~\sidecite{nspk-tsp}.

\section{Сетевой токен vs PSP-токен}
\label{sec:token-network-vs-psp}

Прежде чем сетевая токенизация стала стандартом, PSP и~платёжные
шлюзы давно предлагали собственные токены~--- непрозрачные ссылки
(например, UUID), которые соответствовали PAN в~хранилище PSP.
Эти токены решают часть проблемы: мерчант не~хранит PAN у~себя.
Однако от~сетевых токенов PSP-токены принципиально отличаются
по~шести параметрам~\sidecite{emvco-payment-tokenisation}.

\textbf{Портативность.} PSP-токен валиден исключительно внутри
одного PSP: при~смене платёжного шлюза мерчант теряет все токены
и~вынужден заново запрашивать данные карт у~держателей или~организовывать
миграцию через Account Updater. Это~создаёт жёсткую привязку к~поставщику~---
при~переговорах о~тарифах и~при~оценке альтернатив мерчант оказывается
в~слабой позиции. Сетевой токен принадлежит карточной сети и~остаётся
валидным при~любой смене PSP: достаточно передать новому шлюзу
тот~же DPAN~\sidecite{emvco-payment-tokenisation}.

\textbf{PCI scope.} PSP-токен выводит PAN из~систем мерчанта,
но~PSP сам хранит настоящий PAN в~своём vault и~остаётся в~полном
PCI DSS scope. Сетевая токенизация идёт дальше: если~PSP использует
сетевые токены сквозным образом, настоящий PAN не~попадает
ни~к~мерчанту, ни~в~среду PSP~--- только в~Token Vault карточной
сети. Это~принципиально сужает область применения PCI DSS:
меньше систем в~scope означает меньше контролей, меньше аудитов
и~меньше затрат на~комплаенс (подробнее в~главе~\ref{ch:pci-dss}).

\textbf{Authorization rate.} Сетевой токен несёт метаданные~---
Assurance Level и~domain restriction,~--- которые видны эмитенту
при~авторизации. Высокий Assurance Level служит сигналом доверия:
эмитент знает, что~провизионирование прошло со~строгой
аутентификацией, и~транзакция идёт именно с~того~устройства,
для~которого выпущен токен. PSP-токен таких метаданных не~несёт;
эмитент видит обычный PAN-запрос без~признаков подтверждённой
личности. Разница в~approval rate между токенизированными
и~нетокенизированными транзакциями составляет порядка
2--4~процентных пункта~\todocheck{разница в~approval rate
  между сетевыми токенами и~PAN~--- Visa/MC публичные данные}.

\textbf{Lifecycle management.} При~перевыпуске карты~--- новый PAN,
новый срок действия~--- сетевой токен обновляется автоматически
через TSP: эмитент уведомляет VTS или~MDES, маппинг в~Token Vault
меняется, и~мерчанту не~нужно предпринимать никаких действий.
PSP-токен жёстко привязан к~конкретному PAN в~vault; при~перевыпуске
мерчант или~PSP обязан запустить Account Updater (\scheme{Visa} VAU
или~\scheme{Mastercard} ABU) или~напрямую запросить новые данные
у~держателя. Неудачный Account Updater или~пропущенное уведомление
означают failed recurring payment~--- прямые потери выручки.

\textbf{Стоимость интеграции.} PSP-токенизация технически минимальна:
мерчант просто сохраняет token reference вместо PAN в~своей базе,
а~вся логика остаётся за~PSP. Сетевая токенизация требует либо
прямой интеграции с~TSP карточной сети (Token Requestor registration,
provisioning API), либо использования PSP, который уже поддерживает
сетевые токены на~стороне своей инфраструктуры. В~последнем случае
мерчант фактически получает преимущества сетевых токенов без~прямой
интеграции, просто выбирая PSP с~соответствующей поддержкой.

\textbf{Усилие поддержки.} После первоначальной интеграции
сетевой токен требует значительно меньших операционных усилий:
lifecycle management автоматизирован, перевыпуски карт обрабатываются
TSP, а~мерчанту достаточно обрабатывать lifecycle-события
(token suspended, token deleted) через web-hook. PSP-токен требует
постоянного ведения Account Updater, ручной работы с~упавшими
транзакциями из-за устаревших данных карты и~периодических
сверок между vault PSP и~фактическим состоянием карт.

\section{Де-токенизация: кто и когда видит настоящий PAN}
\label{sec:token-detokenization}

В~токенизированной транзакции настоящий PAN (FPAN) не~перемещается
по~всей цепочке~--- его~видят только те~участники, которым это
необходимо. Поток выглядит следующим образом:

\begin{enumerate}
  \item \textbf{Мерчант} отправляет транзакцию с~DPAN (сетевым
        токеном). Мерчант \emph{не~видит} FPAN~--- он~работает
        исключительно с~токеном.
  \item \textbf{Эквайер / PSP} передаёт транзакцию с~DPAN в~карточную
        сеть. В~зависимости от~модели, эквайер может не~знать, что
        перед ним токен, а~не~настоящий PAN.
  \item \textbf{Карточная сеть (TSP)} распознаёт DPAN по~BIN-диапазону,
        поднимает из~Token Vault FPAN и~метаданные
        (Assurance Level, domain, TRID), а~затем подставляет
        FPAN в~сообщение для~эмитента.
  \item \textbf{Эмитент} получает FPAN плюс индикаторы токенизации
        (token present, Assurance Level, TRID)~--- и~видит полную
        картину: какая карта, через какой кошелёк, с~каким уровнем
        доверия.
\end{enumerate}

Де-токенизация не~всегда завершается успехом~--- и~платёжная
инфраструктура должна уметь обрабатывать ошибки. Если~TSP при~обращении
к~Token Vault обнаруживает, что~DPAN приостановлен (suspended)~---
например, потому что~эмитент заподозрил компрометацию устройства,~---
транзакция отклоняется с~признаком~\enquote{token not active}
до~того как~запрос достигает эмитента. Удалённый токен (deleted)
даёт аналогичный результат. Истёкший срок действия самого DPAN~---
отдельный сценарий: TSP хранит срок действия токена независимо
от~срока действия физической карты, и~приложению Requestor необходимо
инициировать обновление. Разработчик платёжного flow обязан
различать эти состояния: silent retry с~тем~же DPAN~--- не~решение,
нужно либо обновить токен, либо попросить держателя заново
добавить карту~\sidecite{emvco-payment-tokenisation}.

В~клиринге правила различаются по~сетям. \scheme{Visa} в~клиринговом
файле TC33 может передавать уже де-токенизированный FPAN~---
для~мерчанта это означает, что~клиринговые и~расчётные записи
содержат настоящий номер карты, тогда как~авторизационный запрос
проходил с~DPAN. Такое поведение требует учёта при~сверке:
система, ожидающая один и~тот~же идентификатор на~всех этапах,
получит расхождение. \scheme{Mastercard} исторически сохраняет
DPAN до~клиринга в~IPM-файлах, что~упрощает сверку, но~требует
наличия маппинга на~стороне эмитента для~последующей аналитики.
Для~мерчанта обе модели прозрачны~--- он~работает с~токеном
на~всех этапах и~не~знает, что~происходит в~клиринге.

Отдельного внимания заслуживают пограничные сценарии в~мультиэквайерной
среде. Если~мерчант работает через двух эквайеров одновременно~---
например, один для~внутренних транзакций, другой для~международных~---
оба должны принимать один и~тот~же DPAN, поскольку токен принадлежит
карточной сети, а~не~PSP. Де-токенизация в~этом случае работает
корректно: какой~бы эквайер ни~маршрутизировал транзакцию,
TSP обращается к~одному Token Vault. Трансграничные транзакции
добавляют сложность на~стороне клиринга~--- процессинг эмитента
должен корректно обработать FPAN независимо от~того, через какую
страну проходил токенизированный запрос.


\section{Токенизация и 3-D Secure}
\label{sec:token-and-3ds}

Токенизация и~3-D~Secure решают задачи разного класса: токен
отвечает на~вопрос \enquote{что~за~карта и~откуда пришли credentials},
аутентификация~--- на~вопрос \enquote{это~действительно законный
  держатель}. Каждый из~этих механизмов в~отдельности снижает риск,
но~именно совместная работа даёт эмитенту максимальную уверенность~---
и~максимальный approval rate.

Механизм step-up особенно важен при~провизионировании мобильного
кошелька. Когда TSP запрашивает у~эмитента разрешение на~токенизацию,
эмитент может потребовать 3-D~Secure challenge~--- не~для~конкретной
транзакции, а~чтобы подтвердить, что~карту добавляет именно
владелец. Успешный challenge повышает Assurance Level
выпускаемого токена: эмитент видит, что~провизионирование
прошло со~строгой проверкой. Такой токен обычно одобряется чаще,
чем~токен с~низким Assurance Level после тихого скоринга.
Подробнее о~механизмах challenge и~step-up~--- в~главе~\ref{ch:3ds}.

При~e-commerce транзакции с~card-on-file сетевой токен и~результат
3DS-аутентификации передаются в~одном авторизационном сообщении.
В~\iso{8583} токен содержится в~\de{2} (DPAN вместо FPAN),
данные о~нём~--- в~специфичных полях сети, а~3DS authentication value
(CAVV или~AAV)~--- в~\de{126} или~эквивалентном расширении.
Эмитент получает сразу два слоя подтверждения: DPAN с~Assurance Level
говорит, что~credentials прошли проверку при~провизионировании,
а~CAVV~--- что~держатель подтвердил личность непосредственно
перед этой транзакцией. Комбинация снижает основания для~отказа
и~переносит ответственность на~эмитента при~возможном
мошенничестве~\sidecite{emvco-payment-tokenisation}.

Для~рекуррентных транзакций (MIT~--- Merchant-Initiated Transactions)
связка работает иначе. Первая транзакция проходит как~CIT
(Customer-Initiated Transaction) с~полным 3DS.

TSP сохраняет Network Transaction ID~--- идентификатор этой
аутентификационной цепочки. Последующие MIT используют сетевой
токен с~chaining по~этому Network Transaction ID, и~3DS для~них
уже не~требуется. Эмитент видит ссылку на~первоначальное согласие
держателя и~принимает его как достаточное основание.

Такая архитектура сохраняет непрерывность подписок и~рекуррентных
платежей без~ежемесячного вмешательства держателя
~\sidecite{emvco-payment-tokenisation}.

Подробно о~3-D~Secure~--- в~главе~\ref{ch:3ds}; о~влиянии
токенизации на~PCI scope~--- в~главе~\ref{ch:pci-dss};
о~влиянии на~антифрод~--- в~главе~\ref{ch:fraud}.


\section*{Итоги главы}

\begin{itemize}
  \item Токенизация заменяет настоящий PAN (FPAN) суррогатом
        (DPAN), ограниченным конкретным доменом (NFC, e-commerce,
        конкретный мерчант); утечка токена не~эквивалентна утечке PAN.
  \item Каждый сетевой токен несёт атрибуты: Token Requestor (кто~запросил),
        Assurance Level (уровень доверия), domain restriction (контекст
        использования), lifecycle (создан, активен, приостановлен, удалён).
  \item Три TSP~--- VTS (\scheme{Visa}), MDES (\scheme{Mastercard}),
        TSP~НСПК (Мир)~--- архитектурно аналогичны (Token Vault,
        маппинг DPAN--FPAN), но~управляются независимо.
  \item Сетевой токен превосходит PSP-токен по~портативности
        (не~привязан к~PSP), по~PCI scope
        (FPAN не~попадает к~мерчанту), по~authorization rate
        (+2--4~п.п.) и~по~lifecycle management
        (автообновление при~перевыпуске карты).
  \item Де-токенизация происходит в~карточной сети: мерчант и~эквайер
        видят только DPAN; эмитент получает FPAN плюс метаданные токена.
  \item Токенизация и~3-D~Secure работают в~паре: токен идентифицирует
        credentials, 3DS аутентифицирует держателя; вместе они~дают
        эмитенту максимальную уверенность в~легитимности транзакции.
\end{itemize}
