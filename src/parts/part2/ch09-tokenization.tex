\chapter{Токенизация: PAN, который не PAN}
\label{ch:tokenization}

\begin{kaobox}[title=Что вы узнаете из этой главы]
В~главе~\ref{ch:card-data} мы~увидели, что PAN~--- одновременно
идентификатор и~квази-секрет, хранящийся повсюду в~платёжной цепочке.
В~главе~\ref{ch:contactless}~--- что~мобильный кошелёк получает
не~настоящий PAN, а~DPAN. Здесь мы~разберём механизм, стоящий
за~этой подменой: сетевую токенизацию. Что~такое DPAN, Token Requestor
и~Assurance Level; чем~отличаются три провайдера (VTS, MDES, TSP~НСПК);
почему сетевой токен повышает approval rate; и~кто в~итоге видит
настоящий PAN.
\end{kaobox}


\section{Проблема, которую решает токенизация}
\label{sec:token-problem}

PAN путешествует через десятки систем: мерчант записывает его
при~оплате, PSP хранит для~возвратов, подписочный сервис использует
для~рекуррентных списаний, маркетплейс~--- для~выплат. Каждая копия~---
потенциальная точка утечки: взлом базы мерчанта, перехват логов,
компрометация резервной копии. Масштаб проблемы иллюстрируют
регулярные утечки миллионов карточных номеров~--- каждая из~которых
начинается с~того, что~PAN хранился там, где~в~этом не~было
криптографической необходимости.

Токенизация решает проблему заменой: вместо настоящего PAN
по~платёжной цепочке перемещается токен~--- суррогатное значение,
которое выглядит как~PAN (проходит проверку Луна, имеет собственный
BIN-диапазон), но~бесполезно за~пределами конкретного контекста
использования. Утечка токена не~эквивалентна утечке PAN: токен
ограничен доменом (только NFC, только конкретный мерчант, только
e-commerce), и~попытка использовать его~в~другом контексте будет
отклонена~\sidecite{emvco-payment-tokenisation}.

Идея замены PAN суррогатом не~нова: PSP и~платёжные шлюзы
давно генерировали собственные \enquote{токены}~--- непрозрачные
ссылки на~PAN в~своём хранилище. Принципиальное отличие \emph{сетевой}
токенизации~--- токен выпускается и~распознаётся карточной сетью,
что~превращает его~из~локальной ссылки в~сквозной идентификатор,
валидный на~всём пути от~мерчанта до~эмитента.


\section{Анатомия сетевого токена}
\label{sec:token-anatomy}

\begin{termdef}{DPAN}
DPAN (Device PAN, Digital PAN) --- токен, заменяющий настоящий PAN
(FPAN, Funding PAN) в~платёжной транзакции. DPAN выпускается Token
Service Provider (TSP) карточной сети, проходит проверку Луна, имеет
длину и~формат стандартного PAN, но~принадлежит отдельному
BIN-диапазону, зарезервированному для~токенов.
\end{termdef}

Каждый сетевой токен характеризуется несколькими атрибутами:

\begin{itemize}
  \item \textbf{Token Requestor}~--- сущность, запросившая токен:
        мобильный кошелёк (Apple Pay, Google Pay), мерчант (для~card-on-file),
        PSP или~IoT-устройство. Token Requestor идентифицируется
        уникальным TRID (Token Requestor ID), который позволяет
        сети и~эмитенту понять, кто~владеет токеном.
  \item \textbf{Assurance Level}~--- уровень уверенности TSP в~том,
        что~токен привязан к~законному держателю карты. Шкала от~0
        до~99 (EMVCo определяет общую рамку; конкретные значения
        задаются схемой). Высокий Assurance Level (например, после
        3-D~Secure-аутентификации или~step-up через мобильный банк)
        означает, что~эмитент может доверять токену больше,
        чем~необработанному PAN~--- и~это~напрямую влияет на~решение
        об~одобрении.
  \item \textbf{Domain restriction}~--- контекст, в~котором токен
        валиден: NFC (только бесконтактные транзакции на~конкретном
        устройстве), e-commerce (только онлайн-покупки у~конкретного
        мерчанта), card-on-file (рекуррентные списания). Попытка
        использовать токен за~пределами его~домена приведёт
        к~отклонению.
  \item \textbf{Lifecycle}~--- токен проходит состояния: создан
        (inactive), активирован (active), приостановлен (suspended),
        возобновлён (resumed), удалён (deleted). Эмитент может
        приостановить токен при~подозрении на~компрометацию,
        не~затрагивая саму карту и~другие токены.
\end{itemize}


\section{VTS, MDES, TSP НСПК --- три провайдера}
\label{sec:token-providers}

Каждая крупная карточная сеть управляет собственным Token Service
Provider:

\subsection*{VTS (\scheme{Visa} Token Service)}

VTS~--- платформа токенизации \scheme{Visa}, хранящая маппинг
\mbox{DPAN~$\leftrightarrow$~FPAN} в~централизованном Token Vault.
При~авторизации сеть \scheme{Visa} де-токенизирует DPAN до~FPAN,
добавляет метаданные (Assurance Level, TRID, domain) и~передаёт
эмитенту. VTS поддерживает провизионирование мобильных кошельков
(Apple Pay, Google Pay, Samsung Pay), card-on-file токенизацию
для~мерчантов и~IoT-устройства~\sidecite{visa-vts}.

\subsection*{MDES (Mastercard Digital Enablement Service)}

MDES~--- аналог VTS для~\scheme{Mastercard}. Архитектурно
схожа с~VTS: централизованный Token Vault, маппинг DPAN--FPAN,
поддержка множества Token Requestor. MDES также управляет
жизненным циклом токенов: при~перевыпуске карты (новый PAN
или~новый срок действия) MDES автоматически обновляет все
привязанные токены~--- мерчанту не~нужно запрашивать новые
данные у~держателя~\sidecite{mc-mdes}.

\subsection*{TSP НСПК}

Сервис токенизации НСПК обеспечивает токенизацию карт Мир.
Архитектура локальна: Token Vault расположен в~инфраструктуре
НСПК, де-токенизация выполняется ОПКЦ НСПК при~обработке
внутрироссийских транзакций. TSP НСПК использует ГОСТ-криптографию
для~защиты ключевого материала токенов и~поддерживает
провизионирование через Mir Pay и~банковские мобильные
приложения~\sidecite{nspk-tsp}.

\section{Сетевой токен vs PSP-токен}
\label{sec:token-network-vs-psp}

Прежде чем сетевая токенизация стала стандартом, PSP и~платёжные
шлюзы давно предлагали собственные токены~--- непрозрачные ссылки
(например, UUID), которые маппились на~PAN в~хранилище PSP. Эти
токены решают часть проблемы (мерчант не~хранит PAN), но~принципиально
отличаются от~сетевых по~нескольким
параметрам~\sidecite{emvco-payment-tokenisation}:

\begin{itemize}
  \item \textbf{Портативность.} PSP-токен валиден только внутри
        одного PSP: при~смене платёжного шлюза мерчант теряет все
        токены и~должен заново запрашивать данные карт у~держателей.
        Сетевой токен принадлежит карточной сети и~перемещается
        вместе с~мерчантом при~смене PSP.
  \item \textbf{PCI scope.} PSP-токен выводит PAN из~систем мерчанта,
        но~PSP сам по~себе хранит PAN в~своём vault и~остаётся
        в~полном PCI scope. Сетевой токен идёт дальше: настоящий
        PAN вообще не~попадает в~среду PSP (если~PSP использует
        сетевые токены end-to-end), что~сужает scope и~PSP.
  \item \textbf{Authorization rate.} Сетевой токен содержит
        метаданные (Assurance Level, domain restriction), которые
        видны эмитенту. Высокий Assurance Level~--- сигнал доверия,
        и~эмитенты одобряют токенизированные транзакции чаще,
        чем~обычные PAN-транзакции: разница составляет
        2--4~процентных пункта~\todocheck{разница в~approval rate
        между сетевыми токенами и~PAN~--- Visa/MC публичные данные}.
  \item \textbf{Lifecycle management.} При~перевыпуске карты
        (новый PAN, новый срок действия) сетевой токен обновляется
        автоматически через TSP~--- мерчанту не~нужно ничего делать.
        PSP-токен привязан к~конкретному PAN в~vault; при~перевыпуске
        мерчант или~PSP должен запустить Account Updater (VAU/ABU)
        или~запросить новые данные у~держателя.
  \item \textbf{Стоимость и~усилие.} Сетевая токенизация требует
        интеграции с~TSP карточной сети (или~через PSP, который
        поддерживает сетевые токены); PSP-токенизация~--- минимальна
        по~усилию, но~ограничена по~преимуществам.
\end{itemize}

\section{Де-токенизация: кто и когда видит настоящий PAN}
\label{sec:token-detokenization}

В~токенизированной транзакции настоящий PAN (FPAN) не~перемещается
по~всей цепочке~--- его~видят только те~участники, которым это
необходимо. Поток выглядит следующим образом:

\begin{enumerate}
  \item \textbf{Мерчант} отправляет транзакцию с~DPAN (сетевым
        токеном). Мерчант \emph{не~видит} FPAN~--- он~работает
        исключительно с~токеном.
  \item \textbf{Эквайер / PSP} передаёт транзакцию с~DPAN в~карточную
        сеть. В~зависимости от~модели, эквайер может не~знать, что
        перед ним токен, а~не~настоящий PAN.
  \item \textbf{Карточная сеть (TSP)} распознаёт DPAN по~BIN-диапазону,
        обращается к~Token Vault, извлекает FPAN и~метаданные
        (Assurance Level, domain, TRID). Сеть подставляет FPAN
        в~сообщение для~эмитента.
  \item \textbf{Эмитент} получает FPAN плюс индикаторы токенизации
        (token present, Assurance Level, TRID)~--- и~видит полную
        картину: какая карта, через какой кошелёк, с~каким уровнем
        доверия.
\end{enumerate}

В~клиринге правила различаются по~сетям: \scheme{Visa} может
передавать FPAN в~клиринговом файле (TC33), тогда как~в~некоторых
сценариях DPAN сохраняется до~клиринга. Для~мерчанта это прозрачно~---
он~работает с~токеном на~всех этапах.


\section{Токенизация и 3-D Secure}
\label{sec:token-and-3ds}

Токенизация и~3-D~Secure решают разные задачи (идентификация
vs~аутентификация), но~работают в~паре. При~добавлении карты
в~мобильный кошелёк TSP может запросить step-up аутентификацию
через 3-D~Secure~--- это~повышает Assurance Level токена. При~e-commerce
транзакции с~card-on-file мерчант одновременно отправляет сетевой
токен (вместо PAN) и~3DS-данные (результат аутентификации
держателя)~--- эмитент получает и~подтверждение личности (3DS),
и~подтверждение подлинности credentials
(токен с~Assurance Level).

Подробно о~3-D~Secure~--- в~главе~\ref{ch:3ds}; о~влиянии
токенизации на~PCI scope~--- в~главе~\ref{ch:pci-dss}.


\section*{Итоги главы}

\begin{itemize}
  \item Токенизация заменяет настоящий PAN (FPAN) суррогатом
        (DPAN), ограниченным конкретным доменом (NFC, e-commerce,
        конкретный мерчант); утечка токена не~эквивалентна утечке PAN.
  \item Каждый сетевой токен несёт атрибуты: Token Requestor (кто~запросил),
        Assurance Level (уровень доверия), domain restriction (контекст
        использования), lifecycle (создан, активен, приостановлен, удалён).
  \item Три TSP~--- VTS (\scheme{Visa}), MDES (\scheme{Mastercard}),
        TSP~НСПК (Мир)~--- архитектурно аналогичны (Token Vault,
        маппинг DPAN--FPAN), но~управляются независимо.
  \item Сетевой токен превосходит PSP-токен по~портативности
        (не~привязан к~PSP), PCI scope (FPAN не~попадает к~мерчанту),
        authorization rate (+2--4~п.п.) и~lifecycle management
        (автообновление при~перевыпуске карты).
  \item Де-токенизация происходит в~карточной сети: мерчант и~эквайер
        видят только DPAN; эмитент получает FPAN плюс метаданные токена.
  \item Токенизация и~3-D~Secure работают в~паре: токен идентифицирует
        credentials, 3DS аутентифицирует держателя; вместе они~дают
        эмитенту максимальную уверенность в~легитимности транзакции.
\end{itemize}
