\section{Симметричная криптография: 3DES и AES}
\label{sec:crypto-symmetric}

В~симметричной криптографии один и~тот~же ключ используется
для~шифрования и~расшифрования: отправитель и~получатель должны
заранее договориться о~секретном ключе и~хранить его в~тайне.
Платёжная индустрия исторически строилась на~симметричных алгоритмах~---
они быстры, хорошо изучены и~отлично подходят для~задач, где обе
стороны принадлежат одной доверенной инфраструктуре (например,
эмитент и~его~карта).

\subsection*{3DES: рабочая лошадка унаследованных систем}

3DES (Triple DES, также TDEA~--- Triple Data Encryption Algorithm)~---
это трёхкратное применение алгоритма DES, разработанного в~1970-х~годах.
Одиночный DES с~56-битным ключом был взломан методом полного перебора
ещё в~конце 1990-х, но~тройное применение с~двумя или~тремя
независимыми ключами увеличивает эффективную длину ключа
до~112~или~168~бит соответственно, что~делает полный перебор
вычислительно непрактичным~\sidecite{emvco-book2}.

В~платёжной инфраструктуре 3DES встречается повсюду: вычисление CVV
на~магнитной полосе (глава~\ref{ch:card-data}), генерация
криптограммы ARQC на~чипе (глава~\ref{ch:emv}), шифрование
PIN-блока при~онлайн-авторизации, вычисление MAC (Message
Authentication Code) для~защиты целостности \iso{8583}-сообщений
в~канале между эквайером и~сетью. Десятилетия развития платёжной
инфраструктуры привели к~тому, что~3DES глубоко встроен в~firmware
терминалов, HSM-модули и~процессинговые системы, и~его замена~---
процесс медленный и~дорогостоящий.

\subsection*{AES: современная замена}

AES (Advanced Encryption Standard)~--- блочный шифр, принятый NIST
в~2001~году на~замену DES. AES оперирует 128-битными блоками
и~поддерживает длину ключа 128, 192 или~256~бит. По~сравнению
с~3DES он~быстрее, требует меньше вычислительных ресурсов
и~обеспечивает больший запас прочности: даже 128-битный ключ AES
находится далеко за~пределами возможностей перебора на~обозримую
перспективу~\sidecite{nist-sp800-57}.

В~платежах AES постепенно вытесняет 3DES. PCI требует миграцию
с~3DES на~AES для~шифрования PIN: стандарт PCI PIN Security
установил крайний срок вывода 3DES из~эксплуатации
в~этом контексте~\todocheck{дата вывода 3DES для~PIN-шифрования~---
PCI PIN Security Requirements, уточнить конкретный deadline}.
Новые терминалы и~HSM-модули поддерживают AES из~коробки, однако
миграция осложняется необходимостью сохранять обратную совместимость
с~парком устаревших устройств: пока хотя~бы один участник цепочки
не~поддерживает AES, всей цепочке приходится работать на~3DES.

\subsection*{MAC: защита целостности сообщений}

MAC (Message Authentication Code)~--- значение, вычисляемое по~содержимому
сообщения с~использованием разделяемого секретного ключа, позволяющее
получателю убедиться, что сообщение не~было изменено в~пути
и~отправлено именно тем, кто владеет ключом. В~\iso{8583} MAC
передаётся в~\de{64} (MAC, primary bitmap) или~\de{128} (MAC,
secondary bitmap) и~защищает содержимое авторизационного сообщения
на~участке между терминалом и~эквайером или~между эквайером и~сетью.

Два наиболее распространённых алгоритма MAC в~платежах~--- CBC-MAC
(на~основе 3DES или~AES в~режиме CBC) и~CMAC (усовершенствованный
вариант, рекомендованный NIST для~новых реализаций). CBC-MAC с~3DES
по-прежнему доминирует в~унаследованных системах, CMAC с~AES~---
в~новых развёртываниях.


\section{Асимметричная криптография: RSA и ECC}
\label{sec:crypto-asymmetric}

В~отличие от~симметричной криптографии, где один ключ на~двоих,
асимметричная использует пару ключей: открытый (public), который
можно свободно распространять, и~закрытый (private), который
хранится в~тайне. Данные, зашифрованные открытым ключом, можно
расшифровать только закрытым; подпись, созданная закрытым ключом,
можно проверить открытым. Эта асимметрия решает задачу, с~которой
симметричная криптография не~справляется: как~установить доверие
между сторонами, которые заранее не~обменивались
секретами~\sidecite{emvco-book2}.

\subsection*{RSA в EMV}

RSA~--- наиболее широко используемый асимметричный алгоритм
в~платёжной индустрии. В~главе~\ref{ch:emv} мы~видели, как~RSA
обеспечивает аутентификацию данных на~чипе: удостоверяющий центр
карточной сети (CA) подписывает сертификат эмитента, эмитент подписывает
сертификат карты (ICC), и~терминал проверяет всю цепочку
\mbox{CA~$\to$~Issuer~$\to$~ICC}, используя открытый ключ CA,
заранее загруженный в~терминал.

Длина ключа RSA в~EMV исторически составляла 1024~бит, что
к~середине 2010-х~годов стало считаться недостаточным: теоретические
оценки показывали, что~факторизация 1024-битного модуля может стать
практически осуществимой в~обозримом будущем. Карточные сети объявили
миграцию на~2048-битные ключи CA, и~к~настоящему времени все
основные схемы~--- \scheme{Visa}, \scheme{Mastercard} и~Мир~---
перешли на~2048~бит для~сертификатов CA и~эмитента. Ключи карты
(ICC) при~этом могут оставаться короче (например, 1408 или~1984~бит),
поскольку время жизни карты ограничено сроком
действия~\todocheck{текущие требования EMVCo к~минимальной длине
RSA-ключей CA, Issuer, ICC}.

\subsection*{ECC: короче, быстрее}

ECC (Elliptic Curve Cryptography)~--- семейство алгоритмов,
основанных на~задаче дискретного логарифмирования в~группе точек
эллиптической кривой. Главное преимущество ECC перед RSA~---
значительно меньшая длина ключа при~сопоставимом уровне безопасности:
256-битный ключ ECC примерно эквивалентен 3072-битному ключу RSA.
Меньший размер ключа означает более компактные сертификаты и~более
быстрые операции подписи и~верификации~--- что~критично для~чиповых
карт с~ограниченными вычислительными ресурсами и~особенно для~бесконтактных
транзакций, где~каждая миллисекунда на~счету.

EMVCo включил поддержку ECC в~спецификации контактных и~бесконтактных
транзакций, а~Мир использует ECC (в~форме ГОСТ 34.10, основанного
на~эллиптических кривых) в~платёжном приложении MPA. На~практике
миграция с~RSA на~ECC в~глобальных схемах идёт медленно~---
терминальный парк должен поддерживать новые алгоритмы, а~переходный
период требует одновременной поддержки обоих
стандартов~\sidecite{emvco-book2}.


\section{HSM: зачем нужен физически защищённый модуль}
\label{sec:crypto-hsm}

Любой криптографический алгоритм безопасен ровно настолько, насколько
безопасен ключ. Если~ключ хранится в~файле на~сервере, его~можно
украсть: прочитать из~памяти, извлечь из~дампа, скопировать
при~бэкапе. Аппаратный модуль безопасности (HSM, Hardware Security
Module)~--- это физически защищённое устройство, единственная задача
которого~--- хранить ключи и~выполнять криптографические операции,
не~выпуская ключевой материал за~пределы своего корпуса.

HSM спроектирован так, чтобы противостоять физическим атакам:
корпус оснащён датчиками вскрытия (tamper detection), которые
при~попытке проникновения уничтожают ключевой материал; чипы
защищены от~считывания электронным микроскопом (mesh layers);
побочные каналы (электромагнитное излучение, потребление тока)
минимизированы. Сертификация HSM выполняется по~стандарту
FIPS~140-2 (или~более новому FIPS~140-3): уровень~3 требует
tamper evidence и~tamper response, уровень~4~--- полной физической
защиты от~всех известных
атак~\todocheck{различие FIPS 140-2 Level 3 и Level 4 в~контексте
платёжных HSM~--- уточнить по~NIST SP 800-140}.

В~платёжной инфраструктуре HSM выполняет десятки операций:

\begin{itemize}
  \item Верификация криптограммы ARQC при~онлайн-авторизации:
        эмитент (или~его процессор) отправляет данные транзакции
        в~HSM, тот~деривирует сессионный ключ из~мастер-ключа
        эмитента, вычисляет ожидаемую криптограмму и~сравнивает.
  \item Генерация ARPC~--- ответной криптограммы эмитента.
  \item Верификация PIN: HSM расшифровывает PIN-блок, сравнивает
        с~эталоном и~возвращает \enquote{да/нет}, никогда не~выпуская
        PIN наружу в~открытом виде.
  \item PIN translation: перешифрование PIN-блока из~одного ключа
        в~другой при~передаче между зонами безопасности.
  \item Генерация и~проверка CVV/CVV2.
  \item Генерация и~загрузка ключей при~персонализации карт.
  \item Вычисление MAC для~защиты \iso{8583}-сообщений.
\end{itemize}

Основные производители платёжных HSM~--- Thales (серия payShield),
Utimaco (серия CryptoServer) и~Futurex (серия Excrypt). Каждый
крупный процессинговый центр содержит ферму из~десятков HSM,
обеспечивающих тысячи криптографических операций в~секунду~---
каждая авторизация чиповой карты проходит через HSM минимум
дважды (верификация ARQC и~генерация ARPC).


\section{Иерархия ключей: от мастер-ключа до сессионного}
\label{sec:crypto-key-hierarchy}

В~платёжной криптографии ключи организованы в~иерархию~--- дерево,
в~котором каждый уровень порождает ключи следующего уровня
через процедуру деривации (key derivation). Смысл иерархии~---
компартментализация: компрометация ключа нижнего уровня не~ведёт
к~компрометации всей системы~\sidecite{nist-sp800-57}.

Типичная иерархия для~эмитентской криптографии EMV выглядит
следующим образом:

\begin{enumerate}
  \item \textbf{Мастер-ключ (Issuer Master Key, IMK)}~--- секретный
        ключ эмитента, хранящийся исключительно в~HSM. Для~каждого
        типа операции может существовать отдельный мастер-ключ:
        IMK-AC (для~криптограмм), IMK-SMI (для~Secure Messaging
        Integrity), IMK-SMC (для~Secure Messaging Confidentiality).
  \item \textbf{Ключ карты (Card Key)}~--- производный от~мастер-ключа
        с~использованием PAN и~порядкового номера карты (PAN Sequence
        Number) в~качестве входных данных для~деривации. Ключ карты
        записывается в~чип при~персонализации.
  \item \textbf{Сессионный ключ (Session Key)}~--- производный от~ключа
        карты с~использованием ATC (Application Transaction Counter)
        в~качестве диверсификатора. Сессионный ключ уникален для~каждой
        транзакции и~используется для~вычисления криптограммы.
\end{enumerate}

% \label{fig:crypto-key-hierarchy}

Деривация работает как односторонняя функция: зная мастер-ключ,
можно вычислить ключ любой карты; зная ключ карты, можно вычислить
сессионный ключ для~любого ATC. Но~обратная операция невозможна:
компрометация сессионного ключа одной транзакции не~позволяет
восстановить ключ карты (и~тем~более мастер-ключ), а~компрометация
ключа одной карты не~раскрывает ключи других карт, поскольку
деривация по~PAN даёт различные результаты для~разных
номеров~\sidecite{emvco-book2}.

Именно поэтому защита мастер-ключа~--- задача первостепенной важности:
его~компрометация означает компрометацию всех карт данного эмитента.
Мастер-ключ никогда не~покидает HSM в~открытом виде и~загружается
при~начальной настройке через формальную процедуру~--- Key
Ceremony, о~которой речь пойдёт в~разделе~\ref{sec:crypto-key-ceremony}.


\section{DUKPT: почему компрометация текущего ключа не~раскрывает прошлое}
\label{sec:crypto-dukpt}

Иерархия ключей EMV (мастер-ключ~$\to$~ключ карты~$\to$~сессионный
ключ) хорошо работает для~чиповых карт, где~ключевой материал
защищён внутри чипа. Но~POS-терминал~--- устройство иного класса
защищённости: он~стоит на~прилавке магазина, его~можно украсть,
разобрать и~попытаться извлечь ключ из~памяти. Если~терминал хранит
мастер-ключ и~деривирует из~него сессионные ключи по~счётчику,
компрометация мастер-ключа раскроет все~--- и~прошлые, и~будущие
транзакции.

DUKPT (Derived Unique Key Per Transaction)~--- протокол, разработанный
специально для~решения этой проблемы. Его~ключевое свойство~---
\emph{forward secrecy}: компрометация текущего ключа транзакции
не~позволяет вычислить ключи прошлых
транзакций~\sidecite{ansi-x9-24}.

Протокол работает следующим образом:

\begin{enumerate}
  \item Центральная система (HSM эквайера) хранит BDK (Base Derivation
        Key)~--- корневой ключ, из~которого деривируются ключи
        для~всех терминалов.
  \item При~инициализации терминала из~BDK деривируется IPEK (Initial
        PIN Encryption Key)~--- начальный ключ конкретного терминала,
        с~использованием его серийного номера (KSN~--- Key Serial
        Number) как~диверсификатора.
  \item IPEK загружается в~терминал (при~key injection), после чего
        терминал порождает из~IPEK дерево производных ключей~--- набор
        \enquote{будущих ключей}, из~которых при~каждой транзакции
        выбирается следующий по~счётчику.
  \item После использования ключа для~транзакции терминал
        \emph{уничтожает} его~--- он~больше не~нужен и~не~может быть
        восстановлен.
  \item IPEK также уничтожается после генерации дерева производных
        ключей: в~памяти терминала остаются только ещё не~использованные
        ключи.
\end{enumerate}

Стрелка вниз с подписью «деривация по KSN» к IPEK (в терминале,

к IPEK. Внизу: подпись «Каждый транзакционный ключ уникален и
уничтожается после использования». Стиль: дерево, растущее вниз,
использованные ключи перечёркнуты, неиспользованные --- залиты}
% \label{fig:crypto-dukpt}

Почему это работает? Дерево деривации DUKPT устроено так, что
каждый транзакционный ключ порождается из~промежуточных узлов
дерева через одностороннюю операцию. Даже если злоумышленник
извлечёт текущий транзакционный ключ из~памяти терминала, он~не~сможет
подняться по~дереву к~промежуточным узлам и~вычислить ключи прошлых
транзакций. Будущие ключи тоже не~скомпрометированы~--- они~деривируются
из~других ветвей дерева. Единственный ключ, компрометация которого
раскрывает всё,~--- BDK, но~он~хранится исключительно в~HSM эквайера
и~никогда не~покидает его~периметр.


\section{PIN-блок: форматы ISO 0, 1, 3, 4}
\label{sec:crypto-pin-block}

Когда держатель вводит PIN на~клавиатуре терминала для~онлайн-авторизации,
PIN не~передаётся в~открытом виде~--- он~преобразуется в~PIN-блок:
структурированное восьмибайтовое значение, которое затем шифруется
ключом (DUKPT-ключом терминала или~зональным ключом) и~отправляется
в~\de{52} авторизационного сообщения. Стандарт \iso{9564} определяет
несколько форматов PIN-блока, из~которых в~платежах используются
четыре~\sidecite{iso-9564}.

\subsection*{Format 0 (ISO 9564 Format 0)}

Наиболее распространённый унаследованный формат. PIN-блок формируется
как XOR двух восьмибайтовых полей: поле PIN (код формата~\hex{0},
длина PIN, цифры PIN, заполнение~\hex{F}) и~поле PAN (четыре нуля,
12~правых цифр PAN без контрольной цифры). Результат XOR шифруется
ключом.

Зависимость от~PAN~--- одновременно и~особенность, и~уязвимость
Format~0: если~злоумышленник знает PAN (а~он~часто известен из~тех~же
логов), задача восстановления PIN из~перехваченного PIN-блока
упрощается. Кроме того, при~смене PAN (например, при~перевыпуске
карты) PIN-блок меняется, даже если сам PIN остался прежним.

\subsection*{Format 1}

В~Format~1 поле PAN заменяется случайным числом~---
непредсказуемым восьмибайтовым значением, которое терминал
генерирует отдельно для~каждой транзакции, и~именно с~этим
случайным полем выполняется XOR с~полем PIN. Принципиальное
следствие~--- отсутствие зависимости от~PAN: один и~тот~же
PIN при~одном и~том~же PAN будет давать разный PIN-блок
при~каждой транзакции, делая атаки на~основе известного номера
карты неприменимыми. Оборотная сторона~--- получатель PIN-блока
должен знать случайное поле, использованное при~его~формировании,
а~значит, оно должно передаваться в~сообщении или~по~согласованному
каналу, что~усложняет протокол. Именно это~усложнение объясняет,
почему Format~1 редко встречается в~карточных транзакциях
и~применяется преимущественно в~отдельных межбанковских системах,
где~структура сообщений~позволяет его~разместить~\sidecite{iso-9564}.

\subsection*{Format 3}

Format~3 по~структуре идентичен Format~0: поле PIN точно
так~же XOR-ится с~производным от~PAN полем, и~результат
шифруется тем~же ключом. Единственное, но~существенное отличие:
позиции, не~занятые цифрами PIN, заполняются не~фиксированным
\hex{F}, а~случайными значениями из~диапазона \hex{A}--\hex{F}.
Это~означает, что~два PIN-блока Format~3 для~одного и~того~же
PIN и~PAN из~разных транзакций будут отличаться, поскольку
заполнение каждый раз уникально~--- что~существенно затрудняет
статистический анализ перехваченных блоков и~атаки по~известному
открытому тексту. Зависимость от~PAN при~этом сохраняется
в~полной мере: злоумышленник, знающий PAN, по-прежнему сужает
пространство перебора, а~смена PAN при~перевыпуске карты
изменяет PIN-блок. Format~3~--- умеренный шаг вперёд
относительно Format~0, применяемый в~системах, которые
не~могут перейти на~Format~4 по~техническим или~организационным
причинам~\sidecite{iso-9564}.

\subsection*{Format 4 (AES)}

Format~4~--- принципиально новый подход, разработанный для~AES.
PIN-блок формируется в~два этапа: сначала PIN и~PAN комбинируются
в~промежуточный блок (аналогично Format~0, но~с~16-байтовой длиной),
затем этот блок шифруется AES-256 в~режиме, обеспечивающем
дополнительную рандомизацию. Format~4 устраняет ряд криптографических
слабостей предыдущих форматов и~является обязательным
для~новых развёртываний по~стандарту PCI PIN
Security~\sidecite{pci-pin-security}.

\begin{important}
PCI PIN Security требует миграции с~Format~0 (3DES) на~Format~4
(AES) для~всех новых терминалов и~постепенного вывода 3DES
из~эксплуатации. Если~вы~проектируете новый процессинг или~обновляете
существующий~--- закладывайте поддержку Format~4 и~AES с~самого начала;
поддержка Format~0 нужна только для~обратной совместимости
с~унаследованным парком терминалов.
\end{important}

\subsection*{PIN translation}

Когда PIN-блок передаётся между зонами безопасности~--- например,
от~эквайера к~карточной сети и~далее к~эмитенту,~--- каждый участник
использует свой ключ шифрования. На~каждой границе HSM выполняет
PIN translation: расшифровывает PIN-блок одним ключом и~тут~же
зашифровывает другим, не~выпуская PIN в~открытом виде в~память
сервера. Операция атомарна и~происходит исключительно внутри HSM~---
это одна из~причин, по~которой HSM является обязательным элементом
любого узла, обрабатывающего PIN.


\section{Key Ceremony и Key Injection}
\label{sec:crypto-key-ceremony}

Мастер-ключ~--- самый ценный секрет в~платёжной инфраструктуре:
его~компрометация означает компрометацию всех производных ключей,
всех карт, всех транзакций. Процедура генерации, загрузки и~хранения
мастер-ключа называется Key Ceremony и~подчиняется строгим правилам,
исключающим возможность того, чтобы один человек когда-либо увидел
ключ целиком~\sidecite{pci-pin-security}.

\begin{practice}
Key Ceremony проходит в~защищённой комнате (secure room), оборудованной
контролем доступа, видеонаблюдением и~без~сетевых подключений к~внешнему
миру. Участвуют минимум три key custodian (хранителя компонентов ключа)
и~один security officer, контролирующий процедуру. Процесс выглядит
следующим образом:

\begin{enumerate}
  \item HSM генерирует мастер-ключ и~разбивает его на~три (или~более)
        компонента~--- ни~один компонент сам по~себе не~позволяет
        восстановить ключ.
  \item Каждый компонент записывается на~отдельный носитель (смарт-карту
        хранителя ключа или~бумажный конверт с~контролем вскрытия)
        и~передаётся своему хранителю.
  \item Хранители подтверждают получение, подписывая акт. Каждый
        хранитель хранит свой компонент независимо~--- в~разных сейфах,
        предпочтительно в~разных физических локациях.
  \item Для~загрузки ключа в~другой HSM (например, при~развёртывании
        резервного центра) хранители собираются вместе и~последовательно
        вводят свои компоненты; HSM восстанавливает полный ключ только
        внутри своего защищённого периметра.
\end{enumerate}

Два ключевых принципа: \emph{dual control}~--- ни~одно действие
с~ключом не~выполняется одним человеком, и~\emph{split knowledge}~---
ни~один человек не~знает полный ключ. Нарушение любого из~этих
принципов~--- инцидент безопасности, требующий перегенерации ключа.
\end{practice}

\subsection*{Key Injection в терминалы}

Ключи попадают в~POS-терминалы одним из~двух способов:

\begin{itemize}
  \item \textbf{Ручная инъекция (manual key injection)}~--- терминал
        привозят в~защищённое помещение (key injection facility),
        где~оператор с~помощью специального устройства (TRSM~--- Tamper
        Resistant Security Module, или~key loading device) загружает
        начальный ключ (IPEK для~DUKPT или~зональный ключ). Процедура
        трудоёмка и~плохо масштабируется, но~по-прежнему используется
        для~первоначальной инициализации.
  \item \textbf{Удалённая инъекция (Remote Key Injection, RKI)}~---
        ключ загружается в~терминал по~защищённому каналу через
        специальный сервер (RKI server). Терминал должен иметь
        предустановленный транспортный ключ или~сертификат,
        позволяющий установить безопасное соединение. RKI
        значительно упрощает логистику, особенно для~больших
        терминальных сетей~\todocheck{стандарт PCI на~RKI~---
        PCI PIN Security Requirement 18, уточнить}.
\end{itemize}


\section{ГОСТ в платёжном приложении Мир}
\label{sec:crypto-gost}

Российские требования к~защите информации в~платёжных системах,
закреплённые в~\law{382-П} и~\law{683-П}
(глава~\ref{ch:regulation}), предусматривают использование
сертифицированных средств криптографической защиты информации
(СКЗИ) на~основе российских стандартов ГОСТ~\sidecite{cbr-382p}.
Платёжная система Мир реализует эти требования в~своём платёжном
приложении MPA (Mir Payment Application), используя три основных
стандарта~\sidecite{gost-34-10}:

\begin{itemize}
  \item \textbf{ГОСТ~Р~34.10--2018}~--- алгоритм электронной
        цифровой подписи, основанный на~эллиптических кривых
        (аналог ECDSA). Используется для~аутентификации данных
        на~чипе~--- аналогично тому, как~RSA используется
        в~DDA/CDA у~\scheme{Visa} и~\scheme{Mastercard}.
  \item \textbf{ГОСТ~Р~34.11--2018} (Стрибог)~--- хеш-функция
        с~длиной выхода 256~или~512~бит. Используется при~вычислении
        подписей и~в~протоколах формирования ключей.
  \item \textbf{ГОСТ~Р~34.12--2018}~--- блочные шифры Кузнечик
        (128-битный блок, 256-битный ключ) и~Магма (64-битный блок,
        256-битный ключ). Кузнечик~--- современный алгоритм, Магма~---
        обратно совместимый с~предыдущим поколением ГОСТ~28147--89.
        Используются для~шифрования данных и~вычисления
        MAC~\sidecite{gost-34-12}.
\end{itemize}

На~практике MPA реализует \enquote{двойной стек}: внутри России
транзакции обрабатываются с~использованием ГОСТ-криптографии
(аутентификация данных, генерация криптограммы, защита каналов),
а~при~работе за~рубежом (через кобейджинговые карты Мир--UnionPay
или~через двойное приложение на~чипе) может использоваться стандартная
EMV-криптография на~RSA/3DES. Такая архитектура позволяет
соответствовать российским требованиям без~потери международной
совместимости~\sidecite{cbr-683p}.

\begin{practice}
Если~вы~разрабатываете процессинг, который обрабатывает карты Мир,
ваш HSM должен поддерживать ГОСТ-алгоритмы~--- а~не~только RSA/3DES/AES.
Не~все западные HSM поддерживают ГОСТ из~коробки: необходимо либо
использовать HSM российского производства (например, серии
от~компаний Актив, ИнфоТеКС), либо загружать ГОСТ-модули
на~поддерживающие это западные платформы. Требование к~ГОСТ~---
не~рекомендация, а~условие подключения к~ОПКЦ НСПК.
\end{practice}


\section*{Итоги главы}

\begin{itemize}
  \item В~платежах используются два класса криптографии: симметричная
        (3DES, AES~--- для~шифрования данных, вычисления криптограмм
        и~MAC) и~асимметричная (RSA, ECC~--- для~цифровых подписей,
        сертификатов и~обмена ключами).
  \item 3DES постепенно уступает AES: PCI требует миграции
        PIN-шифрования на~AES (Format~4), однако 3DES глубоко
        встроен в~унаследованную инфраструктуру и~будет присутствовать
        ещё годы.
  \item HSM~--- обязательный элемент любого узла, обрабатывающего
        ключи или~PIN: это~физически защищённое устройство, выполняющее
        криптооперации без~выпуска ключей наружу.
  \item Иерархия ключей (мастер-ключ~$\to$~ключ карты~$\to$~сессионный
        ключ) обеспечивает компартментализацию: компрометация нижнего
        уровня не~раскрывает верхний.
  \item DUKPT обеспечивает forward secrecy для~терминалов:
        компрометация текущего транзакционного ключа не~позволяет
        вычислить ключи прошлых транзакций.
  \item PIN-блок Format~4 (AES)~--- обязательный стандарт для~новых
        развёртываний; Format~0 (3DES)~--- только для~обратной
        совместимости с~унаследованным парком.
  \item Key Ceremony~--- формальная процедура генерации мастер-ключей
        с~принципами dual control и~split knowledge; Key Injection~---
        загрузка ключей в~терминалы (вручную или~удалённо через RKI).
  \item Платёжная система Мир использует ГОСТ-криптографию
        (34.10/34.11/34.12) для~внутрироссийских операций
        и~\enquote{двойной стек} для~международной совместимости;
        поддержка ГОСТ в~HSM~--- обязательное условие подключения
        к~НСПК.
\end{itemize}
