\chapter{EMV: зачем карте процессор}
\label{ch:emv}

\begin{kaobox}[title=Что вы узнаете из этой главы]
  В~главе~\ref{ch:card-data} мы~увидели, что магнитная полоса хранит
  статические данные, которые можно скопировать. Здесь мы~разберём
  технологию, которая решила эту проблему,~--- \term{emv}: чип с~процессором,
  способный вычислять криптограмму, уникальную для~каждой транзакции.
  Мы~пройдём весь путь от~выбора приложения на~карте до~генерации
  криптограммы, научимся читать TVR и~поймём, когда и~почему
  терминал возвращается к~магнитной полосе.
\end{kaobox}

Начнём с~простого вопроса: что именно должен уметь сделать чип такого,
чего принципиально не~может магнитная полоса? Ответ не~в~том, что чип
\enquote{умнее}, а~в~том, что каждый следующий шаг \term{emv} закрывает
конкретный класс подделки или злоупотребления. Поэтому дальше мы~будем
читать EMV-flow как цепочку защит, а~не~как набор формальных команд.


\section{От магнитной полосы к чипу}
\label{sec:emv-magstripe-to-chip}

Магнитная полоса, описанная в~главе~\ref{ch:card-data},~--- это
хранилище статических данных: PAN, срок действия, CVV и~сервисный код
записаны на~дорожке раз и~навсегда при~выпуске карты. Каждый
считыватель получает одну и~ту~же последовательность байтов, и~каждую
считанную копию невозможно отличить от~оригинала. Именно это свойство
сделало магнитную полосу идеальной мишенью для~мошенников: достаточно
один раз пропустить карту через скиммер~--- накладку на~терминал
или~банкомат, незаметно считывающую дорожку,~--- и~полученные данные
можно записать на~чистую карту-болванку, создав полноценный клон.

К~середине 1990-х~годов потери от~поддельных карт (counterfeit fraud)
стали настолько значительными, что три крупнейшие карточные организации
того времени~--- Europay, \scheme{Mastercard} и~\scheme{Visa}~---
объединили усилия для~разработки спецификации, которая заменила~бы
пассивную магнитную полосу активным элементом: микропроцессорным чипом,
способным выполнять криптографические операции. В~1996~году был
опубликован первый набор спецификаций \term{emv} (по~первым буквам названий
трёх организаций), а~в~1999~году для~управления стандартом была
создана \term{emvco}~--- организация, которая сегодня объединяет шесть
глобальных платёжных сетей: \scheme{Visa}, \scheme{Mastercard},
JCB, American Express, Discover и~UnionPay~\sidecite{emvco-book1}.

Ключевая идея \term{emv} проста: если карта может вычислять, она может
доказать свою подлинность. Вместо того чтобы передавать терминалу
статические данные, которые тот бездумно транслирует дальше, чиповая
карта вычисляет криптограмму~--- значение, зависящее от~суммы
транзакции, даты, валюты, непредсказуемого числа, сгенерированного
терминалом, и~секретного ключа, который никогда не~покидает чип.
Скопировать криптограмму бесполезно: она валидна только для~одной
конкретной транзакции; для~следующей потребуется вычислить новую,
а~без ключа это невозможно.

Однако одной спецификации недостаточно~--- нужен экономический стимул
для~миграции. Карточные сети нашли его в~механизме, который называется
liability shift (перенос ответственности за~мошенничество). До~внедрения
\term{emv} убытки от~counterfeit fraud распределялись по~правилам сети, и~эмитент
нередко нёс потери в~одиночку. После объявления liability shift правила
изменились: если мерчант не~поддерживает чип (использует magstripe-only
терминал), а~карта чиповая~--- ответственность за~мошенническую
транзакцию переносится на~эквайера (и,~через него, на~мерчанта).
Если~же мерчант поддерживает чип, а~карта не~чиповая~---
ответственность остаётся на~эмитенте. Этот двусторонний стимул
запустил миграцию: эквайеры обновляли терминалы, чтобы не~нести потери,
а~эмитенты выпускали чиповые карты, чтобы не~оставаться крайними
в~обратном сценарии~\sidecite{visa-core-rules}.

  временная шкала от 1996 до 2025 года. Горизонтальные полосы разного

  (синий для EMV, серый для magstripe-эры)}
% \label{fig:emv-migration-timeline}

Европа перешла на~чип раньше всех: liability shift \scheme{Visa}
и~\scheme{Mastercard} в~Европе наступил в~2005~году, и~к~2010-му
подавляющее большинство европейских карт и~терминалов были чиповыми.
Россия двигалась примерно на~десять лет позже Западной Европы, но~быстрее
США: к~2015~году основные российские банки уже выпускали чиповые карты,
а~крупные эквайеры обновили терминальный парк. Соединённые Штаты стали
последним крупным рынком: liability shift наступил 1~октября 2015~года,
и~переход затянулся из-за масштаба инфраструктуры~--- более десяти
миллионов торговых точек и~сотни миллионов карт~\sidecite{emvco-specs}.

Результаты миграции впечатляют: по~данным \scheme{Visa}, в~странах,
завершивших переход на~EMV, counterfeit fraud сократился
на~75--80~процентов~\todocheck{статистика снижения counterfeit fraud
  после EMV-миграции~--- Visa Chip Security Report или~EMVCo статистика}.
Однако фрод не~исчез~--- он~мигрировал в~канал card-not-present (CNP),
где~чип бесполезен, и~именно этот сдвиг породил потребность
в~3-D~Secure, которую мы~разберём в~главе~\ref{ch:3ds}.

Итак, \term{emv} не~устраняет мошенничество вообще; он~закрывает именно
проблему клонирования статических карточных данных в~card-present
среде и~перестраивает экономические стимулы так, чтобы рынок
действительно мигрировал на~новую модель. Дальше важно понять,
какими именно шагами эта защита достигается внутри одной транзакции.


\section{Архитектура чиповой транзакции}
\label{sec:emv-architecture}

Чип на~платёжной карте~--- это не~просто носитель данных, а~полноценный
микрокомпьютер, хотя и~чрезвычайно компактный. Внутри контактной
площадки размером шесть на~восемь миллиметров находится интегральная
схема (ICC, Integrated Circuit Card), содержащая процессор (CPU),
криптографический сопроцессор, оперативную память (RAM), постоянную
память (ROM, хранящую операционную систему и~платёжное приложение)
и~перезаписываемую память (EEPROM или~Flash, хранящую данные эмитента,
счётчики транзакций и~ключи)~\sidecite{emvco-book1}.

\todofig{Блок-схема архитектуры чипа платёжной карты. Прямоугольники:
  CPU (центральный), Crypto coprocessor (рядом), ROM (операционная система),
  EEPROM/Flash (данные, ключи, счётчики), RAM (рабочая память), I/O
  interface (контактная площадка). Стрелки: двунаправленные шины между
  CPU и остальными блоками; одна стрелка наружу от I/O к терминалу.
  Подпись внизу: «Контактная площадка ISO 7816». Стиль: технический,
  монохромный, без 3D-эффектов}
% \label{fig:emv-chip-architecture}

Терминал общается с~чипом по~протоколу, определённому стандартом
\iso{7816}: физический интерфейс (контакты C1--C8 на~площадке), протокол
передачи данных (T=0 или~T=1) и~формат команд~--- APDU (Application
Protocol Data Unit). Каждая команда APDU состоит из~заголовка (класс,
код инструкции, два параметра) и~необязательных данных; чип отвечает
командой APDU с~данными и~двухбайтовым кодом статуса (SW1~SW2):
\hex{9000}~означает успех, \hex{6A82}~--- файл не~найден,
\hex{6985}~--- условия использования не~выполнены, и~так
далее~\sidecite{emvco-book3}.

Важная особенность чиповой карты~--- она может хранить несколько
независимых платёжных приложений. На~одной физической карте могут
сосуществовать, например, приложение \scheme{Visa} Credit
и~приложение Мир Debit~--- каждое со~своим набором ключей,
сертификатов и~параметров. Какое приложение будет использовано
для~конкретной транзакции, определяется на~первом шаге EMV-flow~---
при~выборе приложения.

Весь EMV-flow контактной транзакции состоит из~пяти последовательных
фаз, каждая из~которых решает свою задачу; вместе они образуют
конечный автомат, в~котором результат каждой фазы влияет на~следующую:

\begin{enumerate}
  \item \textbf{Выбор приложения}~--- терминал и~карта согласовывают,
        какое платёжное приложение будет использовано.
  \item \textbf{Аутентификация данных}~--- терминал проверяет, что
        данные на~карте подлинны и~не~были подменены.
  \item \textbf{Верификация держателя}~--- карта или~терминал
        проверяют, что~операцию инициировал законный держатель
        (PIN, подпись или~иной метод).
  \item \textbf{Риск-менеджмент}~--- терминал и~карта решают, можно~ли
        одобрить транзакцию офлайн или~необходима онлайн-авторизация
        у~эмитента.
  \item \textbf{Генерация криптограммы}~--- карта вычисляет
        криптографическое значение, подтверждающее результат всех
        предыдущих проверок.
\end{enumerate}

В~следующих разделах мы~разберём каждую фазу подробно.


\section{Выбор приложения}
\label{sec:emv-app-selection}

Прежде чем начать транзакцию, терминал должен определить, какое
платёжное приложение на~карте он~будет использовать. Для~этого
существует двухэтапный механизм, определённый в~EMV Book~1:
сначала терминал запрашивает у~карты каталог приложений, затем
согласовывает конкретное приложение из~списка
кандидатов~\sidecite{emvco-book1}.

\begin{termdef}{AID}
  AID (Application Identifier) --- уникальный идентификатор платёжного
  приложения на чипе. Состоит из двух частей: RID (Registered
  Application Provider Identifier, 5~байт, идентифицирует карточную
  сеть) и PIX (Proprietary Application Identifier Extension, до~11~байт,
  идентифицирует конкретный продукт). Например, AID
  \hex{A0000000031010}~--- это \scheme{Visa} Credit/Debit
  (RID~\hex{A000000003}, PIX~\hex{1010}).
\end{termdef}

Терминал начинает с~отправки команды SELECT для~каталога \term{pse}
(Payment System Environment, имя файла \texttt{1PAY.SYS.DDF01})~---
или~\term{ppse} (\texttt{2PAY.SYS.DDF01}) для~бесконтактного интерфейса,
о~котором речь пойдёт в~главе~\ref{ch:contactless}. В~ответ карта
возвращает список записей (SFI и~номера записей), каждая из~которых
содержит \term{aid} одного из~установленных приложений. Терминал считывает
эти записи и~формирует \emph{список кандидатов}~--- приложений,
поддерживаемых и~картой, и~терминалом.

Если \term{pse} на~карте отсутствует (например, на~старых картах
или~в~некоторых эмитентских конфигурациях), терминал использует
альтернативный метод: последовательно посылает команды SELECT для~каждого
\term{aid} из~своего собственного списка поддерживаемых приложений и~проверяет,
отвечает~ли карта положительно.

Когда в~списке кандидатов оказывается несколько приложений, терминал
сортирует их по~приоритету (тег \hex{87}, Application Priority Indicator)
и~может либо автоматически выбрать приложение с~наивысшим приоритетом,
либо предложить выбор держателю~--- в~зависимости от~настроек
терминала и~правил эквайера. На~практике в~большинстве торговых точек
выбор происходит автоматически, и~держатель видит на~экране терминала
лишь название выбранного приложения: \enquote{VISA CREDIT},
\enquote{MIR DEBIT} или~аналогичное.

Несколько примеров распространённых \term{aid}:

\medskip
\begin{center}
  \small
  \begin{tabular}{ll}
    \textbf{AID}         & \textbf{Приложение}                \\[3pt]
    \hex{A0000000031010} & \scheme{Visa} Credit / Debit       \\
    \hex{A0000000041010} & \scheme{Mastercard} Credit / Debit \\
    \hex{A0000006581010} & Мир Debit                          \\
    \hex{A0000006582010} & Мир Credit                         \\
    \hex{A0000000651010} & JCB                                \\
  \end{tabular}
\end{center}
\medskip

После успешного выбора приложения терминал отправляет команду
\term{gpo} (Get Processing Options), которая сообщает карте, что~транзакция
начинается, и~передаёт ей~первичные данные: сумму, валюту, дату
и~другие параметры. Карта в~ответ возвращает \term{aip} (Application
Interchange Profile)~--- битовое поле, определяющее, какие функции
поддерживает приложение (\term{sda}, \term{dda}, \term{cda}, риск-менеджмент карты,
верификация эмитента), и~\term{afl} (Application File Locator)~--- список
файлов и~записей, которые терминал должен прочитать для~продолжения
транзакции~\sidecite{emvco-book3}.


\section{Аутентификация данных}
\label{sec:emv-data-auth}

Получив \term{aip} и~\term{afl}, терминал знает, какие файлы читать и~какой метод
аутентификации данных поддерживает карта. Цель аутентификации данных~---
убедиться, что информация, хранящаяся на~чипе, подлинна: она была
записана эмитентом при~персонализации и~не~была подменена
злоумышленником. \term{emv} определяет три метода аутентификации,
расположенных по~возрастанию надёжности: \term{sda}, \term{dda}
и~\term{cda}~\sidecite{emvco-book2}.

\subsection*{SDA --- статическая аутентификация данных}

\term{sda} (Static Data Authentication)~--- самый простой метод. При~выпуске
карты эмитент формирует цифровую подпись (на~основе RSA) над
определённым набором данных карты~--- и~эта подпись записывается
на~чип. При~каждой транзакции терминал считывает подписанные данные
и~подпись, затем проверяет её, используя цепочку сертификатов:
сертификат удостоверяющего центра (CA) карточной
сети~$\to$~сертификат эмитента~$\to$~подпись данных карты.

SDA доказывает, что данные не~были изменены после персонализации,
но~у~него есть принципиальный недостаток: подпись статична. Если
злоумышленник скопирует все данные с~чипа, включая подпись, на~другой
чип~--- терминал не~обнаружит подмену, поскольку подпись останется
валидной. По~этой причине \term{sda} защищает только от~модификации данных,
но~не~от~клонирования чипа, и~современные карточные сети практически
полностью отказались от~\term{sda} в~пользу динамических
методов~\todocheck{доля карт с~SDA в~мировом обращении~--- к~2020-м
  близка к~нулю для~новых эмиссий}.

\subsection*{DDA --- динамическая аутентификация данных}

\term{dda} (Dynamic Data Authentication) решает проблему клонирования: помимо
статической подписи эмитента, карта содержит собственную пару ключей
RSA~--- приватный ключ хранится в~защищённой памяти чипа, а~открытый
ключ заверен сертификатом эмитента. При~каждой транзакции терминал
отправляет карте случайное число (unpredictable number), карта подписывает
его своим приватным ключом и~возвращает подпись. Терминал проверяет
подпись, используя открытый ключ карты из~сертификата.

\todofig{Sequence diagram аутентификации DDA. Участники слева направо:
  Терминал, Карта (ICC). Шаги: (1) Терминал генерирует случайное число
  (UN). (2) Терминал отправляет INTERNAL AUTHENTICATE с UN.

  справа от диаграммы --- вертикальная шкала «Цепочка доверия: CA → Issuer → ICC».
  Стиль: чёрно-белый, строгий, с пунктирными линиями жизни}
% \label{fig:emv-dda-sequence}

Почему DDA устойчив к~клонированию? Приватный ключ карты хранится
в~области памяти чипа, которая физически недоступна для~чтения
извне~--- чип спроектирован так, чтобы ключ использовался только
внутри криптосопроцессора и~никогда не~покидал микросхему. Даже если
злоумышленник скопирует все читаемые данные с~чипа, без приватного
ключа он~не~сможет подписать случайное число, которое предъявит
терминал при~следующей транзакции. Подробнее о~криптографических
примитивах, стоящих за~RSA-подписями в~EMV, мы~поговорим
в~главе~\ref{ch:crypto}~\sidecite{emvco-book2}.

\subsection*{CDA --- комбинированная аутентификация}

\term{cda} (Combined DDA / Application Cryptogram Generation) объединяет
динамическую аутентификацию данных и~генерацию криптограммы в~один
шаг: карта подписывает своим приватным ключом не~только случайное
число терминала, но~и~саму криптограмму транзакции (\term{arqc}, \term{tc} или~\term{aac},
о~которых речь пойдёт ниже). Это гарантирует, что криптограмма
была вычислена именно этой картой и~не~была подменена в~процессе
передачи. \term{cda}~--- наиболее безопасный метод аутентификации данных
в~\term{emv}, и~он~является рекомендованным для~всех новых
эмиссий~\sidecite{emvco-book2}.


\section{Верификация держателя}
\label{sec:emv-cvm}

Аутентификация данных подтверждает, что карта подлинная. Но~подлинная
карта может оказаться в~чужих руках~--- украденная, найденная, забытая
на~прилавке. Верификация держателя (Cardholder Verification) решает
другую задачу: убедиться, что транзакцию инициирует законный владелец
карты, а~не~посторонний.

EMV определяет несколько методов верификации (CVM~--- Cardholder
Verification Method), и~карта хранит упорядоченный список этих методов
в~объекте CVM List~\sidecite{emvco-book3}. При~транзакции терминал
обходит список сверху вниз, проверяя условия применимости каждого метода
(например, \enquote{если сумма выше~X} или~\enquote{если терминал
  поддерживает PIN-pad}), и~выполняет первый метод, для~которого
условие истинно.

Основные методы верификации:

\begin{itemize}
  \item \textbf{Offline PIN (открытый текст)}~--- держатель вводит
        PIN на~клавиатуре терминала, терминал передаёт его чипу командой
        VERIFY, и~чип сравнивает введённое значение с~эталоном,
        хранящимся в~защищённой памяти. PIN никогда не~покидает
        периметр \enquote{терминал~+~чип}.
  \item \textbf{Offline PIN (зашифрованный)}~--- аналогичен открытому,
        но~PIN передаётся чипу в~зашифрованном виде с~использованием
        открытого ключа карты, что~защищает от~перехвата на~интерфейсе
        терминал--чип.
  \item \textbf{Online PIN}~--- PIN вводится на~клавиатуре терминала,
        шифруется в~PIN-блок (подробно о~форматах PIN-блока~---
        в~главе~\ref{ch:crypto}) и~отправляется в~авторизационном
        сообщении эмитенту через \de{52}. Чип в~проверке не~участвует~---
        верификацию выполняет хост эмитента.
  \item \textbf{Подпись}~--- кассир визуально сверяет подпись на~чеке
        с~образцом на~обратной стороне карты. Метод архаичный и~ненадёжный,
        но~по-прежнему встречается в~некоторых конфигурациях.
  \item \textbf{No CVM}~--- верификация не~требуется (обычно для~мелких
        сумм или~для~определённых категорий мерчантов, например
        вендинговых автоматов).
  \item \textbf{CDCVM (Consumer Device CVM)}~--- биометрия или~пароль
        на~мобильном устройстве вместо PIN на~терминале; этот метод
        применяется при~оплате телефоном и~подробно разбирается
        в~главе~\ref{ch:contactless}.
\end{itemize}

Если ни~один метод из~CVM List не~применим (например, терминал
не~имеет клавиатуры, а~карта требует PIN), результат верификации
фиксируется как \enquote{CVM failed}, и~это записывается
в~соответствующий бит TVR~--- что, в~свою очередь, влияет на~решение
о~криптограмме на~последнем шаге flow.

\begin{practice}
  При~отладке транзакций обращайте внимание на~тег \hex{9F34}
  (CVM Results) в~\de{55}: он~содержит три байта, описывающих
  использованный метод верификации, условие его применения и~результат.
  Если вы~видите, что~транзакция отклонена эмитентом с~\rc{55}
  (Incorrect PIN), но~CVM Results указывают на~подпись или~No~CVM,~---
  перед вами рассогласование между конфигурацией терминала
  и~ожиданиями эмитента, а~не~ошибка ввода PIN держателем.
\end{practice}


\section{Риск-менеджмент терминала и карты}
\label{sec:emv-risk-mgmt}

После аутентификации данных и~верификации держателя перед системой
встаёт главный вопрос: одобрить транзакцию офлайн (без обращения
к~эмитенту) или~отправить запрос онлайн? EMV предусматривает
двухуровневый риск-менеджмент~--- на~стороне терминала
и~на~стороне карты,~--- и~оба уровня вносят свои результаты
в~битовое поле TVR (Terminal Verification Results), которое затем
используется для~принятия итогового решения~\sidecite{emvco-book3}.

\subsection*{Риск-менеджмент терминала}

Терминал выполняет три вида проверок:

\begin{itemize}
  \item \textbf{Floor limit check}~--- если сумма транзакции превышает
        порог (floor limit), установленный эквайером для~данного типа
        терминала, транзакция должна быть отправлена онлайн.
        При~нулевом floor limit (что~является нормой для~большинства
        POS-терминалов сегодня) все транзакции идут онлайн.
  \item \textbf{Random transaction selection}~--- даже если сумма ниже
        floor limit, терминал с~определённой вероятностью (задаваемой
        конфигурацией) может потребовать онлайн-авторизацию. Это
        механизм статистического контроля, позволяющий эмитенту
        периодически \enquote{видеть} офлайн-транзакции.
  \item \textbf{Velocity checking}~--- терминал может проверить,
        не~превысила~ли карта лимит последовательных офлайн-транзакций
        (Lower Consecutive Offline Limit и~Upper Consecutive Offline
        Limit), записанный в~данных карты. Если~лимит достигнут~---
        транзакция отправляется онлайн.
\end{itemize}

\subsection*{Риск-менеджмент карты}

Карта ведёт собственный учёт: счётчик транзакций (ATC~--- Application
Transaction Counter, тег~\hex{9F36}), кумулятивные суммы
офлайн-транзакций, число последовательных офлайн-операций. Карта
сравнивает текущие значения с~лимитами, установленными эмитентом
при~персонализации, и~может самостоятельно потребовать
онлайн-авторизацию или~даже отклонить транзакцию, если лимиты
исчерпаны.

\subsection*{TAC и IAC --- матрица решений}

Результаты всех проверок собраны в~TVR~--- пятибайтовом поле, каждый
бит которого отражает итог конкретной проверки (подробный разбор TVR~---
в~разделе~\ref{sec:emv-tvr}). Но~TVR сам по~себе не~принимает
решения~--- для~этого существуют два набора масок:

\begin{itemize}
  \item \textbf{TAC (Terminal Action Codes)}~--- настраиваются
        эквайером и~определяют, какие биты TVR терминал считает
        основанием для~онлайн-авторизации (TAC Online), отклонения
        (TAC Denial) или~допуска офлайн (TAC Default).
  \item \textbf{IAC (Issuer Action Codes)}~--- записаны на~чипе
        эмитентом при~персонализации и~определяют аналогичные
        пороги с~точки зрения эмитента: IAC Online, IAC Denial,
        IAC Default.
\end{itemize}

Терминал выполняет побитовое AND между TVR и~каждой парой масок
(TAC/IAC). Если результат AND с~Denial-маской ненулевой~--- транзакция
отклоняется (карта генерирует AAC). Если результат AND с~Online-маской
ненулевой~--- транзакция отправляется онлайн (карта генерирует ARQC).
Если ни~одно условие не~сработало~--- транзакция может быть одобрена
офлайн (карта генерирует TC). На~практике большинство контактных
транзакций сегодня идут онлайн: floor limit установлен в~ноль,
и~TAC Online содержит маску, требующую онлайн-авторизации
практически для~любого состояния TVR.


\section{Криптограмма: ARQC, TC, AAC}
\label{sec:emv-cryptogram}

Криптограмма~--- кульминация EMV-транзакции: восьмибайтовое значение,
вычисленное чипом с~использованием секретного ключа, которое
одновременно фиксирует решение карты о~транзакции и~доказывает
подлинность чипа. В~зависимости от~результатов предыдущих фаз~---
аутентификации данных, верификации держателя и~риск-менеджмента~---
карта генерирует одну из~трёх криптограмм~\sidecite{emvco-book3}:

\begin{termdef}{ARQC}
  ARQC (Authorization Request Cryptogram) --- криптограмма, означающая
  \enquote{карта просит онлайн-авторизацию у~эмитента}. Это самый
  распространённый исход контактной EMV-транзакции: карта считает, что
  транзакция не~может быть одобрена офлайн (из-за настроек IAC, превышения
  лимитов или~требования терминала), и~формирует криптограмму для~отправки
  эмитенту.
\end{termdef}

\begin{termdef}{TC}
  TC (Transaction Certificate) --- криптограмма, означающая \enquote{карта
    одобряет транзакцию офлайн}. Эмитент не~участвует в~решении; TC
  отправляется в~клиринге как подтверждение того, что~транзакция была
  одобрена чипом.
\end{termdef}

\begin{termdef}{AAC}
  AAC (Application Authentication Cryptogram) --- криптограмма, означающая
  \enquote{карта отклоняет транзакцию}. Терминал не~должен продолжать
  операцию.
\end{termdef}

Все три криптограммы вычисляются одним и~тем~же алгоритмом~--- различается
лишь тип, записанный в~старших битах тега \hex{9F27} (Cryptogram
Information Data). Входными данными для~вычисления служат сумма
транзакции, валюта, дата, ATC (счётчик транзакций карты),
непредсказуемое число терминала и~TVR~--- набор параметров, гарантирующих,
что каждая криптограмма уникальна и~привязана к~конкретной транзакции.
Ключ, которым вычисляется криптограмма,~--- сессионный, производный
от~мастер-ключа эмитента через деривацию по~PAN и~ATC; подробно
об~иерархии ключей мы~поговорим в~главе~\ref{ch:crypto}.

Криптограмма передаётся в~авторизационном сообщении внутри \de{55}
под~тегом \hex{9F26}~--- именно тот тег, который мы~упоминали
в~главе~\ref{ch:iso8583} при~разборе EMV-данных в~\iso{8583}.

\subsection*{ARPC --- ответ эмитента}

Когда эмитент получает авторизационный запрос с~ARQC, он~выполняет
зеркальную операцию: вычисляет ожидаемую криптограмму по~тем~же
входным данным и~своей копии мастер-ключа и~сравнивает результат
с~полученным ARQC. Совпадение подтверждает, что криптограмму
вычислил подлинный чип с~правильным ключом. Затем эмитент принимает
решение об~одобрении или~отклонении транзакции и~формирует ARPC
(Authorization Response Cryptogram)~--- ответную криптограмму,
доказывающую чипу, что~ответ пришёл от~настоящего эмитента,
а~не~был подделан на~промежуточном узле~\sidecite{emvco-book2}.

Терминал передаёт ARPC чипу, и~карта проверяет его. Если~ARPC валиден
и~эмитент одобрил транзакцию, карта перегенерирует криптограмму~---
на~этот раз TC (Transaction Certificate), подтверждая офлайн, что
транзакция завершена успешно. Если~ARPC невалиден или~эмитент
отклонил транзакцию, карта генерирует AAC.

\subsection*{Issuer scripting}

Вместе с~ARPC эмитент может отправить чипу набор команд~--- issuer
scripts, передаваемых в~\de{71} (Script Template~1, выполняется
до~генерации финальной криптограммы) или~\de{72} (Script Template~2,
выполняется после). Через issuer scripts эмитент может обновить
параметры карты: сбросить офлайн-счётчик PIN-попыток, изменить
офлайн-лимиты, заблокировать приложение или~даже заблокировать всю
карту. Это мощный инструмент удалённого управления чипом, работающий
через стандартную платёжную инфраструктуру: эмитенту не~нужен прямой
канал к~карте~--- достаточно, чтобы держатель совершил любую
транзакцию через терминал~\sidecite{emvco-book3}.


\section{TVR: как читать битовое поле}
\label{sec:emv-tvr}

TVR (Terminal Verification Results)~--- пять байт (40~бит), в~которых
закодированы результаты всех проверок, выполненных терминалом и~картой
в~ходе EMV-транзакции. TVR передаётся в~\de{55} под~тегом \hex{95}
и~является одним из~самых полезных диагностических инструментов
при~разборе отклонённых или~аномальных
транзакций~\sidecite{emvco-book3}.

Каждый байт TVR отвечает за~свою группу проверок. Разберём наиболее
важные биты (нумерация~--- от~старшего бита к~младшему внутри каждого
байта):

\medskip
\textbf{Байт~1: аутентификация данных и~версии приложения}

\begin{itemize}
  \item Бит~8 (0x80)~--- Offline data authentication was not performed
        (аутентификация данных не~выполнялась).
  \item Бит~7 (0x40)~--- SDA failed (статическая аутентификация
        не~прошла).
  \item Бит~6 (0x20)~--- ICC data missing (на~чипе отсутствуют
        обязательные данные).
  \item Бит~5 (0x10)~--- Card appears on terminal exception file
        (карта в~стоп-листе терминала).
  \item Бит~4 (0x08)~--- DDA failed (динамическая аутентификация
        не~прошла).
  \item Бит~3 (0x04)~--- CDA failed.
\end{itemize}

\textbf{Байт~2: управление терминалом}

\begin{itemize}
  \item Бит~8 (0x80)~--- ICC and terminal have different application
        versions (версии приложения не~совпадают).
  \item Бит~7 (0x40)~--- Expired application (приложение просрочено).
  \item Бит~6 (0x20)~--- Application not yet effective (приложение
        ещё не~вступило в~силу).
  \item Бит~5 (0x10)~--- Requested service not allowed for card product
        (запрошенная услуга не~разрешена для~данного продукта).
  \item Бит~4 (0x08)~--- New card (первая транзакция после выпуска).
\end{itemize}

\textbf{Байт~3: верификация держателя}

\begin{itemize}
  \item Бит~8 (0x80)~--- Cardholder verification was not successful
        (верификация не~пройдена).
  \item Бит~7 (0x40)~--- Unrecognised CVM (метод верификации
        не~распознан терминалом).
  \item Бит~6 (0x20)~--- PIN Try Limit exceeded (исчерпан лимит
        попыток ввода PIN).
  \item Бит~5 (0x10)~--- PIN entry required and PIN pad not present
        or~not working (требуется PIN, но~клавиатура отсутствует).
  \item Бит~4 (0x08)~--- PIN entry required, PIN pad present, but
        PIN was not entered (PIN не~введён).
  \item Бит~3 (0x04)~--- Online PIN entered (введён онлайн-PIN).
\end{itemize}

\textbf{Байт~4: риск-менеджмент терминала}

\begin{itemize}
  \item Бит~8 (0x80)~--- Transaction exceeds floor limit
        (сумма выше floor limit).
  \item Бит~7 (0x40)~--- Lower consecutive offline limit exceeded
        (нижний лимит последовательных офлайн-транзакций превышен).
  \item Бит~6 (0x20)~--- Upper consecutive offline limit exceeded
        (верхний лимит превышен).
  \item Бит~5 (0x10)~--- Transaction selected randomly for online
        processing (случайный отбор).
  \item Бит~4 (0x08)~--- Merchant forced transaction online
        (мерчант принудительно отправил транзакцию онлайн).
\end{itemize}

\textbf{Байт~5: разное}

\begin{itemize}
  \item Бит~8 (0x80)~--- Default TDOL used (используется TDOL
        по~умолчанию).
  \item Бит~7 (0x40)~--- Issuer authentication failed (ARPC
        не~прошёл проверку).
  \item Бит~6 (0x20)~--- Script processing failed before final
        GENERATE AC (issuer script до~криптограммы завершился
        с~ошибкой).
  \item Бит~5 (0x10)~--- Script processing failed after final
        GENERATE AC (issuer script после криптограммы завершился
        с~ошибкой).
\end{itemize}

\begin{practice}
  Допустим, в~логе авторизации вы~видите TVR = \hex{0000080000}.
  Третий байт = \hex{08}: бит~4 установлен~--- \enquote{PIN entry
    required, PIN pad present, but PIN was not entered}. Это~означает,
  что~карта потребовала ввод PIN, клавиатура терминала работала,
  но~PIN не~был введён (держатель нажал \enquote{Отмена} или~кассир
  пропустил шаг). В~сочетании с~IAC Online, требующим онлайн-авторизацию
  при~неуспешной верификации, транзакция была отправлена онлайн, и~эмитент
  отклонил её~с~\rc{55} (Incorrect PIN)~--- хотя фактически PIN вообще
  не~вводился. Зная структуру TVR, вы~сразу видите причину и~не~тратите
  время на~поиск \enquote{бага} в~криптографии.
\end{practice}


\section{Fallback на магнитную полосу}
\label{sec:emv-fallback}

Не~каждая чиповая транзакция завершается успешно на~чипе. Бывает, что
контакт между терминалом и~площадкой чипа нестабилен (карта изношена,
загрязнена, терминал повреждён), или~приложение на~чипе не~отвечает,
или~происходит ошибка APDU-обмена. В~таких случаях терминал может
выполнить fallback~--- переключиться на~считывание магнитной полосы
и~провести транзакцию как обычную
magstripe-операцию~\sidecite{emvco-book4}.

Fallback бывает двух видов:

\begin{itemize}
  \item \textbf{Technical fallback}~--- терминал автоматически переходит
        на~магнитную полосу после неудачного чтения чипа (обычно после
        трёх попыток вставки карты). В~авторизационном сообщении это
        отражается специальным значением POS Entry Mode (\de{22}),
        которое сигнализирует эмитенту: \enquote{чип не~прочитался,
          используется полоса}.
  \item \textbf{Merchant-initiated fallback}~--- кассир намеренно
        проводит карту через магнитный считыватель, минуя чип. Это
        может быть следствием нетерпения (очередь, медленный терминал)
        или~злонамеренных действий, и~именно этот сценарий карточные
        сети стремятся минимизировать.
\end{itemize}

\begin{important}
  Fallback~--- это возврат к~заведомо менее безопасному каналу:
  статическим данным магнитной полосы вместо динамической криптограммы
  чипа. Карточные сети относятся к~fallback как к~исключению, а~не~норме,
  и~применяют серьёзные экономические стимулы для~его снижения.
  \scheme{Visa} и~\scheme{Mastercard} мониторят долю fallback-транзакций
  по~каждому эквайеру и~мерчанту; при~превышении порогов (обычно~1--2\%
  от~общего числа чиповых транзакций) эквайер получает предупреждение,
  а~затем штраф~\todocheck{актуальные пороги и~штрафы за~excessive
    fallback~--- Visa Core Rules / MC Rules}. Кроме того, liability shift
  при~fallback-транзакции работает против мерчанта: если чиповая карта
  была проведена через магнитную полосу и~оказалась мошеннической,
  ответственность несёт эквайер.
\end{important}

Последний рубеж защиты при~fallback~--- механизм iCVV, который мы~уже
разбирали в~главе~\ref{ch:card-data}: чиповые данные Track~2 содержат
другой сервисный код и, следовательно, другое значение CVV. Если
злоумышленник скопирует Track~2 из~чипа (EMV equivalent data)
и~попытается записать его на~магнитную полосу клона, эмитент обнаружит
несоответствие CVV и~отклонит транзакцию. Однако iCVV защищает только
от~этого конкретного вектора атаки~--- если злоумышленник скопирует
данные непосредственно с~магнитной полосы (через скиммер), iCVV
не~поможет, и~именно поэтому полный отказ от~magstripe~--- конечная
цель миграции на~EMV~\sidecite{visa-core-rules}.


\section*{Итоги главы}

\begin{itemize}
  \item EMV заменил статические данные магнитной полосы динамической
        криптограммой: чип вычисляет уникальное значение для~каждой
        транзакции, и~копирование данных не~позволяет создать
        работающий клон.
  \item Миграцию на~EMV ускорил механизм liability shift: сторона,
        не~поддерживающая чип, несёт ответственность за~мошенничество.
        В~странах с~завершённой миграцией counterfeit fraud сократился
        на~75--80\%.
  \item EMV-транзакция проходит пять фаз: выбор приложения (AID),
        аутентификация данных (SDA/DDA/CDA), верификация держателя
        (PIN, подпись, CDCVM), риск-менеджмент (TVR, TAC, IAC)
        и~генерация криптограммы (ARQC/TC/AAC).
  \item Три криптограммы~--- три исхода: ARQC (\enquote{отправить
          онлайн эмитенту}), TC (\enquote{одобрить офлайн}),
        AAC (\enquote{отклонить}).
  \item TVR~--- пятибайтовое поле, побитово описывающее результаты
        всех проверок; в~сочетании с~TAC/IAC масками оно определяет
        тип криптограммы.
  \item Fallback на~магнитную полосу~--- вынужденное исключение при~отказе
        чипа; карточные сети штрафуют за~избыточный fallback, а~iCVV
        блокирует попытки записи чиповых данных на~поддельную полосу.
  \item Успех EMV в~борьбе с~counterfeit fraud привёл к~миграции
        мошенничества в~канал card-not-present, что~породило
        потребность в~3-D~Secure (глава~\ref{ch:3ds}).
\end{itemize}
