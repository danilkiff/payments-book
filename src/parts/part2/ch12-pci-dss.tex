\label{ch:pci-dss}

\begin{kaobox}[title=Что вы узнаете из этой главы]
PCI~DSS~--- не~набор из~трёхсот случайных контролей, а~логическая
рамка, построенная вокруг одной идеи: свести к~минимуму места,
где~хранятся карточные данные, и~защитить оставшиеся. Здесь мы~разберём
логику двенадцати требований, научимся определять границы CDE,
увидим, как~токенизация, P2PE и~iFrame сокращают scope с~SAQ~D
(330~контролей) до~SAQ~A (30), разберём болезненные новшества
PCI~DSS~4.0.1 и~проведём мост к~российским требованиям \law{382-П}
и~\law{683-П}.
\end{kaobox}


\section{Логика двенадцати требований}
\label{sec:pci-logic}

PCI~DSS (Payment Card Industry Data Security Standard) определяет
требования к~защите данных платёжных карт для~любой организации,
которая хранит, обрабатывает или~передаёт cardholder data. Стандарт
разрабатывается и~поддерживается PCI SSC (Security Standards Council),
основанным пятью глобальными карточными
сетями~\sidecite{pci-dss-4}.

Двенадцать требований PCI~DSS не~являются произвольным списком~---
они организованы в~шесть групп, каждая из~которых защищает
определённый слой:

\begin{enumerate}
  \item \textbf{Безопасная сеть и~системы} (Req~1--2).
        Требование~1: установить и~поддерживать сетевые средства
        безопасности (firewalls, segmentation). Требование~2:
        применять безопасные конфигурации ко~всем компонентам
        (не~использовать заводские пароли, отключать неиспользуемые
        сервисы).
  \item \textbf{Защита данных карт} (Req~3--4).
        Требование~3: защитить хранимые данные (шифрование PAN,
        запрет хранения SAD~--- как~мы~разбирали
        в~главе~\ref{ch:card-data}). Требование~4: шифровать
        передачу cardholder data по~открытым сетям (TLS).
  \item \textbf{Управление уязвимостями} (Req~5--6).
        Требование~5: защитить все системы от~вредоносного ПО.
        Требование~6: разрабатывать и~поддерживать безопасные
        системы и~приложения (secure SDLC, управление патчами,
        защита веб-приложений).
  \item \textbf{Контроль доступа} (Req~7--9).
        Требование~7: ограничить доступ к~данным карт по~принципу
        need-to-know. Требование~8: идентифицировать пользователей
        и~аутентифицировать доступ (MFA). Требование~9: ограничить
        физический доступ к~данным карт.
  \item \textbf{Мониторинг и~тестирование} (Req~10--11).
        Требование~10: журналировать и~мониторить все обращения
        к~сетевым ресурсам и~данным карт. Требование~11:
        регулярно тестировать безопасность (сканы уязвимостей,
        penetration testing).
  \item \textbf{Политика информационной безопасности} (Req~12).
        Поддерживать политику, охватывающую все аспекты
        информационной безопасности для~всего персонала.
\end{enumerate}

Ключевой инсайт: требования~--- концентрическая оборона, а~не~чеклист.
Если~данные карт не~попадают в~вашу среду (благодаря токенизации,
P2PE или~hosted payment page), большинство требований к~вам
просто не~применяются~--- и~именно в~этом состоит стратегия
минимизации scope.


\section{CDE: как определить границы}
\label{sec:pci-cde}

\begin{termdef}{CDE}
CDE (Cardholder Data Environment) --- совокупность людей, процессов
и~технологий, которые хранят, обрабатывают или~передают cardholder
data (CHD) или~sensitive authentication data (SAD), а~также
все~системы и~сегменты сети, напрямую подключённые к~ним
или~влияющие на~их~безопасность.
\end{termdef}

Определение границ CDE~--- самая сложная и~критически важная задача
при~PCI-комплаенсе. Ошибка в~определении scope означает либо
недозащищённость (данные карт оказываются за~пределами защитного
периметра), либо избыточность (организация защищает системы,
которые никогда не~видят карточных данных, тратя ресурсы впустую).

CDE включает три категории систем:

\begin{itemize}
  \item \textbf{Системы, обрабатывающие CHD/SAD}~--- серверы
        авторизации, базы данных с~PAN, терминалы~--- очевидно
        в~scope.
  \item \textbf{Connected-to systems}~--- системы, напрямую
        подключённые к~CDE по~сети, даже если они~сами не~работают
        с~карточными данными (например, DNS-сервер в~одном сегменте
        с~платёжным сервером).
  \item \textbf{Security-impacting systems}~--- системы, влияющие
        на~безопасность CDE: серверы аутентификации, системы
        мониторинга, антивирусное ПО, NTP-серверы.
\end{itemize}

Сетевая сегментация~--- основной инструмент ограничения CDE:
если~карточные данные обрабатываются в~изолированном сегменте сети,
отделённом от~остальной инфраструктуры файрволом с~правилами
deny-by-default, то~только этот сегмент (и~connected-to системы)
попадают в~scope. Без~сегментации в~scope оказывается \emph{вся}
сеть организации.

\section{Scope reduction: токенизация, P2PE, iFrame}
\label{sec:pci-scope-reduction}

Если CDE определяет, что~в~scope, то~scope reduction~--- стратегия
уменьшения того, что~в~этот scope попадает. Четыре основных
механизма:

\subsection*{Токенизация}

Если мерчант использует сетевые токены
(глава~\ref{ch:tokenization}), настоящий PAN никогда не~попадает
в~его~системы~--- мерчант работает только с~DPAN. Поскольку DPAN
не~является CHD (он~бесполезен вне~контекста конкретного домена),
системы мерчанта, обрабатывающие только DPAN, могут быть выведены
из~CDE. PSP-токенизация (когда PSP хранит PAN в~своём vault)
выводит из~scope системы мерчанта, но~не~PSP.

\subsection*{P2PE (Point-to-Point Encryption)}

P2PE~--- стандарт PCI, при~котором данные карты шифруются
непосредственно в~терминале (в~момент считывания карты) и~расшифровываются
только в~HSM на~стороне эквайера или~процессора. Мерчант физически
не~может прочитать карточные данные~--- через его~инфраструктуру
проходит только зашифрованный blob. Вся~среда мерчанта (серверы,
сеть, рабочие станции) выходит из~scope, а~терминал оценивается
по~стандарту P2PE, а~не~PCI~DSS~\sidecite{pci-p2pe}.

\subsection*{Hosted payment fields (iFrame)}

Для~e-commerce: вместо того чтобы собирать данные карты на~своей
платёжной странице, мерчант встраивает iFrame или~hosted fields,
отрендеренные PSP. Карточные данные вводятся в~элемент, принадлежащий
PSP, и~никогда не~касаются серверов или~JavaScript мерчанта.
Мерчант получает от~PSP токен (payment token) и~работает только
с~ним.

\subsection*{Redirect (hosted payment page)}

Максимальное scope reduction: при~оплате держатель перенаправляется
на~страницу PSP, вводит карточные данные \emph{целиком на~стороне
PSP} и~возвращается на~сайт мерчанта после завершения. Мерчант
вообще не~контактирует с~карточными данными. Минус~--- разрыв
пользовательского опыта при~редиректе.

Количественный эффект scope reduction значителен: организация,
не~применяющая ни~одного из~этих механизмов, оценивается по~SAQ~D
($\sim$330~контролей); организация с~полным redirect~--- по~SAQ~A
($\sim$30~контролей). Разница~--- порядок: не~просто меньше
документации, а~принципиально иной масштаб усилий.

\section{SAQ A vs SAQ D: разница в 300 контролей}
\label{sec:pci-saq}

SAQ (Self-Assessment Questionnaire)~--- инструмент самооценки,
который PCI~SSC предоставляет организациям для~валидации
соответствия PCI~DSS. Тип SAQ определяется тем, как~организация
обрабатывает карточные данные~\sidecite{pci-saq}:

\begin{itemize}
  \item \textbf{SAQ~A}~--- для~мерчантов, полностью передавших
        обработку карточных данных третьей стороне (redirect
        или~iFrame, без~электронного хранения/обработки/передачи
        CHD). $\sim$30~вопросов.
  \item \textbf{SAQ~A-EP}~--- для~e-commerce мерчантов, использующих
        hosted payment fields (iFrame), но~чей~веб-сервер
        потенциально влияет на~безопасность платёжной страницы.
        $\sim$140~вопросов.
  \item \textbf{SAQ~B}~--- для~мерчантов с~импринтерами или~автономными
        dial-up терминалами (без~электронного хранения CHD).
  \item \textbf{SAQ~B-IP}~--- для~мерчантов с~PTS-сертифицированными
        терминалами, подключёнными к~PSP по~IP.
  \item \textbf{SAQ~C-VT}~--- для~мерчантов, использующих
        виртуальный терминал (веб-интерфейс PSP) для~ручного ввода.
  \item \textbf{SAQ~C}~--- для~мерчантов с~платёжными приложениями,
        подключёнными к~интернету.
  \item \textbf{SAQ~D}~--- для~всех остальных: мерчантов, хранящих
        CHD, или~сервис-провайдеров. Полный набор:
        $\sim$330~вопросов.
\end{itemize}

Выбор SAQ~--- не~формальность: он~определяет объём работы
по~документации, тестированию и~ежеквартальным сканам. Для~многих
мерчантов переход с~SAQ~D на~SAQ~A (через внедрение hosted payment
page или~iFrame + токенизацию) окупается за~один цикл
комплаенса~--- экономия на~сканах, penetration testing
и~внутренних ресурсах превышает стоимость интеграции.


\section{PCI DSS 4.0.1: новые требования, которые причиняют боль}
\label{sec:pci-401}

Версия~4.0.1 (актуальная на~момент написания) ввела ряд требований,
которые существенно повышают планку для~организаций,
обрабатывающих карточные данные~\sidecite{pci-dss-4}.

\begin{important}
Два требования, вызвавших наибольшую дискуссию в~индустрии:

\textbf{Requirement~6.4.3: инвентаризация и~авторизация скриптов
на~платёжной странице.} Каждый JavaScript, загружаемый на~страницу,
где~вводятся карточные данные, должен быть инвентаризирован,
обоснован и~авторизован. Для~e-commerce это~болезненно: типичная
платёжная страница подгружает десятки скриптов~--- аналитику,
A/B-тестирование, tag manager, чат-виджеты,~--- и~каждый из~них
теперь требует документирования и~контроля целостности
(Content Security Policy, Subresource Integrity). Требование
направлено против атак типа Magecart, при~которых злоумышленник
внедряет вредоносный скрипт на~платёжную страницу для~перехвата
карточных данных.

\textbf{Requirement~11.3.1.1: аутентифицированные внутренние сканы
уязвимостей.} Внутренние сканы должны выполняться с~аутентификацией
(с~учётной записью, позволяющей сканеру видеть конфигурацию
и~уязвимости, недоступные при~неаутентифицированном сканировании).
Это~увеличивает объём выявляемых уязвимостей и~усилия по~их~устранению.
\end{important}

Другие значимые изменения 4.0.1:

\begin{itemize}
  \item MFA (Multi-Factor Authentication) обязательна для~всех
        обращений к~CDE, а~не~только для~удалённого доступа.
  \item Шифрование PAN требуется даже при~передаче по~доверенным
        внутренним сетям (ранее допускалось передавать PAN
        в~открытом виде внутри сегментированной сети).
  \item Targeted Risk Analysis~--- организация должна документировать
        обоснование периодичности для~контролей, где~стандарт
        позволяет гибкость (например, частота пересмотра правил
        файрвола).
\end{itemize}

Большинство новых требований версии~4.0 стали обязательными
с~31~марта 2025~года (ранее допускались как~\enquote{best
practice})~\todocheck{точная дата перехода best practice → mandatory
для~PCI DSS 4.0 new requirements~--- 31 марта 2025}.


\section{Customized Approach}
\label{sec:pci-customized}

PCI~DSS~4.0 ввёл альтернативный путь выполнения требований~---
Customized Approach. Вместо того чтобы реализовать контроль
в~точности так, как~описан в~стандарте (Defined Approach),
организация может достичь \emph{цели безопасности} требования
иным контролем, если~может доказать его~эффективность.

Для~использования Customized Approach необходимо:

\begin{itemize}
  \item Определить цель безопасности (Customized Approach Objective),
        указанную в~стандарте для~каждого требования.
  \item Провести Targeted Risk Analysis, демонстрирующую, что
        альтернативный контроль достигает той~же цели.
  \item Предоставить документацию и~свидетельства QSA (Qualified
        Security Assessor) для~валидации.
\end{itemize}

Customized Approach~--- не~для~всех: он~требует зрелой программы
информационной безопасности, опытного QSA и~готовности инвестировать
в~документацию. Для~организаций со~стандартной архитектурой Defined
Approach проще и~дешевле. Customized Approach предназначен
для~случаев, когда архитектура организации не~укладывается
в~стандартные контроли, но~безопасность обеспечивается
иными средствами.


\section{PCI DSS и российские требования: \law{382-П}, \law{683-П}}
\label{sec:pci-russian}

Организация, обрабатывающая карточные платежи в~России, подчиняется
двум параллельным режимам безопасности: PCI~DSS (требование
карточных сетей) и~положениям Банка России \law{382-П}
и~\law{683-П} (законодательное требование), которые мы~разбирали
в~главе~\ref{ch:regulation}.

Области пересечения значительны:

\begin{itemize}
  \item \textbf{Шифрование данных}~--- и~PCI~DSS (Req~3, 4),
        и~\law{382-П} требуют криптографической защиты; различие
        в~том, что~\law{382-П} требует использования
        сертифицированных СКЗИ (средств криптографической защиты
        информации) на~базе ГОСТ-алгоритмов
        (глава~\ref{ch:crypto}), тогда как~PCI~DSS не~предписывает
        конкретный алгоритм.
  \item \textbf{Контроль доступа}~--- оба режима требуют разграничения
        доступа, идентификации пользователей, MFA.
  \item \textbf{Мониторинг и~журналирование}~--- Req~10 PCI~DSS
        и~требования \law{382-П} к~регистрации инцидентов
        и~событий безопасности.
  \item \textbf{Управление уязвимостями}~--- Req~5--6 PCI~DSS
        и~требования \law{683-П} к~анализу уязвимостей
        и~сертификации ПО.
\end{itemize}

Области расхождения:

\begin{itemize}
  \item \textbf{Криптография.} PCI~DSS допускает AES, RSA, любой
        \enquote{strong cryptography}. \law{382-П} и~\law{683-П}
        требуют ГОСТ для~определённых операций~--- применение
        СКЗИ, сертифицированных ФСБ~\sidecite{cbr-382p}.
  \item \textbf{Отчётность.} PCI~DSS предполагает отчёт перед
        карточной сетью (через ROC или~SAQ). \law{382-П}~--- перед
        Банком России (оценка соответствия, отчётность
        по~инцидентам через ФинЦЕРТ)~\sidecite{cbr-683p}.
  \item \textbf{Сертификация ПО.} \law{683-П} требует использования
        ПО, прошедшего оценку соответствия по~ОУД4 (Оценочный
        уровень доверия~4)~--- требование, не~имеющее прямого
        аналога в~PCI~DSS.
\end{itemize}

\begin{practice}
На~практике единственный разумный подход~--- построить единую
матрицу контролей, маппирующую требования PCI~DSS на~\law{382-П}
и~\law{683-П}. Многие контроли удовлетворяют оба режима одновременно
(шифрование, контроль доступа, журналирование); для~расходящихся
требований (ГОСТ vs AES, ОУД4 vs secure SDLC, ФинЦЕРТ vs ASV)
создаются дополнительные контроли. Попытка вести два отдельных
комплаенс-процесса~--- путь к~дублированию, ошибкам и~завышенным
затратам.
\end{practice}


\section*{Итоги главы}

\begin{itemize}
  \item PCI~DSS~--- 12~требований, организованных в~6~групп
        (сеть, данные, уязвимости, доступ, мониторинг, политика);
        логика~--- концентрическая оборона, а~не~чеклист.
  \item CDE~--- всё, что~хранит, обрабатывает или~передаёт CHD/SAD,
        плюс connected-to и~security-impacting системы. Сегментация
        сети~--- ключевой инструмент ограничения scope.
  \item Scope reduction через токенизацию, P2PE, iFrame и~redirect
        позволяет перейти от~SAQ~D ($\sim$330~контролей)
        к~SAQ~A ($\sim$30)~--- на~порядок меньше усилий.
  \item PCI~DSS~4.0.1 ужесточает требования: инвентаризация
        скриптов на~платёжной странице (Req~6.4.3),
        аутентифицированные внутренние сканы (Req~11.3.1.1),
        MFA для~всех обращений к~CDE, шифрование PAN
        по~доверенным сетям.
  \item Customized Approach позволяет достичь цели безопасности
        альтернативным контролем, но~требует зрелой программы ИБ
        и~валидации QSA.
  \item В~России PCI~DSS и~\law{382-П}/\law{683-П} действуют
        параллельно; области пересекаются (шифрование, доступ,
        мониторинг), но~расходятся в~криптографии (ГОСТ vs AES),
        отчётности (ФинЦЕРТ vs ASV) и~сертификации ПО (ОУД4).
        Единая матрица контролей~--- оптимальный подход.
\end{itemize}
