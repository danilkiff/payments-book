\chapter{PCI~DSS: минимизация того, что нужно защищать}
\label{ch:pci-dss}

\begin{kaobox}[title=Что вы узнаете из этой главы]
  PCI~DSS~--- не~набор из~трёхсот случайных контролей, а~логическая
  рамка, построенная вокруг одной идеи: свести к~минимуму места,
  где~хранятся карточные данные, и~защитить оставшиеся. Здесь мы~разберём
  логику двенадцати требований, научимся определять границы CDE,
  увидим, как~токенизация, P2PE и~iFrame сокращают scope с~SAQ~D
  (330~контролей) до~SAQ~A (30), разберём болезненные новшества
  PCI~DSS~4.0.1 и~проведём мост к~российским требованиям \law{382-П}
  и~\law{683-П}.
\end{kaobox}

Если убрать из~PCI весь язык про аудит и~контроли, останется
один архитектурный вопрос: где именно в~вашей системе появляются
данные карты и~сколько компонентов действительно должны их~видеть.
Всё остальное~--- следствие ответа на~этот вопрос. Поэтому дальше
будем читать PCI~DSS как задачу проектирования границ, а~не~как
бесконечный список требований.


\section{Логика двенадцати требований}
\label{sec:pci-logic}

PCI~DSS (Payment Card Industry Data Security Standard) определяет
требования к~защите данных платёжных карт для~любой организации,
которая хранит, обрабатывает или~передаёт cardholder data. Стандарт
разрабатывается и~поддерживается PCI SSC (Security Standards Council),
основанным пятью глобальными карточными
сетями~\sidecite{pci-dss-4}.

До~появления единого стандарта каждая карточная сеть поддерживала
собственную программу безопасности. \scheme{Visa} запустила CISP
(Cardholder Information Security Program) в~2001~году,
\scheme{Mastercard}~--- SDP (Site Data Protection), American~Express~---
DSOP (Data Security Operating Policy). Мерчанты, принимающие карты
нескольких сетей, были обязаны соответствовать нескольким
несовпадающим требованиям одновременно, а~рассогласованность
программ порождала системные пробелы: мерчант мог успешно пройти
проверку по~одной программе и~при~этом нарушать требования другой.

PCI~DSS~1.0 был опубликован в~декабре 2004~года пятью сетями совместно~---
впервые в~истории конкурирующие платёжные схемы договорились
об~общей базовой планке безопасности. В~2006~году те~же сети
учредили PCI~SSC как~независимый орган, взяв на~себя разработку
стандарта, программу сертификации QSA (Qualified Security Assessors)
и~управление сопутствующими стандартами (PA-DSS, P2PE, PTS).
Катализаторами объединения послужили масштабные утечки: в~2007~году
крупнейший ритейлер TJX Companies раскрыл компрометацию более
45~миллионов записей карт, в~2009-м процессор Heartland Payment
Systems~--- свыше 130~миллионов~\todocite{TJX breach 2007 и Heartland
  breach 2009~--- данные из~SEC filings или Verizon DBIR}. Эти
инциденты показали, что~отраслевой стандарт существует, но~реальное
следование ему оставалось делом доброй воли.

Двенадцать требований PCI~DSS не~являются произвольным списком~---
они организованы в~шесть групп, каждая из~которых защищает
определённый слой:

\begin{enumerate}
  \item \textbf{Безопасная сеть и~системы} (Req~1--2).
        Требование~1: установить и~поддерживать сетевые средства
        безопасности (firewalls, segmentation). Требование~2:
        применять безопасные конфигурации ко~всем компонентам
        (не~использовать заводские пароли, отключать неиспользуемые
        сервисы).
  \item \textbf{Защита данных карт} (Req~3--4).
        Требование~3: защитить хранимые данные (шифрование PAN,
        запрет хранения SAD~--- как~мы~разбирали
        в~главе~\ref{ch:card-data}). Требование~4: шифровать
        передачу cardholder data по~открытым сетям (TLS).
  \item \textbf{Управление уязвимостями} (Req~5--6).
        Требование~5: защитить все системы от~вредоносного ПО.
        Требование~6: разрабатывать и~поддерживать безопасные
        системы и~приложения (secure SDLC, управление патчами,
        защита веб-приложений).
  \item \textbf{Контроль доступа} (Req~7--9).
        Требование~7: ограничить доступ к~данным карт по~принципу
        need-to-know. Требование~8: идентифицировать пользователей
        и~аутентифицировать доступ (MFA). Требование~9: ограничить
        физический доступ к~данным карт.
  \item \textbf{Мониторинг и~тестирование} (Req~10--11).
        Требование~10: журналировать и~мониторить все обращения
        к~сетевым ресурсам и~данным карт. Требование~11:
        регулярно тестировать безопасность (сканы уязвимостей,
        penetration testing).
  \item \textbf{Политика информационной безопасности} (Req~12).
        Поддерживать политику, охватывающую все аспекты
        информационной безопасности для~всего персонала.
\end{enumerate}

Ключевой инсайт: требования~--- концентрическая оборона, а~не~чеклист.
Если~данные карт не~попадают в~вашу среду (благодаря токенизации,
P2PE или~hosted payment page), большинство требований к~вам
просто не~применяются~--- и~именно в~этом состоит стратегия
минимизации scope.


Итак, двенадцать требований PCI~DSS работают как концентрическая
оборона вокруг cardholder data, а~лучший способ соответствовать
стандарту~--- не~героически защищать всё подряд, а~сузить среду,
в~которой данные вообще присутствуют. Поэтому следующий логичный
шаг~--- точно определить границы CDE.

\section{CDE: как определить границы}
\label{sec:pci-cde}

\begin{termdef}{CDE}
  CDE (Cardholder Data Environment) --- совокупность людей, процессов
  и~технологий, которые хранят, обрабатывают или~передают cardholder
  data (CHD) или~sensitive authentication data (SAD), а~также
  все~системы и~сегменты сети, напрямую подключённые к~ним
  или~влияющие на~их~безопасность.
\end{termdef}

Определение границ CDE~--- самая сложная и~критически важная задача
при~PCI-комплаенсе. Ошибка в~определении scope означает либо
недозащищённость (данные карт оказываются за~пределами защитного
периметра), либо избыточность (организация защищает системы,
которые никогда не~видят карточных данных, тратя ресурсы впустую).

CDE включает три категории систем:

\begin{itemize}
  \item \textbf{Системы, обрабатывающие CHD/SAD}~--- серверы
        авторизации, базы данных с~PAN, терминалы~--- очевидно
        в~scope.
  \item \textbf{Connected-to systems}~--- системы, напрямую
        подключённые к~CDE по~сети, даже если они~сами не~работают
        с~карточными данными (например, DNS-сервер в~одном сегменте
        с~платёжным сервером).
  \item \textbf{Security-impacting systems}~--- системы, влияющие
        на~безопасность CDE: серверы аутентификации, системы
        мониторинга, антивирусное ПО, NTP-серверы.
\end{itemize}

Сетевая сегментация~--- основной инструмент ограничения CDE:
если~карточные данные обрабатываются в~изолированном сегменте сети,
отделённом от~остальной инфраструктуры файрволом с~правилами
deny-by-default, то~только этот сегмент (и~connected-to системы)
попадают в~scope. Без~сегментации в~scope оказывается \emph{вся}
сеть организации.

\section{Scope reduction: токенизация, P2PE, iFrame}
\label{sec:pci-scope-reduction}

Если CDE определяет, что~в~scope, то~scope reduction~--- стратегия
уменьшения того, что~в~этот scope попадает. Четыре основных
механизма:

\subsection*{Токенизация}

Токенизация как~инструмент scope reduction существует в~двух
принципиально разных вариантах, которые важно не~смешивать.
Сетевые токены (подробно разобранные в~главе~\ref{ch:tokenization})~--- это DPAN, выдаваемый карточной
сетью. Настоящий PAN при~этом подходе никогда не~попадает
в~системы мерчанта: кошелёк или~платёжное приложение передаёт
терминалу DPAN, а~не~FPAN. Поскольку DPAN бесполезен за~пределами
своего домена и~не~раскрывает настоящий номер карты, системы
мерчанта, обрабатывающие исключительно DPAN, могут быть выведены
из~CDE~\sidecite{pci-dss-4}.

PSP-токенизация~--- иная модель: мерчант один раз передаёт PAN
в~платёжный шлюз, шлюз хранит его в~своём vault и~возвращает
мерчанту суррогатное значение (payment token). При~последующих
транзакциях мерчант оперирует только этим суррогатом. Scope
мерчанта при~этом подходе действительно сокращается (его системы
не~хранят PAN постоянно), однако из~scope не~выпадает сам шлюз:
PSP, владеющий vault-ом, остаётся полноценным субъектом PCI~DSS.
Кроме того, первоначальная транзакция, при~которой PAN всё~же
попадает в~систему до~токенизации, должна быть защищена отдельно
(см.~P2PE ниже)~\sidecite{pci-dss-4}.

\subsection*{P2PE (Point-to-Point Encryption)}

P2PE~--- стандарт PCI, при~котором данные карты шифруются
непосредственно в~терминале в~момент считывания и~расшифровываются
только в~HSM на~стороне эквайера или~процессора. Ключевой элемент
решения~--- SRED-модуль (Secure Reading and Exchange of Data)
внутри терминала: это~аппаратно защищённый компонент, который
перехватывает считанные данные ещё до~их~попадания в~общую память
терминала и~немедленно шифрует их~на~ключе, введённом в~ходе
процедуры key injection. Сам мерчант физически не~может прочитать
карточные данные~--- через его~инфраструктуру проходит только
зашифрованный blob~\sidecite{pci-p2pe}.

Модель P2PE разделяет ответственность между двумя ролями: P2PE
Component Provider (поставщик отдельных компонентов~--- терминала,
HSM, системы управления ключами) и~P2PE Solution Provider (поставщик
сертифицированного конечного решения, объединяющего компоненты).
Мерчант, использующий готовое сертифицированное P2PE-решение
провайдера, не~обязан проходить собственный PCI~DSS-аудит CDE:
вся~среда мерчанта (серверы, сеть, рабочие станции) выходит
из~scope, а~терминал оценивается по~листингу PCI~P2PE,
а~не~полному PCI~DSS~\sidecite{pci-p2pe}. Точкой расшифровки
остаётся HSM процессора~--- именно эта среда сохраняет полный
PCI~DSS scope и~несёт ответственность за~ключевую защиту.

\subsection*{Hosted payment fields (iFrame)}

Для~e-commerce: вместо того чтобы собирать данные карты
на~собственной платёжной странице, мерчант встраивает iFrame
или~hosted fields, отрендеренные PSP. Принципиальный момент
в~этой схеме~--- принадлежность DOM-элемента: форма ввода
существует в~контексте домена PSP, нажатия клавиш передаются
непосредственно на~серверы PSP, и~ни~PAN, ни~CVV никогда
не~касаются JavaScript-кода или~серверов мерчанта. Мерчант
получает от~PSP платёжный токен и~работает только с~ним
при~дальнейшей обработке заказа~\sidecite{pci-dss-4}.

Именно против этого подхода направлено требование PCI~DSS~4.0.1
Req~6.4.3: хотя карточные данные и~не~попадают в~DOM мерчанта,
скимминг-атаки на~платёжные страницы внедряют вредоносный JavaScript
в~родительскую страницу мерчанта и~перехватывают ввод ещё
до~его~передачи в~iFrame. Поэтому PCI~DSS требует полной
инвентаризации и~авторизации каждого скрипта на~странице оплаты:
его источник, назначение и~контрольная сумма должны быть
задокументированы, а~Content Security Policy~--- исключить загрузку
незарегистрированных скриптов. Мерчанты, прошедшие по~маршруту
iFrame, нередко обнаруживают, что~Req~6.4.3 является для~них
наиболее трудоёмким требованием~\sidecite{pci-dss-4}.

\subsection*{Redirect (hosted payment page)}

Redirect~--- архитектурно наиболее радикальный способ вывести
мерчанта из~CDE. При~оплате держатель карты полностью покидает
сайт мерчанта: браузер перенаправляется на~URL платёжного шлюза,
карточные данные вводятся и~обрабатываются \emph{целиком на~стороне
  PSP}, и~только после завершения держатель возвращается на~сайт
мерчанта с~уведомлением об~успехе или~отказе. PAN, CVV и~срок
действия карты в~среде мерчанта не~появляются ни~на~одном этапе,
что~открывает доступ к~SAQ~A с~минимальным набором
из~$\sim$30~контролей~\sidecite{pci-saq}.

Очевидный недостаток~--- разрыв пользовательского опыта: покупатель
замечает смену домена в~адресной строке, что~может вызывать
недоверие и~увеличивает вероятность незавершённых заказов.
Современные PSP смягчают этот эффект, предоставляя white-label
страницы оплаты, стилизованные под~сайт мерчанта. Тем~не~менее
redirect остаётся предпочтительным выбором для~малого бизнеса
и~любых сценариев, где~упрощение комплаенса важнее бесшовности
интерфейса~\sidecite{pci-saq}.

Количественный эффект scope reduction значителен: организация,
не~применяющая ни~одного из~этих механизмов, оценивается по~SAQ~D
($\sim$330~контролей); организация с~полным redirect~--- по~SAQ~A
($\sim$30~контролей). Разница~--- порядок: не~просто меньше
документации, а~принципиально иной масштаб усилий.

\section{SAQ A vs SAQ D: разница в 300 контролей}
\label{sec:pci-saq}

SAQ (Self-Assessment Questionnaire)~--- инструмент самооценки,
который PCI~SSC предоставляет мерчантам для~подтверждения
соответствия стандарту без~привлечения QSA-аудитора. Тип SAQ
определяется не~размером компании и~не~отраслью, а~конкретным
способом взаимодействия с~карточными данными: каким образом
карта попадает в~платёжный процесс и~есть~ли у~мерчанта
хоть~какой-либо контакт с~CHD~\sidecite{pci-saq}.

Интернет-магазин, который при~оплате полностью перенаправляет
покупателя на~страницу PSP (redirect), не~хранит, не~обрабатывает
и~не~передаёт CHD. Для~него применим \textbf{SAQ~A}~---
около 30~контролей, сосредоточенных преимущественно на~политиках
безопасности и~управлении доступом к~системам, обслуживающим
сайт. Это~минимальный документальный объём во~всей системе
SAQ.

Если тот~же магазин использует iFrame от~PSP, но~его
веб-сервер обслуживает страницу оплаты (то~есть потенциально
может влиять на~DOM вокруг iFrame через собственный
JavaScript), он~попадает в~\textbf{SAQ~A-EP}~--- около
140~контролей. Дополнительные контроли охватывают безопасность
веб-приложения, управление скриптами (Req~6.4.3) и~защиту
серверной инфраструктуры. Разница с~SAQ~A значительна и~прямо
отражает риск, который несёт сервер, способный внедрить
скрипт на~платёжную страницу.

Мерчанты с~физическими терминалами охватываются отдельными
типами.

\textbf{SAQ~B-IP} применяется к~торговым точкам
с~PTS-сертифицированными терминалами, подключёнными
к~платёжному процессору по~IP. В~отличие от~SAQ~B
(для~автономных dial-up терминалов и~импринтеров),
он~добавляет требования к~сетевой безопасности.

\textbf{SAQ~C-VT} покрывает сценарий колл-центра
или~back-office: сотрудник вводит карточные данные
вручную через виртуальный терминал PSP, то~есть через
веб-интерфейс, который сам не~хранит CHD.

На~другом конце спектра~--- \textbf{SAQ~D}: применяется ко~всем
мерчантам, которые не~подходят ни~под~один из~упрощённых типов,
а~также ко~всем сервис-провайдерам без~исключения. SAQ~D
содержит полный набор из~$\sim$330~контролей и~соответствует
полному объёму требований PCI~DSS. Разница между SAQ~A
и~SAQ~D~--- буквально на~порядок больше работы: больше тестов,
больше документации, ежеквартальные ASV-сканы, ежегодный
penetration test. Для~многих мерчантов вложение в~техническую
архитектуру, позволяющую перейти с~SAQ~D на~SAQ~A, окупается
уже в~первый комплаенс-цикл~\sidecite{pci-saq}.

\section{PCI DSS 4.0.1: новые требования, которые причиняют боль}
\label{sec:pci-401}

Версия~4.0.1 (актуальная на~момент написания) ввела ряд требований,
которые существенно повышают планку для~организаций,
обрабатывающих карточные данные~\sidecite{pci-dss-4}.

\begin{important}
  Два требования, вызвавших наибольшую дискуссию в~индустрии:

  \textbf{Requirement~6.4.3: инвентаризация и~авторизация скриптов
    на~платёжной странице.} Каждый JavaScript, загружаемый на~страницу,
  где~вводятся карточные данные, должен быть инвентаризирован,
  обоснован и~авторизован. Для~e-commerce это~болезненно: типичная
  платёжная страница подгружает десятки скриптов~--- аналитику,
  A/B-тестирование, tag manager, чат-виджеты,~--- и~каждый из~них
  теперь требует документирования и~контроля целостности
  (Content Security Policy, Subresource Integrity). Требование
  направлено против атак типа Magecart, при~которых злоумышленник
  внедряет вредоносный скрипт на~платёжную страницу для~перехвата
  карточных данных.

  \textbf{Requirement~11.3.1.1: аутентифицированные внутренние сканы
    уязвимостей.} Внутренние сканы должны выполняться с~аутентификацией
  (с~учётной записью, позволяющей сканеру видеть конфигурацию
  и~уязвимости, недоступные при~неаутентифицированном сканировании).
  Это~увеличивает объём выявляемых уязвимостей и~усилия по~их~устранению.
\end{important}

Другие значимые изменения 4.0.1:

\begin{itemize}
  \item MFA (Multi-Factor Authentication) обязательна для~всех
        обращений к~CDE, а~не~только для~удалённого доступа.
  \item Шифрование PAN требуется даже при~передаче по~доверенным
        внутренним сетям (ранее допускалось передавать PAN
        в~открытом виде внутри сегментированной сети).
  \item Targeted Risk Analysis~--- организация должна документировать
        обоснование периодичности для~контролей, где~стандарт
        позволяет гибкость (например, частота пересмотра правил
        файрвола).
\end{itemize}

Большинство новых требований версии~4.0 стали обязательными
с~31~марта 2025~года (ранее допускались как~\enquote{best
  practice})~\todocheck{точная дата перехода best practice → mandatory
  для~PCI DSS 4.0 new requirements~--- 31 марта 2025}.


\section{Customized Approach}
\label{sec:pci-customized}

PCI~DSS~4.0 ввёл альтернативный путь выполнения требований~---
Customized Approach. Вместо того чтобы реализовать контроль
в~точности так, как~описан в~стандарте (Defined Approach),
организация может достичь \emph{цели безопасности} требования
иным контролем, если~может доказать его~эффективность.

Для~использования Customized Approach необходимо:

\begin{itemize}
  \item Определить цель безопасности (Customized Approach Objective),
        указанную в~стандарте для~каждого требования.
  \item Провести Targeted Risk Analysis, демонстрирующую, что
        альтернативный контроль достигает той~же цели.
  \item Предоставить документацию и~свидетельства QSA (Qualified
        Security Assessor) для~валидации.
\end{itemize}

Customized Approach~--- не~для~всех: он~требует зрелой программы
информационной безопасности, опытного QSA и~готовности инвестировать
в~документацию. Для~организаций со~стандартной архитектурой Defined
Approach проще и~дешевле. Customized Approach предназначен
для~случаев, когда архитектура организации не~укладывается
в~стандартные контроли, но~безопасность обеспечивается
иными средствами.


\section{PCI DSS и российские требования: \law{382-П}, \law{683-П}}
\label{sec:pci-russian}

Организация, обрабатывающая карточные платежи в~России, подчиняется
двум параллельным режимам безопасности: PCI~DSS (требование
карточных сетей) и~положениям Банка России \law{382-П}
и~\law{683-П} (законодательное требование), которые мы~разбирали
в~главе~\ref{ch:regulation}.

Области пересечения значительны:

\begin{itemize}
  \item \textbf{Шифрование данных}~--- и~PCI~DSS,
        и~\law{382-П} требуют криптографической защиты.
        Но~\law{382-П} дополнительно требует сертифицированные
        СКЗИ на~базе ГОСТ-алгоритмов
        (глава~\ref{ch:crypto}), тогда как~PCI~DSS
        не~предписывает конкретный алгоритм.
  \item \textbf{Контроль доступа}~--- оба режима требуют разграничения
        доступа, идентификации пользователей, MFA.
  \item \textbf{Мониторинг и~журналирование}~--- Req~10 PCI~DSS
        и~требования \law{382-П} к~регистрации инцидентов
        и~событий безопасности.
  \item \textbf{Управление уязвимостями}~--- Req~5--6 PCI~DSS
        и~требования \law{683-П} к~анализу уязвимостей
        и~сертификации ПО.
\end{itemize}

Области расхождения:

\begin{itemize}
  \item \textbf{Криптография.} PCI~DSS допускает AES, RSA, любой
        \enquote{strong cryptography}. \law{382-П} и~\law{683-П}
        требуют ГОСТ для~определённых операций~--- применение
        СКЗИ, сертифицированных ФСБ~\sidecite{cbr-382p}.
  \item \textbf{Отчётность.} PCI~DSS предполагает отчёт перед
        карточной сетью (через ROC или~SAQ). \law{382-П}~--- перед
        Банком России (оценка соответствия, отчётность
        по~инцидентам через ФинЦЕРТ)~\sidecite{cbr-683p}.
  \item \textbf{Сертификация ПО.} \law{683-П} требует использования
        ПО, прошедшего оценку соответствия по~ОУД4 (Оценочный
        уровень доверия~4)~--- требование, не~имеющее прямого
        аналога в~PCI~DSS.
\end{itemize}

Требование \law{683-П} об~оценке соответствия программного
обеспечения по~ОУД4 заслуживает отдельного пояснения. ОУД4
(Оценочный уровень доверия~4) определён ГОСТ~Р~ИСО/МЭК~15408-3
(российская адаптация Common Criteria) и~предполагает
методическое проектирование, тестирование и~независимый
анализ уязвимостей со~стороны аккредитованной лаборатории ФСБ.
На~практике это означает, что~операционная система, СУБД
и~прикладное платёжное ПО на~объектах критической
инфраструктуры должны иметь действующий сертификат
соответствия. Это~требование не~имеет прямого аналога в~PCI~DSS:
стандарт Req~6 описывает внутренние процессы безопасной
разработки (secure SDLC, управление уязвимостями), но~не~требует
независимой сертификации ПО третьей стороной.
Следствие~--- для~многих компонентов стека организация должна
либо выбирать сертифицированные отечественные продукты,
либо инициировать дорогостоящую процедуру сертификации
используемого иностранного ПО~\sidecite{cbr-683p}.

\begin{practice}
  Единственный разумный подход к~параллельному комплаенсу~---
  построить общую матрицу контролей, в~которой каждому требованию
  PCI~DSS поставлены в~соответствие пункты \law{382-П}
  и~\law{683-П}. Работа ведётся в~три слоя: сначала выделяются
  контроли, удовлетворяющие оба режима одновременно (контроль
  доступа, журналирование, сетевая сегментация, управление
  инцидентами)~--- как~правило, это большинство. Затем
  фиксируются расхождения в~технических требованиях:
  криптографические операции внутри периметра реализуются
  на~ГОСТ-алгоритмах с~сертифицированными СКЗИ, тогда как~тот~же
  трафик может дополнительно защищаться TLS для~соответствия
  Req~4. Наконец, документируются требования, не~имеющие
  пересечения: ОУД4-сертификация ПО (только для~\law{683-П}),
  ФинЦЕРТ-нотификации об~инцидентах (только для~\law{382-П}),
  SAQ или~ROC отчётность перед карточной сетью (только PCI~DSS).
  Единый годовой аудит по~объединённой программе~--- с~единой
  доказательной базой и~одним QSA, понимающим российское
  регулирование~--- существенно дешевле двух параллельных
  процессов.
\end{practice}

\section*{Итоги главы}

\begin{itemize}
  \item PCI~DSS~--- 12~требований, организованных в~6~групп
        (сеть, данные, уязвимости, доступ, мониторинг, политика);
        логика~--- концентрическая оборона, а~не~чеклист.
  \item CDE~--- всё, что~хранит, обрабатывает или~передаёт CHD/SAD,
        плюс connected-to и~security-impacting системы. Сегментация
        сети~--- ключевой инструмент ограничения scope.
  \item Scope reduction через токенизацию, P2PE, iFrame и~redirect
        позволяет перейти от~SAQ~D ($\sim$330~контролей)
        к~SAQ~A ($\sim$30)~--- на~порядок меньше усилий.
  \item PCI~DSS~4.0.1 ужесточает требования: инвентаризация
        скриптов на~платёжной странице (Req~6.4.3),
        аутентифицированные внутренние сканы (Req~11.3.1.1),
        MFA для~всех обращений к~CDE, шифрование PAN
        по~доверенным сетям.
  \item Customized Approach позволяет достичь цели безопасности
        альтернативным контролем, но~требует зрелой программы ИБ
        и~валидации QSA.
  \item В~России PCI~DSS и~\law{382-П}/\law{683-П} действуют
        параллельно; области пересекаются (шифрование, доступ,
        мониторинг), но~расходятся в~криптографии (ГОСТ vs AES),
        отчётности (ФинЦЕРТ vs ASV) и~сертификации ПО (ОУД4).
        Единая матрица контролей~--- оптимальный подход.
\end{itemize}
