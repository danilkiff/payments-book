В~операциях без физического предъявления карты (card-not-present,
CNP) продавец одновременно оценивает риск перед

\section{Происхождение 3DS: фрод по CNP}\label{sec:3ds-cnp-fraud}

В~базовой модели CNP продавец располагает лишь карточными реквизитами:
номером карты, сроком действия, CVV2 и, возможно, адресом или
почтовым индексом. Этих данных достаточно, чтобы сформировать
авторизационный запрос, но недостаточно, чтобы доказать,
что покупку инициирует именно законный держатель. Реквизиты можно
выманить фишингом или получить из утечки, и~после этого
злоумышленник оказывается почти в~той же позиции, что и~реальный
клиент.

По мере того как чип EMV сделал клонирование физической карты
менее выгодным, атаки сместились в~дистанционный канал.
Миграция на чип в~точке продажи заняла больше десяти лет,
и~всё это время канал CNP оставался без сопоставимой защиты.
На этот открытый фланг и~сместилось мошенничество.

Для европейского рынка это хорошо видно по~данным ЕЦБ. В~отчёте по
SEPA фрод по~CNP составлял примерно 84\,\% от~общей стоимости карточного
мошенничества и~в~2020, и~в~2021 году, а~сам этот переход ЕЦБ связывает
с~ростом электронной коммерции и~тем, что защита канала
с~физическим предъявлением карты
стала заметно сильнее~\cite{ecb-card-fraud-2021}. Даже
на фоне снижения абсолютной стоимости фрода по~CNP в~2021 году
именно этот канал оставался главным источником карточных потерь.

\highlight{Карточным сетям нужен был механизм, который решал бы для транзакций CNP
ту же задачу, что чип решает в~терминале: давал независимое
подтверждение того, что транзакцию инициировал законный держатель.
Им стал 3-D Secure}. Сначала рынок знал этот протокол по~брендам
\enquote{Verified by Visa} и~\enquote{Mastercard SecureCode},
а~затем по~единой спецификации EMVCo~\cite{emvco-3ds-whitepaper}.

Распространённость 3DS по~рынкам до сих пор неоднородна. В~Европе обязательность
усиленной аутентификации плательщика (Strong Customer Authentication, SCA)
по~второй европейской директиве о~платёжных услугах
(Payment Services Directive~2, PSD2~\cite{psd2}; см.~раздел~\ref{sec:psd2-sca})
привела к~массовому внедрению 3DS~2; в~США обязательного требования нет, и~продавцы
чаще выбирают между конверсией и~защитой самостоятельно.
В~России 3DS используется повсеместно по~инициативе эмитентов
и~по~правилам системы Мир.

\section{Три домена 3DS}\label{sec:3ds-three-domains}

Название протокола, 3-D Secure, буквально означает
\enquote{безопасность в~трёх доменах}. В~спецификации EMVCo они называются
\textit{Issuer Domain}, \textit{Acquirer Domain} и~\textit{Interoperability Domain};
в~книге мы используем краткие имена~--- домен эмитента, домен эквайера
и~домен интеграции соответственно.
В~домене эмитента находится ACS (Access Control Server)~--- сервер
эмитента или его процессора, который выносит решение об~аутентификации
держателя. В~домене эквайера со~стороны продавца различают
\textit{3DS Requestor}~--- продуктовую роль инициатора аутентификации (сам продавец
или его PSP~--- Payment Service Provider, платёжный провайдер) и~\textit{3DS Server}~---
технический компонент, который от~имени
Requestor формирует Authentication Request (AReq) и~общается с~Directory Server;
оба обычно поставляются платёжным провайдером и~встроены в~инфраструктуру
продавца. Между ними расположен домен
интеграции, в~котором Directory Server карточной сети определяет по~данным
карты нужный ACS и~доставляет сообщения между сторонами.

Распределение ролей жёсткое: сервер 3DS поставляет контекст, DS
маршрутизирует, ACS принимает решение. Продавец сам клиента
не подтверждает; сеть в~решение эмитента не вмешивается. Позже менялись интерфейсы,
объём передаваемых данных и~способы подтверждения, но указанное
распределение ролей оставалось неизменным.

Третий домен, домен интеграции, необходим, потому что эквайер и~эмитент
не знают друг друга напрямую. Эквайер не может по~одному
PAN найти нужный ACS: у~сотен эмитентов разные реализации, разные
эндпоинты и~разные версии протокола. Directory Server маршрутизирует запрос: по~BIN карты (Bank Identification Number,
банковский идентификационный диапазон) определяет
нужный ACS и~доставляет сообщение туда. Без этого промежуточного
звена каждый эквайер был бы вынужден хранить адресную книгу всех
эмитентов и~обновлять её при каждом изменении.

\section{3DS v1: редирект и~его проблемы}\label{sec:3ds-v1}

Первая версия протокола работала через браузерный редирект. При оплате
на~сайте продавца браузер перенаправлялся на~страницу ACS эмитента,
где держатель вводил пароль, связанный с~программой
\enquote{Verified by Visa}, \enquote{SecureCode} или её локальным
аналогом. На~стороне продавца за~обмен с~Directory Server отвечал
MPI (Merchant Plug-In)~--- встроенный компонент 3DS v1, формировавший пару
VEReq/VERes (Verify Enrollment Request/Response) для проверки участия
карты в~программе и~пару PAReq/PARes (Payer Authentication Request/Response)
для собственно аутентификации.

Пользовательский сценарий
держался на~статическом пароле, который вспоминали раз в~несколько
месяцев. Этот пароль путали с~PIN и~паролем от~интернет-банка,
вводили на~странице с~незнакомым доменом, и~часть пользователей
бросала покупку до~конца процедуры. Переход на~отдельный экран подтверждения
сам по~себе показывал клиенту, что страница оплаты внезапно вышла
за~пределы магазина, и~значительная часть клиентов на~этом теряла доверие к~покупке~\cite{visa-secure-ui-3ds1-0-2}.
К~концу жизненного цикла 3DS v1 многие эмитенты заменили статический
пароль на~одноразовый код по~SMS, но сам сценарий полноэкранного
редиректа на~чужой домен и~связанные с~ним риски остались прежними.

Именно этот редирект на~незнакомый домен стал благодатной почвой
для фишинга: пользователь был приучен вводить пароль на~странице,
которая визуально не относится ни к~магазину, ни к~привычному
интернет-банку, и~атакующему оставалось воспроизвести похожий
экран ACS на~подконтрольном ему домене.

С~переходом электронной коммерции на~мобильные устройства
эта проблема стала только заметнее. 3DS v1 был спроектирован
для эпохи десктопного браузера. Редирект посреди оформления
заказа ещё можно было терпеть на~большом экране, но в~мобильном
вебе и~тем более в~нативных приложениях он ломал сценарий почти
целиком. Протокол работал в~отдельном окне и~плохо вписывался в~формат
покупки, ставший для рынка основным.

Логика принятия решения тоже была слишком грубой.
ACS в~3DS v1 выбирал одну из~двух крайностей:
пропустить транзакцию без явной аутентификации или
потребовать статический пароль. У~него не было ни богатого
контекста покупки, ни нативного способа перенести подтверждение
в~мобильное банковское приложение. Первая версия решала
экономическую задачу переноса ответственности, но почти не справлялась
с~задачей точной и~удобной аутентификации.

Эмитенты всё равно поддерживали 3DS v1, потому что
протокол переносил ответственность (\textit{liability shift}): если
продавец использовал
3DS и~аутентификация была одобрена, риск мошенничества
смещался на~эмитента, а~продавец получал защиту от~чарджбэков
по~причине фрода. Для продавцов это был компромисс: потеря
конверсии в~обмен на~снижение чарджбэков, а~не
однозначная выгода~\cite{emvco-3ds-whitepaper}. Поддержка 3DS v1
карточными сетями была прекращена в~октябре 2022 года: с~этого
момента аутентификация в~e-commerce перешла исключительно
на~EMV 3DS~v2~\cite{visa-3ds1-sunset-2022}.

\section{3DS v2: аутентификация на основе риска}\label{sec:3ds-v2}

Вторая версия 3DS была спроектирована как полноценная замена
первой. Протокол сохранил идею дополнительной аутентификации,
но отказался от~обязательного редиректа и~статического пароля.
EMVCo развивает это семейство спецификаций как набор версий 2.x~\cite{emvco-3ds-whitepaper}. Главное нововведение~--- аутентификация
на~основе риска (Risk-Based Authentication, RBA). ACS больше не делает
жёсткий выбор \enquote{пароль или ничего}, а~оценивает риск и~выбирает
подходящий сценарий.

\subsection*{Незаметный сценарий (frictionless flow)}

\highlight{Если ACS считает транзакцию низкорисковой, аутентификация
происходит \emph{невидимо} для держателя}. Сервер 3DS отправляет
контекст операции, ACS оценивает его, возвращает положительный
результат, и~транзакция идёт дальше на~авторизацию без лишней
паузы на~экране оплаты. Этот сценарий и~сделал 3DS пригодным
для массовой электронной коммерции~--- дополнительная проверка
осталась в~процессе, но перестала ломать сценарий оплаты на~каждом шаге.

Качество такого решения напрямую зависит от~контекста, который
передаёт сервер 3DS. ACS получает сведения об~устройстве, браузере или приложении,
адресе доставки, истории покупок у~конкретного продавца, характере
самой операции и~поведенческих признаках. По~этому набору он решает,
похожа ли текущая покупка
на~привычное действие держателя или требует дополнительного
подтверждения. Чем богаче и~точнее эти данные, тем чаще
эмитент пропускает операцию незаметно для клиента, и~тем больше выручки сохраняет продавец.

Полный цикл сообщений в~EMV 3DS выглядит так. Ещё до~AReq браузер
держателя может быть незаметно перенаправлен на~\textit{3DS Method URL}
ACS~--- служебный URL, вызываемый до~AReq в~скрытом iframe, в~котором ACS собирает сигнатуру устройства
(device fingerprinting; механика и~сигналы~--- в~\ref{sec:fraud-fingerprint})
и~привязывает её к~идентификатору будущей транзакции.
Затем 3DS Server отправляет в~Directory Server Authentication Request (AReq),
DS маршрутизирует его в~нужный ACS и~возвращает Authentication Response
(ARes). В~незаметном сценарии на~ARes цикл и~заканчивается. Если же
ACS решил запросить подтверждение, дальше идёт пара Challenge Request~/
Challenge Response (CReq/CRes) между ACS и~браузером или 3DS SDK (клиентская
библиотека для нативного мобильного приложения, заменяющая браузер в~потоке
аутентификации), а~по~её
завершении ACS отправляет в~DS итог в~Results Request (RReq); DS
ретранслирует его 3DS Server, который отвечает Results Response (RRes).
Финальный статус аутентификации доставляет 3DS Server именно DS,
а~не ACS напрямую. Сам статус приходит в~поле transStatus: \texttt{Y}~---
аутентификация пройдена, \texttt{N}~--- не пройдена, \texttt{A}~---
attempted authentication (см.\ \autoref{sec:3ds-liability}).

Конкретный набор полей AReq определяет качество этих данных.
Среди десятков полей спецификации EMVCo несколько групп влияют
на долю \textit{frictionless} сильнее остальных~\cite{emvco-3ds-whitepaper}:

\begin{itemize}
  \item \textbf{deviceChannel}, параметры браузера
    (\texttt{browserScreenHeight}, \texttt{browserLanguage} и~др.):
    сигнатура устройства; ACS сопоставляет текущую
    сессию с~ранее виденными.
  \item \textbf{threeDSRequestorAuthenticationInd}: тип операции
    (оплата, рекуррент, добавление карты)~--- задаёт порог риска.
  \item \textbf{acctInfo}: возраст аккаунта у~продавца, дата
    последнего изменения адреса, количество покупок за~6~месяцев~---
    прямые индикаторы доверия.
  \item \textbf{merchantRiskIndicator}: сроки доставки, тип товара
    (физический, цифровой, подписка), предзаказ.
  \item \textbf{shipAddrMatch} и~адресные поля: расхождение
    адреса доставки с~адресом биллинга~--- повод для ACS.
  \item \textbf{threeDSRequestorPriorAuthenticationInfo}: ссылка
    на~предыдущую успешную аутентификацию у~того же продавца;
    снижает порог для повторных покупок.
\end{itemize}

Чем больше полей с~данными о~риске передаёт продавец, тем выше
у~него доля \textit{frictionless}-аутентификаций. Это транзакции,
которые ACS принимает по~риску без вызова дополнительного
подтверждения. Точные доли \textit{frictionless} и~\textit{completion rate}
для сценариев с~подтверждением (\textit{challenge}) зависят от~сегмента ТСП, региона
и~качества данных по~риску; публичные агрегированные числа
по~этим показателям в~открытых источниках встречаются редко
и~обычно происходят из~маркетинговых материалов конкретных
вендоров, поэтому в~книге они не приводятся.\footnote{Внутренние
A/B-измерения у~крупных PSP стабильно показывают, что улучшение
качества переданных в~3DS-сообщении полей даёт измеримый рост
доли \textit{frictionless} и~совокупной конверсии страницы оплаты. Конкретные
цифры остаются предметом внутреннего бенчмарка продавца; общего
правила здесь нет.} Каждая лишняя сессия подтверждения
отсеивает часть покупателей, поэтому правильно настроенная
передача данных в~3DS-сообщении напрямую влияет на~конверсию.

\subsection*{Сценарий дополнительного подтверждения (challenge flow)}

Если ACS видит повышенный риск~--- например, новое устройство или нетипичную сумму,~---
он запускает запрос
дополнительного подтверждения. В~отличие от~первой версии, протокол
поддерживает несколько методов подтверждения
в~цифровом канале. ACS инициирует
Challenge Request (CReq) и~передаёт его напрямую в~браузер или 3DS SDK;
после ответа держателя Challenge Response (CRes) возвращается к~ACS~\cite{emvco-3ds-whitepaper}.

Метод подтверждения подбирается под реальную среду:
одноразовый пароль по~SMS или push-подтверждение
с~биометрией в~мобильном банке. 3DS v2 работает внутри тех интерфейсов, которыми
держатель пользуется каждый день, и~не зависит от~редкого пароля.

Большая часть транзакций в~3DS v2 проходит незаметно для держателя,
а~те операции, для которых подтверждение всё-таки нужно, проходят
в~интерфейсе, ближе к~обычному мобильному банку, чем к~чужой
странице с~паролем. Для ACS наиболее ценны те данные,
которые помогают отличить привычную покупку от~нетипичной:
характер устройства и~история покупок у~продавца~\cite{emvco-3ds-whitepaper}.

\section{Асинхронная аутентификация}\label{sec:3ds-decoupled}

3DS v2 ввёл ещё один сценарий~--- асинхронную аутентификацию
(decoupled authentication), то есть подтверждение, которое происходит
\emph{не в~момент} покупки и~\emph{не в~контексте} страницы оплаты.
Обычный сценарий
дополнительного подтверждения предполагает, что держатель находится
рядом с~устройством и~может немедленно ответить на~запрос. Но
устройство интернета вещей без экрана~--- например, умная колонка
или подключённый автомобиль~--- инициирует покупку без участия
человека. В~рекуррентном или ином 3RI-сценарии (3RI~--- 3DS Requestor Initiated,
аутентификация, инициированная не держателем, а~продавцом или его 3DS Requestor)
запрос тоже приходит без присутствия держателя. Для таких случаев работает
асинхронная аутентификация~\cite{emvco-3ds-decoupled}.

Технически это выглядит иначе, чем обычный challenge. ACS
подтверждает использование развязанной \textit{decoupled}-аутентификации в~сообщениях
3DS и~затем передаёт итог аутентификации серверу 3DS в~Results
Request (RReq). Для продавца это означает промежуточный статус: заказ
нельзя считать окончательно подтверждённым, пока не получен
результат аутентификации~\cite{emvco-3ds-decoupled}.

Дальше сервер 3DS либо периодически запрашивает итоговый статус,
либо получает его по~предусмотренному интеграцией асинхронному
каналу. Параллельно эмитент связывается с~держателем доступным
способом: push-уведомлением или SMS~\cite{emvco-3ds-decoupled}.

Спецификация задаёт максимальное время ожидания
(\textit{decoupled max timeout}), после которого аутентификация считается
неудавшейся. Если она не завершена вовремя, продавец либо отменяет
заказ, либо отправляет транзакцию без результата аутентификации,
принимая повышенный риск~\cite{emvco-3ds-decoupled}.

Пользовательский сценарий асинхронной аутентификации требует
отдельного проектирования. Типичная формулировка~--- \enquote{Мы отправили
вам уведомление в~приложении: подтвердите, пожалуйста,
покупку, и~мы отгрузим заказ}. Это условное подтверждение
заказа вместо немедленного. Оно требует специальной
обработки в~системе управления заказами: отдельный статус,
таймаут, уведомление при подтверждении или отказе. Нужно ещё и~понятное
объяснение для держателя, чтобы он не забыл нажать кнопку
в~банковском приложении~\cite{emvco-3ds-decoupled}.

\section{Перенос ответственности}\label{sec:3ds-liability}

Перенос ответственности (\textit{liability shift}) означает договорное
перераспределение риска мошеннического чарджбэка с~продавца
и~эквайера на~эмитента. Он остаётся главным экономическим стимулом
для внедрения 3DS, но \highlight{включается не автоматически}:
зависит от~результата аутентификации, статуса операции, правил
сети и, в~Европе, от~того, использовалось ли освобождение от~SCA~\cite{visa-core-rules}.
Сценарии, которые необходимо различать:

\begin{enumerate}
  \item \textbf{3DS выполнен, эмитент одобрил.} Незаметный сценарий
    или подтверждение пройдено. Перенос ответственности
    обычно работает, риск уходит к~эмитенту.
    Продавец защищён от~чарджбэка по~коду причины,
    связанному с~мошенничеством.
  \item \textbf{Сработал сценарий attempted authentication.}
    Попытка аутентификации, при~которой ACS эмитента не отвечает
    или эмитент не участвует в~программе; результат фиксируется
    stand-in-сервером сети, отвечающим от~имени эмитента.
    Исторически некоторые программы действительно давали
    продавцу защиту и~в~этом случае, но этот эффект нельзя
    считать универсальным правилом EMV 3DS, он зависит
    от~конкретной сети и~её текущей программы.
  \item \textbf{Продавец не использует 3DS.} Перенос ответственности
    не работает, продавец и~эквайер несут ответственность
    за~мошенничество.
  \item \textbf{Продавец использует освобождение.} См.\ следующий раздел.
    Перенос ответственности здесь \emph{не~работает}~---
    продавец и~эквайер осознанно отказались от~аутентификации и~приняли риск
    на~себя.
\end{enumerate}

Правила различаются по~сетям и~регионам. У~Visa
и~Mastercard есть собственные нюансы для попытки
аутентификации, рекуррентных операций и~отдельных категорий
продавцов.

Правила для attempted authentication меняются чаще остальных.
По~действующему Visa Secure Program Guide эмитент в~программе
на~EMV 3DS возвращает \enquote{Not Authenticated}; ответ
\enquote{Attempts} формирует Visa Attempts Server в~режиме
\textit{stand-in} (когда сеть отвечает от~имени неучаствующего эмитента)
за~неучаствующих эмитентов~\cite{visa-secure-rules-update}.

В~России, где PSD2 не действует, перенос ответственности для
внутрироссийских операций определяется правилами НСПК для карт Мир.
До~марта 2022 года аналогичные правила действовали для Visa и~Mastercard;
после приостановки их операций в~России этот механизм утратил значение.

\section{Освобождения PSD2 и специфика России}\label{sec:3ds-exemptions}

PSD2 и~регуляторные технические стандарты (Regulatory Technical Standards, RTS)
Европейской банковской службы (European Banking Authority, EBA)~\cite{eba-rts-sca}
требуют SCA для электронных платежей, кроме ряда освобождений: low value
(операции на~небольшие суммы), TRA (Transaction Risk Analysis~--- анализ
риска транзакции эквайером или эмитентом), trusted beneficiary
(доверенный получатель в~белом списке держателя), recurring
(рекуррентные списания с~одинаковой суммой) и~secure corporate
(защищённые корпоративные платежи по~выделенным протоколам). Общая модель
и~состав исключений разобраны в~\autoref{sec:psd2-sca}.

Освобождение запрашивает продавец или эквайер, последнее слово
остаётся за~эмитентом. Применение TRA или low value не отменяет его
риск-модели, и~если транзакция выглядит подозрительно, эмитент
вправе потребовать challenge. Использование освобождения
обычно ослабляет или полностью выключает перенос ответственности:
уменьшая трение, продавец принимает риск мошенничества на~себя~\cite{eba-rts-sca}.
3DS v2 нужен и~в~этом случае как канал
для индикаторов риска, запрошенных послаблений и~итогового решения
эмитента~\cite{eba-rts-sca}.

\subsection*{Специфика России}

Россия не входит в~зону действия PSD2, и~национальное законодательство
не содержит прямого аналога PSD2/RTS с~тем же обязательным набором
освобождений и~тем же способом их описания. Решение
о~необходимости аутентификации по~3DS для внутрироссийских транзакций
остаётся предметом политики эмитента и~правил карточной сети.
Влияние PSD2 и~SCA на~модель доступа к~счёту рассмотрено
в~разделе~\ref{sec:psd2-sca}.

\section{3DS и токенизация}\label{sec:3ds-tokenization}

Токенизация (глава~\ref{ch:tokenization}) и~3DS решают разные задачи.
Токен отвечает на~вопрос, какие именно карточные реквизиты
используются в~операции, а~3DS отвечает на~вопрос, действительно ли
эту операцию инициирует законный держатель. Вместе эти два механизма
дают эмитенту более полную картину риска.
В~сценарии \textit{card-on-file} e-commerce с~EMV 3DS
продавец передаёт в~сервер 3DS платёжный токен и~связанные с~ним
атрибуты; Directory Server запрашивает у~TSP (Token Service Provider,
поставщик услуг токенизации, см.~главу~\ref{ch:tokenization})
de-tokenisation with verification (детокенизацию с~верификацией криптограммы
и~доменных ограничений на~стороне TSP) и~передаёт нужные данные в~ACS,
а~ACS использует их, чтобы решить, нужен ли challenge и~можно ли
сформировать authentication value (криптографическое доказательство успешной
аутентификации~--- CAVV у~Visa, AAV у~Mastercard, передаваемое в~авторизации)~\cite{emvco-payment-tokenisation-guide}.
Токенизация здесь не отменяет 3DS и~не дублирует
его работу. Она добавляет в~протокол дополнительный контекст
о~платёжном инструменте и~делает решение ACS точнее.

Для рекуррентных операций связка выглядит иначе. В~EMV 3DS версии
2.2 и~выше последующие аутентификации повторных списаний \textit{recurring} или рассрочки \textit{instalment} могут
идти как 3RI-запросы. 3DS Requestor указывает 3RI Indicator
и~передаёт DS Transaction ID и/или ACS Transaction ID (уникальные идентификаторы
шага DS и~ACS, выработанные при исходной клиентской аутентификации) из~исходной
аутентификации, чтобы ACS мог соотнести новую операцию
с~первоначальной клиентской установкой~\cite{emvco-3ds-recurring}.
Со~стороны токенизации EMVCo отдельно отмечает, что в~сценариях MIT
(Merchant-Initiated Transaction~--- операция, инициированная продавцом
по~ранее установленному мандату держателя, см.~\autoref{sec:psd2-sca})
свойства токена, способ его проверки и~доменные ограничения
наследуют контекст исходной клиентской операции~\cite{emvco-payment-tokenisation-guide}. Токенизация отвечает за~идентичность и~ограничения платёжного
инструмента, а~3DS отвечает за~контекст аутентификации конкретной операции
или связь с~исходной клиентской аутентификацией. Подробно о~влиянии
токенизации и~3DS на~антифрод в~главе~\ref{ch:fraud}.
