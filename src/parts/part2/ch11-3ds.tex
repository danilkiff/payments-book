\chapter{3-D Secure: кто отвечает за фрод}
\label{ch:3ds}

\begin{kaobox}[title=Что вы узнаете из этой главы]
  EMV-чип (глава~\ref{ch:emv}) закрыл counterfeit fraud, но~фрод
  мигрировал в~канал card-not-present. 3-D~Secure~--- ответ индустрии
  на~CNP-фрод: протокол, определяющий, кто~аутентифицирует держателя
  и~кто~несёт ответственность за~мошенничество. Мы~разберём три домена,
  эволюцию от~v1 к~v2 (и~почему первая версия убивала конверсию),
  frictionless и~challenge flow, liability shift, exemptions PSD2
  и~связку токенизации с~3DS.
\end{kaobox}

Представим обычный checkout в~интернет-магазине: покупатель вводит
PAN, срок действия и~CVV, нажимает \enquote{оплатить}, а~мерчант
должен понять, действительно~ли перед ним законный держатель.
Вся глава отвечает на~один вопрос: как индустрия добавила
в~CNP-мир недостающий слой аутентификации, не~сломав при~этом
конверсию окончательно.


\section{CNP-фрод как причина появления 3DS}
\label{sec:3ds-cnp-fraud}

В~главе~\ref{ch:card-data} мы~видели, что CVV2~--- трёхзначный код
на~обратной стороне карты~--- единственный фактор, отличающий
законного держателя от~того, кто~просто знает номер карты и~срок
действия. CVV2 можно подсмотреть, сфотографировать, выманить
фишингом~--- и~после этого для~совершения покупки в~интернете
ничего больше не~нужно. По~мере того как~EMV-чип сделал подделку
физических карт экономически невыгодной, мошенничество
перетекло в~онлайн-канал: к~середине 2010-х~годов CNP-фрод
составлял более 70\% всех карточных потерь в~регионах
с~завершённой EMV-миграцией~\todocheck{доля CNP-фрода
  в~общем объёме карточного мошенничества~--- Nilson Report
  или~ECB Payment Statistics}.

Карточным сетям нужен был механизм, который решал~бы для~CNP-транзакций
ту~же задачу, что~чип решает для~card-present: подтвердить,
что~транзакцию инициировал законный держатель. Этим механизмом
стал 3-D~Secure~--- протокол, впервые выпущенный \scheme{Visa}
под~брендом \enquote{Verified by Visa} в~2001~году,
а~затем принятый \scheme{Mastercard} (\enquote{Mastercard SecureCode}),
JCB (\enquote{J/Secure}) и~другими
сетями~\sidecite{emvco-3ds}.


Итак, 3-D~Secure появился не~как техническая прихоть схем,
а~как ответ на~структурную проблему CNP-платежей: карточных
реквизитов недостаточно, чтобы убедительно связать транзакцию
с~держателем. Следующий шаг~--- понять, кто именно в~этой
модели собирает данные, кто принимает решение и~кто обеспечивает
совместимость между ними.

\section{Три домена, три участника}
\label{sec:3ds-three-domains}

Название протокола~--- 3-D~Secure~--- буквально означает
\enquote{безопасность в~трёх доменах}. Каждый домен объединяет
участников, выполняющих определённую роль:

\begin{enumerate}
  \item \textbf{Issuer Domain} (домен эмитента)~--- отвечает
        за~аутентификацию держателя карты. Ключевой компонент~---
        ACS (Access Control Server), сервер, принадлежащий эмитенту
        или~его~процессору, который принимает решение: пропустить
        транзакцию без~вмешательства (frictionless) или~запросить
        подтверждение у~держателя (challenge).
  \item \textbf{Acquirer Domain} (домен эквайера)~--- отвечает
        за~инициирование аутентификации. Ключевой компонент~---
        3DS Server, интегрированный в~инфраструктуру мерчанта
        или~его~PSP, который собирает данные о~транзакции и~отправляет
        запрос аутентификации.
  \item \textbf{Interoperability Domain} (домен совместимости)~---
        обеспечивает маршрутизацию между первыми двумя. Ключевой
        компонент~--- DS (Directory Server), управляемый карточной
        сетью, который определяет ACS эмитента по~BIN карты
        и~маршрутизирует сообщения.
\end{enumerate}

\begin{termdef}{ACS}
  ACS (Access Control Server) --- сервер в домене эмитента, принимающий
  решение об~аутентификации держателя. ACS оценивает риск транзакции,
  выбирает между frictionless и challenge flow и~формирует результат
  аутентификации (authentication value), который передаётся
  в~авторизационном сообщении.
\end{termdef}

\begin{termdef}{DS}
  DS (Directory Server) --- сервер в домене совместимости, управляемый
  карточной сетью. DS принимает Authentication Request (AReq) от 3DS Server
  мерчанта, по~BIN карты определяет ACS эмитента и~маршрутизирует
  сообщение к~нему, а~затем возвращает Authentication Response (ARes)
  обратно. DS не~принимает решение об~аутентификации~--- это целиком
  задача ACS; роль DS ограничена маршрутизацией и~обеспечением
  взаимодействия между доменами.
\end{termdef}

\section{3DS v1 --- редирект и его проблемы}
\label{sec:3ds-v1}

Первая версия протокола (3DS~1.0, 2001--2016) работала через
браузерный редирект: при~оплате на~сайте мерчанта браузер
перенаправлялся на~страницу ACS эмитента (часто в~pop-up окне
или~iframe), где~держатель вводил статический пароль~---
тот~самый \enquote{Verified by Visa} или~\enquote{SecureCode},
который эмитент просил задать при~активации сервиса.

Проблемы 3DS~v1 были многочисленны и~системны, и~их~можно
объединить в~три группы: UX-ошибки, технические ограничения
и~архитектурная негибкость.

С~точки зрения пользовательского опыта статический пароль~---
главная болевая точка. Держатели карт регулярно забывали пароль,
которым пользовались раз в~несколько месяцев; путали его с~PIN
или~интернет-банком; не~могли отличить настоящую страницу ACS
от~фишинговой подделки, поскольку обе выглядели одинаково~---
pop-up с~незнакомым доменом. Восстановление пароля требовало
звонка в~банк или~прохождения отдельного flow, и~значительная
часть держателей на~каждом из~этих шагов просто бросала покупку.

Технические проблемы усугублялись по~мере перехода пользователей
на~мобильные устройства. Протокол разрабатывался в~эпоху десктопных
браузеров: редирект на~стороннюю страницу в~середине checkout нарушал
пользовательский сценарий; pop-up блокировался встроенными блокировщиками
браузеров; iframe ломался в~мобильных браузерах и~был
несовместим с~мобильными приложениями. Мерчанты фиксировали падение
конверсии на~10--15\% после включения
3DS~\todocheck{данные о~падении конверсии при~3DS v1~---
  Stripe, Adyen или~Riskified публичные исследования}.

Архитектурная проблема состояла в~бинарности решения: ACS
в~3DS~v1 мог лишь пропустить транзакцию без~пароля~---
фактически игнорируя аутентификацию~--- или~запросить
статический пароль. Возможности оценить риск транзакции
и~принять взвешенное решение не~было: не~существовало ни~понятия
\enquote{уровень риска}, ни~механизма получить 150+~сигналов
о~контексте покупки, ни~способа инициировать challenge через
мобильное банковское приложение.

Несмотря на~это, эмитенты активно поддерживали 3DS~v1, потому что
протокол обеспечивал им~liability shift: если~мерчант использовал
3DS и~аутентификация была одобрена, ответственность за~мошенничество
переходила на~эмитента, а~мерчант получал защиту от~чарджбэков
по~причине фрода. Для~мерчантов этот обмен~--- потеря конверсии
в~обмен на~снижение чарджбэков~--- оказывался выгодным далеко
не~всегда: высокорисковые категории (электроника, luxury) выигрывали
от~защиты, тогда как~низкорисковые (цифровые товары, небольшие суммы)
несли чистые потери выручки~\sidecite{emvco-3ds}.


\section{3DS v2 --- RBA и frictionless flow}
\label{sec:3ds-v2}

EMVCo взял управление стандартом и~в~2017~году выпустил 3DS~2.0~---
полный редизайн протокола, решающий ключевые проблемы первой
версии. Текущая актуальная версия~--- 2.3~\sidecite{emvco-3ds}.
Главное нововведение~--- Risk-Based Authentication (RBA):
ACS больше не~принимает бинарное решение \enquote{пароль
  или~нет}, а~оценивает риск транзакции на~основании богатого
набора данных и~выбирает один из~двух сценариев.

\subsection*{Frictionless flow}

Если ACS считает транзакцию низкорисковой, аутентификация
происходит \emph{невидимо} для~держателя: 3DS Server отправляет
данные в~ACS, ACS анализирует их, возвращает положительный результат,
и~транзакция уходит на~авторизацию~--- держатель ничего не~замечает.
Конверсия не~страдает, а~liability shift всё~равно работает.

Решение ACS основано на~150+ элементах данных, которые 3DS Server
собирает и~передаёт: device fingerprint (тип устройства, ОС, браузер,
разрешение экрана), shipping address, email, номер телефона, история
покупок у~данного мерчанта, индикатор card-on-file и~другие.
Чем~больше данных получает ACS, тем~точнее оценка риска и~тем~чаще
транзакции проходят frictionless.

\subsection*{Challenge flow}

Если ACS оценивает транзакцию как~рискованную~--- новое устройство,
необычная сумма, нетипичная страна, новый мерчант~--- он~инициирует
challenge: запрос дополнительной аутентификации у~держателя.
В~отличие от~v1, challenge в~v2 не~является статическим паролем;
эмитент выбирает из~нескольких методов в~зависимости от~доступной
инфраструктуры. ACS инициирует Challenge Request (CReq) и~передаёт
его через DS на~3DS SDK или~iframe браузера; SDK рендерит нативный
UI аутентификации, держатель отвечает, и~Challenge Response (CRes)
уходит обратно к~ACS~\sidecite{emvco-3ds}.

Технически доступные методы challenge существенно отличаются от~v1.
OTP через SMS или~push~--- наиболее распространённый метод: ACS
инициирует отправку кода, держатель вводит его в~нативном UI
3DS SDK. Биометрический challenge через мобильное банковское
приложение~--- элегантный вариант: push-уведомление в~банковском
приложении предлагает подтвердить транзакцию Face ID или~отпечатком
пальца. Результат биометрии возвращается в~ACS как~подтверждение
без~передачи биометрических данных через сеть. Out-of-band
аутентификация позволяет подтвердить покупку на~сопряжённом
устройстве~--- например, подтвердить платёж на~ноутбуке через
смартфон. Всё это~выглядит значительно лучше pop-up v1: нативный
UI, привычный держателю интерфейс мобильного банка, отсутствие
переходов на~сторонние домены~\sidecite{emvco-3ds}.

Ключевым фактором конверсии при~challenge служит то, что~80+\%
транзакций в~зрелых реализациях v2 проходят frictionless~--- то~есть
большинство держателей challenge вовсе не~замечают. Когда challenge
всё~же инициируется, его~конверсия выше, чем~у~v1: привычный
интерфейс мобильного банка снижает отказы, а~OTP-код не~требует
восстановления забытого пароля. Для~понимания того, какие из~150+
элементов данных наиболее значимы при~RBA-оценке, достаточно
представить себя на~месте ACS: device fingerprint и~история
покупок у~данного мерчанта~--- сигналы с~наибольшим весом,
тогда как~сумма и~категория MCC~--- вторичные факторы.
Поведенческие данные (typing rhythm, scrolling pattern на~checkout-странице)
отдельные ACS учитывают как~дополнительный~слой~\sidecite{emvco-3ds}.

\section{Decoupled authentication}
\label{sec:3ds-decoupled}

3DS~v2 ввёл ещё один сценарий: decoupled authentication~---
аутентификация, которая происходит \emph{не~в~момент} покупки
и~\emph{не~в~контексте} checkout-страницы. Зачем экосистема
получила этот механизм? Стандартный challenge предполагает,
что~держатель находится рядом с~устройством и~может немедленно
ответить на~запрос. Однако существует целый класс ситуаций,
где~это~условие не~выполняется: IoT-устройство без~экрана
(умная колонка, подключённый автомобиль) инициирует покупку
без~участия человека; рекуррентная транзакция запускается
ночью по~расписанию; SDK эмитента несовместим с~платформой
мерчанта и~challenge технически невозможен в~текущей сессии.
Для~всех этих случаев decoupled authentication предоставляет
выход~\sidecite{emvco-3ds}.

Технически ACS возвращает статус \enquote{D} (decoupled) в~ARes.
Мерчант получает ответ, завершает оформление заказа, а~покупатель
видит сообщение вроде \enquote{заказ принят, ожидаем подтверждения}.

Дальше 3DS Server периодически опрашивает DS. Один вариант~---
пара Results Request/Results Response. Другой~--- callback,
когда эмитент завершил асинхронную аутентификацию. Параллельно эмитент
связывается с~держателем доступным каналом: push, SMS,
звонок оператора.

Спецификация задаёт максимальное время ожидания
(decoupled max timeout), после которого аутентификация считается
неудавшейся. Если она не~завершена вовремя, мерчант либо отменяет
заказ, либо отправляет транзакцию без~результата аутентификации,
принимая повышенный риск~\sidecite{emvco-3ds}.

Для~checkout UX decoupled authentication требует отдельного
проектирования. Стандартный сценарий: \enquote{Мы~отправили
  вам~уведомление в~приложении~--- подтвердите, пожалуйста,
  покупку, и~мы~отгрузим заказ}. Это~условное подтверждение
заказа вместо немедленного~--- паттерн, требующий специальной
обработки в~системе управления заказами (отдельный статус,
таймаут, уведомление при~подтверждении или~отказе) и~понятного
объяснения держателю, чтобы он~не~забыл нажать кнопку
в~банковском приложении~\sidecite{emvco-3ds}.


\section{Liability shift: когда работает, когда нет}
\label{sec:3ds-liability}

Liability shift~--- перенос ответственности за~мошенничество
с~мерчанта/эквайера на~эмитента~--- главный экономический стимул
для~использования 3DS. Однако liability shift не~безусловен:
он~работает только при~соблюдении определённых
условий~\sidecite{visa-core-rules}.

\begin{important}
  Четыре сценария, которые необходимо различать:

  \begin{enumerate}
    \item \textbf{3DS выполнен, эмитент одобрил (frictionless или~challenge
            пройден).} Liability shift работает: ответственность на~эмитенте.
          Мерчант защищён от~чарджбэка по~reason code \enquote{fraud}.
    \item \textbf{3DS attempted, ACS недоступен.} Мерчант попытался
          аутентифицировать, но~ACS эмитента не~ответил. В~большинстве
          сетей liability shift всё~равно переносится на~эмитента
          (\enquote{attempted authentication})~--- эмитент не~может
          перекладывать риск на~мерчанта, если~его~собственная
          инфраструктура не~работает.
    \item \textbf{Мерчант не~использует 3DS.} Liability shift
          не~работает: мерчант/эквайер несёт ответственность
          за~мошенничество.
    \item \textbf{Мерчант использует exemption} (освобождение от~SCA,
          см.\ следующий раздел). Liability shift, как~правило,
          \emph{не~переносится} на~эмитента: мерчант/эквайер
          осознанно отказался от~аутентификации и~принял риск
          на~себя.
  \end{enumerate}
\end{important}

Правила различаются по~сетям и~регионам. У~\scheme{Visa}
и~\scheme{Mastercard} есть свои нюансы в~liability shift
для~attempted authentication, recurring-транзакций и~разных
категорий мерчантов.

В~России, где~PSD2 не~действует, liability shift определяется
правилами НСПК для~карт Мир и~правилами международных сетей
для~\scheme{Visa}/\scheme{Mastercard}.


\section{Exemptions PSD2 и российская специфика}
\label{sec:3ds-exemptions}

В~Европейском союзе директива PSD2 и~Regulatory Technical Standards
(RTS) EBA требуют Strong Customer Authentication (SCA)~--- двухфакторную
аутентификацию~--- для~всех электронных платежей, за~исключением
ряда освобождений (exemptions)~\sidecite{psd2}. SCA требует
как~минимум два из~трёх факторов: знание (пароль, PIN), владение
(телефон, карта) и~биометрию (отпечаток, лицо)~\sidecite{eba-rts-sca}.

Логику exemptions удобно разбить на~три группы: низкорисковые,
доверительные и~корпоративные~\sidecite{eba-rts-sca}.

К~низкорисковым относятся low value и~TRA. Low value~--- наиболее
массовый exemption: транзакции ниже 30~EUR освобождаются от~SCA
автоматически, однако с~кумулятивным ограничением. Эмитент ведёт
счётчик: если~с~последней аутентификации держатель накопил более
5~бесконтактных или~remote-транзакций без~SCA, или~если~их~суммарный
объём превысил 100~EUR, следующая транзакция обязана пройти SCA
даже при~сумме ниже 30~EUR. TRA (Transaction Risk Analysis)~---
более тонкий инструмент: эквайер или~мерчант запрашивает exemption,
ссылаясь на~собственный анализ риска. Допустимый порог суммы
напрямую зависит от~fraud rate эквайера за~последний период:
при~fraud rate ниже 0,13\%~--- до~100~EUR; ниже 0,06\%~---
до~250~EUR; ниже 0,01\%~--- до~500~EUR. Эквайер, не~выдерживающий
целевых показателей фрода, теряет право запрашивать TRA
для~высоких сумм~\sidecite{eba-rts-sca}.

Доверительные exemptions формируют долгосрочные отношения между
держателем, эмитентом и~мерчантом. Trusted beneficiary позволяет
держателю добавить конкретного мерчанта в~\enquote{белый список}
через интерфейс своего банка; все последующие транзакции у~этого
мерчанта освобождаются от~SCA до~момента удаления из~списка.
Recurring exemption действует иначе: первая транзакция серии
рекуррентных списаний с~фиксированной суммой обязана пройти
полное SCA, и~именно это первое согласие держателя~--- основание
для~освобождения всех последующих транзакций в~цепочке с~тем~же
мерчантом и~той~же суммой. Изменение суммы~--- даже незначительное~---
требует нового SCA для~первой транзакции нового
размера~\sidecite{eba-rts-sca}.

Secure corporate~--- нишевый exemption для~B2B-платежей через
защищённые корпоративные инструменты: корпоративные карты
с~централизованным контролем и~выделенные каналы межбанковских
расчётов, где~держатель-физическое~лицо как~концепция неприменимо.

\begin{kaobox}[title=Exemptions PSD2 (справка)]
  \begin{itemize}
    \item \textbf{Low value} ($<$30~EUR, кумулятивно: $<$5~транзакций или $<$100~EUR)
    \item \textbf{TRA}: fraud rate $<$0,13\%~$\to$ 100~EUR; $<$0,06\%~$\to$ 250~EUR; $<$0,01\%~$\to$ 500~EUR
    \item \textbf{Trusted beneficiary}~--- мерчант в~белом списке держателя
    \item \textbf{Recurring}~--- первая транзакция с~SCA, последующие освобождены
    \item \textbf{Secure corporate}~--- корпоративные платежи
  \end{itemize}
\end{kaobox}

\subsection*{Российская специфика}

Россия не~входит в~зону действия PSD2, и~российское законодательство
не~содержит прямого аналога требования SCA. Решение
о~необходимости 3DS-аутентификации для~внутрироссийских транзакций
принимается эмитентом на~основании его~собственной политики
и~правил карточной сети. На~практике российские банки активно
используют 3DS~v2 с~RBA: frictionless для~типичных транзакций,
challenge (SMS, push) для~подозрительных. Правила НСПК для~карт
Мир определяют собственные условия liability shift, аналогичные
по~структуре международным, но~с~учётом российского регуляторного
контекста.


\section{3DS и токенизация: как работают вместе}
\label{sec:3ds-tokenization}

Токенизация (глава~\ref{ch:tokenization}) и~3DS решают разные задачи:
токен отвечает на~вопрос \enquote{что~за~карта?} (идентификация
credentials), 3DS~--- на~вопрос \enquote{это~действительно
  держатель?} (аутентификация). Вместе они~дают эмитенту максимальную
уверенность.

Связка работает в~нескольких сценариях, каждый из~которых по-своему
объединяет идентификацию credentials и~аутентификацию держателя.

При~провизионировании мобильного кошелька TSP может инициировать
step-up аутентификацию через 3DS ещё до~того как~токен выпущен.
Держатель добавляет карту в~Apple Pay или~Google Pay~--- и~вместо
того чтобы TSP оценивал риск лишь по~данным карты и~устройства,
эмитент запрашивает прохождение 3DS challenge. Успешный challenge
служит подтверждением, что~карту добавил именно законный держатель,
а~не~тот, кто~нашёл реквизиты в~утечке. В~результате выпущенный
токен получает высокий Assurance Level~--- и~все последующие
транзакции с~этим кошельком будут одобряться
с~более~высокой~вероятностью~\sidecite{emvco-3ds}.

При~e-commerce транзакции с~card-on-file мерчант включает
в~авторизационное сообщение одновременно DPAN и~3DS authentication
value. Эмитент получает два независимых подтверждения: токен
с~Assurance Level говорит о~том, что~credentials проверены при~провизионировании,
а~CAVV или~AAV~--- что~держатель аутентифицировался непосредственно
перед этой покупкой. Ни~один из~этих слоёв в~отдельности
не~даёт той~же уверенности, что~оба вместе; именно поэтому approval
rate токенизированных 3DS-транзакций максимален среди всех
форматов CNP~\sidecite{emvco-3ds}.

Для~рекуррентных подписок паттерн иной. Первая транзакция
(CIT~--- Customer-Initiated Transaction) проходит полный 3DS
с~challenge или~frictionless; TSP фиксирует Network Transaction ID
этой аутентификации. Все последующие платежи по~подписке~---
MIT (Merchant-Initiated Transactions)~--- отправляются с~сетевым
токеном и~ссылкой на~Network Transaction ID первой транзакции,
тогда как~3DS не~требуется: держатель дал согласие однократно,
и~это согласие привязано к~токену. Если~мерчант обновляет условия
(меняет сумму), новая первая CIT должна пройти 3DS заново~---
токен при~этом остаётся прежним~\sidecite{emvco-3ds}.

В~авторизационном \iso{8583}-сообщении 3DS-данные и~данные
токенизации передаются параллельно: DPAN~--- в~\de{2}, token
metadata~--- в~специфичных для~сети расширениях, CAVV/AAV
(3DS authentication value)~--- в~\de{126} или~аналогичном поле,
признак токенизации и~Assurance Level~--- в~дополнительных полях,
определяемых схемой. Подробно о~влиянии токенизации и~3DS
на~антифрод~--- в~главе~\ref{ch:fraud}; о~правилах
для~приложений~--- в~Приложении~E.


\section*{Итоги главы}

\begin{itemize}
  \item 3-D~Secure~--- протокол аутентификации для~CNP-транзакций,
        возникший как~ответ на~миграцию фрода из~card-present
        в~card-not-present после внедрения EMV.
  \item Протокол работает в~трёх доменах: Issuer (ACS), Acquirer
        (3DS Server), Interoperability (DS карточной сети).
  \item 3DS~v1 (редирект + статический пароль) убивал конверсию
        на~10--15\%; v2 (RBA + frictionless/challenge) сохраняет
        конверсию: 80\%+ транзакций проходят frictionless.
  \item Liability shift переносит ответственность за~фрод
        на~эмитента при~успешном 3DS; при~отсутствии 3DS или~при~exemption
        ответственность остаётся на~мерчанте/эквайере.
  \item PSD2 (ЕС) требует SCA с~exemptions (low value, TRA, trusted
        beneficiary, recurring); Россия не~под~PSD2, RBA определяется
        политикой эмитента и~правилами НСПК.
  \item Токенизация + 3DS = максимальная уверенность для~эмитента:
        токен идентифицирует credentials с~Assurance Level, 3DS
        подтверждает личность держателя.
\end{itemize}
