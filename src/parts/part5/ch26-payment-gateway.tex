\chapter{Платёжный шлюз изнутри}
\label{ch:gateway}

\begin{kaobox}[title=Что вы узнаете из этой главы]
  Из~каких компонентов состоит платёжный шлюз и~в~каком порядке
  они обрабатывают транзакцию; чем smart routing отличается от~cascade
  и~почему путаница между ними приводит к~дублям; как выбор
  интеграционной модели (hosted page, API, SDK) определяет PCI scope
  мерчанта; почему timeout~--- не~decline и~что с~этим делать;
  и~как JSON-запрос Stripe превращается в~сообщение \iso{8583}.
\end{kaobox}

Возьмём один запрос мерчанта: \codeword{POST} к~\endpoint{/v1/payments} на~оплату
заказа. Снаружи это обычный JSON поверх HTTPS, но~внутри шлюза
этот запрос должен пройти цепочку контрактов, где каждая ошибка
меняет либо риск, либо конверсию, либо PCI scope. Именно эту
внутреннюю механику мы~и~будем разбирать.

\section{Анатомия шлюза}
\label{sec:gw-anatomy}

В~предыдущих частях книги мы~двигались вдоль жизненного цикла
транзакции, рассматривая её~с~точки зрения протоколов, платёжных
систем и~операционных процессов. Теперь пора заглянуть внутрь
той~системы, через которую большинство мерчантов подключаются
к~платёжной инфраструктуре,~--- платёжного шлюза (payment gateway).

\begin{termdef}{Платёжный шлюз, процессор, оркестратор}
  Три термина часто используют как взаимозаменяемые, но~они описывают
  разные уровни абстракции. \textbf{Платёжный шлюз} (payment gateway)~---
  система, принимающая платёжные запросы от мерчанта по~высокоуровневому
  API и~транслирующая их в~протоколы карточных сетей и~банков-эквайеров.
  \textbf{Платёжный процессор} (payment processor)~--- система,
  непосредственно обрабатывающая авторизацию, клиринг и~settlement,
  часто имеющая прямое подключение к~карточным сетям.
  \textbf{Платёжный оркестратор} (payment orchestrator)~--- надстройка,
  маршрутизирующая транзакции между несколькими шлюзами или процессорами
  по~заданным правилам. На~практике границы размыты: крупные PSP
  (Payment Service Provider) вроде Stripe или Adyen совмещают все три
  роли в~одном продукте.
\end{termdef}

Независимо от~того, как называется конкретный продукт, внутри
платёжного шлюза можно выделить пять ключевых компонентов, через
которые проходит каждая транзакция: модуль приёма запросов (API
gateway), модуль оценки рисков (risk engine), модуль маршрутизации
(routing engine), хранилище токенов (tokenization vault) и~модуль
авторизации (authorization proxy), передающий запрос банку-эквайеру
или напрямую карточной сети.

Порядок, в~котором эти компоненты вызываются, не~произволен
и~определяет как архитектурные свойства системы, так и~её PCI scope.
Рассмотрим типичный путь запроса.

Мерчант отправляет HTTPS-запрос на~API шлюза~--- например,
\codeword{POST} к~\endpoint{/v1/payments} с~JSON-телом, содержащим сумму, валюту,
данные карты (или токен) и~метаданные заказа. API gateway валидирует
запрос, проверяет аутентификацию (API key или OAuth token),
обеспечивает идемпотентность (об~этом подробнее в~разделе~\ref{sec:gw-reliability})
и~передаёт нормализованное внутреннее представление транзакции
следующему компоненту~--- risk engine.

Risk engine применяет к~транзакции набор правил и~моделей:
velocity checks (частота и~суммы транзакций по карте, IP-адресу,
устройству), проверку геолокации, device fingerprinting,
а~в~более зрелых системах~--- ML-модель, присваивающую транзакции
fraud score. Результат~--- решение: пропустить, отправить на~3-D Secure
challenge, или~отклонить. Подробно устройство антифрод-систем
разобрано в~главе~\ref{ch:fraud}; здесь важно зафиксировать,
что risk engine работает \emph{до}~авторизации: отклонить
подозрительную транзакцию до~отправки в~сеть дешевле, чем разбирать
chargeback после.

Если risk engine пропустил транзакцию, она~попадает в~routing engine,
который определяет, через какого эквайера (или какой процессор)
отправить авторизационный запрос. Логика маршрутизации~---
одна из~самых коммерчески значимых частей шлюза, и~мы~посвятим
ей~отдельный раздел~(\ref{sec:gw-routing}).

На~следующем шаге, если мерчант передал полный PAN, шлюз заменяет
его токеном из~tokenization vault: PAN сохраняется в~зашифрованном
хранилище, а~дальше по~внутреннему pipeline и~в~ответе мерчанту
циркулирует только токен. Это~сужает CDE (cardholder data environment)
до одного компонента и~радикально сокращает PCI scope всей
системы~--- механизм, подробно описанный
в~главе~\ref{ch:pci-dss}. Если~же мерчант с~самого начала прислал
сетевой токен (DPAN), vault может не~участвовать, но~шлюз всё равно
проверяет token domain restriction и~запрашивает у~TSP актуальный
криптограмму, если это требуется
протоколом~(см.~главу~\ref{ch:tokenization}).

Наконец, authorization proxy формирует сообщение \iso{8583}
(или~его~эквивалент в~API-based протоколах новых эквайеров),
отправляет его по~защищённому каналу банку-эквайеру, получает ответ,
транслирует response code в~формат API шлюза и~возвращает результат
мерчанту. Весь путь~--- от~HTTPS-запроса до ответа~--- укладывается
в~одну-две секунды, из~которых основная часть приходится
на~ожидание ответа от эмитента через карточную сеть.

Итак, минимальный путь запроса через шлюз уже задаёт архитектурный
инвариант: каждый следующий компонент получает более узкое и~лучше
определённое представление транзакции, чем предыдущий. Из~этого
вырастают и~требования к~изоляции данных, и~PCI scope, и~ограничения
на~отказные режимы.


\section{Smart routing}
\label{sec:gw-routing}

Маршрутизация~--- это~выбор, через какого эквайера (или процессор)
отправить конкретную транзакцию. Для~мерчанта, подключённого
к~единственному эквайеру, выбора нет; но~как только появляется
второй эквайер, возникает задача оптимизации, и~шлюз превращается
из~простого прокси в~routing engine, балансирующий между несколькими
целевыми функциями.

\subsection*{Три оси оптимизации}

Первая ось~--- \textbf{стоимость}. Разные эквайеры предлагают
разные тарифы, а~interchange fee зависит от~типа карты, MCC мерчанта,
региона эмитента и~десятков других параметров. Routing engine может
выбрать эквайера с~минимальной стоимостью для~конкретной комбинации
параметров~--- например, направить внутрироссийскую транзакцию
по~карте Мир через эквайера с~пониженным тарифом НСПК, а~cross-border
транзакцию~--- через другого, у~которого выгоднее FX markup.

Вторая ось~--- \textbf{approval rate}. Разные эквайеры
демонстрируют разный процент одобрений для~одних и~тех~же BIN-диапазонов;
причины могут быть техническими (качество подключения к~сети,
скорость ответа), коммерческими (reputation score мерчанта у~данного
эквайера) или~регуляторными (локальный эквайер в~стране эмитента
получает больше одобрений, чем cross-border). Routing engine,
накапливая статистику одобрений по~BIN, стране и~сумме, может
направить транзакцию туда, где вероятность approval выше~---
и~разница между удачным и~неудачным маршрутом достигает 3--5
процентных пунктов, что для~крупного мерчанта транслируется
в~миллионы рублей выручки.

Третья ось~--- \textbf{латентность}. Мерчанту, работающему
в~реальном времени (e-commerce checkout, in-app purchase), важно,
чтобы ответ пришёл быстро; если один эквайер стабильно отвечает
за~300~мс, а~другой~--- за~1200~мс, routing engine при~прочих равных
предпочтёт первый, потому что каждая дополнительная секунда
на~checkout снижает конверсию.

На~практике три оси конфликтуют: самый дешёвый эквайер может иметь
худший approval rate, а~самый быстрый~--- самый дорогой. Задача
routing engine~--- найти оптимум в~рамках ограничений, заданных
мерчантом: \enquote{минимизировать стоимость при~approval rate
  не~ниже~X\% и~p95 latency не~более~Y~мс}. Продвинутые системы
решают эту задачу как multi-armed bandit problem, динамически
перераспределяя трафик между эквайерами на~основе
текущей статистики.

\subsection*{Cascade routing}

Отдельная и~часто путаемая с~smart routing стратегия~--- cascade
routing: если первый эквайер вернул soft decline (например,
\rc{05}~--- Do Not Honor), шлюз автоматически повторяет транзакцию
через второго эквайера. Логика проста: мягкий отказ одного эквайера
не~означает, что транзакция невозможна в~принципе,~--- другой
эквайер может получить другой ответ от~того~же эмитента,
особенно если его авторизационный запрос выглядит иначе (другой
acquiring BIN, другой MCC, другое поле Terminal ID).

\begin{important}
  Cascade routing~--- \emph{не}~то~же самое, что~retry.
  Cascade отправляет транзакцию \emph{другому} эквайеру после soft
  decline от~первого; retry повторяет запрос \emph{тому~же} эквайеру,
  обычно спустя некоторое время. Путаница между ними приводит
  к~дублям: если шлюз повторил запрос тому~же эквайеру, а~первый
  ответ просто задержался (а~не~был decline), эмитент увидит
  два авторизационных запроса и~может одобрить оба. Cascade
  по~определению идёт к~другому эквайеру и~не~создаёт дублей
  на~стороне первого, но~создаёт другой риск: если первый эквайер
  вернул decline не~мгновенно, а~с~задержкой, и~шлюз уже~отправил
  cascade~--- теперь у~двух эквайеров есть авторизация на~одну
  и~ту~же карту. Грамотная реализация cascade включает немедленный
  reversal первой авторизации, если~она всё-таки~пришла после cascade.
\end{important}

Cascade эффективен для~soft decline, но~бесполезен и~даже вреден
для~hard decline (\rc{14}~--- Invalid Card Number, \rc{54}~---
Expired Card): повторная попытка заведомо провалится, а~избыточные
авторизационные запросы нарушают правила платёжных систем по~retry~---
\scheme{Visa} и~\scheme{Mastercard} ограничивают количество повторных
попыток и~штрафуют за~превышение~(см.~главу~\ref{ch:decline-management}).
Routing engine должен различать soft и~hard decline по~response code
и~применять cascade только к~первым.


\section{Интеграционные модели}
\label{sec:gw-integration}

Мерчант подключается к~платёжному шлюзу одним из~трёх способов,
и~выбор между ними определяет не~только пользовательский опыт
на~checkout, но~и~PCI scope мерчанта, а~значит~--- объём
требований, аудитов и~контролей, которые он~обязан выполнять.

\begin{termdef}{Hosted payment page}
  Hosted payment page (HPP)~--- веб-страница, размещённая
  на~серверах платёжного шлюза, на~которую мерчант перенаправляет
  покупателя для~ввода данных карты. Мерчант не~видит,
  не~принимает и~не~передаёт карточные данные~--- они~вводятся
  и~обрабатываются целиком на~стороне шлюза.
\end{termdef}

\subsection*{Hosted payment page (redirect)}

В~модели HPP мерчант инициирует платёжную сессию через API шлюза
и~получает URL, на~который перенаправляет покупателя. Покупатель
вводит данные карты на~странице шлюза, шлюз обрабатывает транзакцию
и~возвращает покупателя на~сайт мерчанта с~результатом.
Параллельно шлюз отправляет webhook мерчанту с~финальным статусом.

Главное преимущество HPP~--- минимальный PCI scope: поскольку
карточные данные никогда не~проходят через серверы мерчанта,
он~может заполнить SAQ~A~--- самый короткий опросник самооценки
PCI~DSS, содержащий порядка~30 контролей вместо~300+
в~SAQ~D~(см.~главу~\ref{ch:pci-dss}). Для~небольших мерчантов,
у~которых нет выделенной команды безопасности, это~часто
единственный реалистичный вариант.

Недостаток~--- потеря контроля над UX: redirect прерывает checkout
flow, страница шлюза может выглядеть иначе, чем сайт мерчанта,
а~возврат после оплаты не~всегда проходит гладко (закрытая вкладка,
потерянный callback). Кроме того, redirect увеличивает число шагов
в~воронке и~может снижать конверсию на~1--3\%, что для~крупного
e-commerce~--- ощутимая потеря.

\subsection*{Встроенная форма (iFrame / JavaScript SDK)}

Компромиссная модель: мерчант встраивает в~свою страницу
платёжную форму, предоставляемую шлюзом в~виде iFrame
или~JavaScript-библиотеки (drop-in UI). Визуально форма
выглядит как часть сайта мерчанта, но~технически данные карты
вводятся внутри iframe, контролируемого шлюзом, и~отправляются
напрямую на~серверы шлюза, минуя серверы мерчанта.

Эта модель сохраняет UX-контроль (мерчант может стилизовать
форму), не~прерывает checkout redirect-ом и~при~этом позволяет
претендовать на~SAQ~A или~SAQ~A-EP~--- зависит от~деталей
реализации и~того, проходят~ли карточные данные через
JavaScript мерчанта хотя~бы транзитом.

\begin{practice}
  PCI~DSS~v4.0.1 ввёл требование~6.4.3, обязывающее контролировать
  все JavaScript-скрипты, загружаемые на~платёжной странице.
  Даже если карточные данные вводятся в~iFrame шлюза, вредоносный
  скрипт на~странице мерчанта может перехватить ввод через overlay
  или~keylogger~до~того, как~данные попадут в~iFrame. Поэтому
  SAQ~A при~использовании iFrame теперь требует от~мерчанта
  инвентаризации и~мониторинга клиентских скриптов~---
  контроль, которого раньше не~было.
\end{practice}

\subsection*{Прямая интеграция через API (server-to-server)}

В~этой модели мерчант собирает данные карты на~своей стороне
(через собственную форму) и~отправляет их на~API шлюза
в~серверном запросе. Мерчант получает полный контроль над UX,
может реализовать произвольную логику до~отправки (проверку
адреса, split payments, pre-authorization), но~платит за~это
максимальным PCI scope: карточные данные проходят через его~серверы,
что требует SAQ~D или~полного ROC (Report on Compliance)~---
сотни контролей, ежегодный аудит QSA и~квартальные ASV-сканы.

Этот путь выбирают крупные мерчанты и~платёжные платформы,
которые уже~имеют PCI~DSS Level~1 сертификацию и~для~которых
прямой контроль над данными~--- конкурентное преимущество
(например, возможность реализовать собственный tokenization vault
или~оптимизировать retry-логику на~уровне PAN).

\begin{practice}
  Выбор интеграционной модели~--- не~только технический, но~и~
  бизнес-решение. Начинайте с~iFrame/SDK: это~даёт приемлемый UX
  при~минимальном PCI scope. Переходите на~server-to-server
  только когда ограничения iFrame начнут блокировать конкретный
  бизнес-сценарий~--- например, невозможность реализовать
  card-on-file flow с~прозрачным для~покупателя tokenization
  или~необходимость cascade routing на~уровне PAN.
\end{practice}


\section{Надёжность и наблюдаемость}
\label{sec:gw-reliability}

Платёжный шлюз обрабатывает финансовые транзакции, и~каждая ошибка~---
это~либо потерянные деньги, либо потерянный клиент. Ряд инженерных
практик, тривиальных на~словах, но~нетривиальных в~реализации,
отличает надёжный шлюз от~хрупкого.

\subsection*{Идемпотентность}

Идемпотентность~--- свойство операции, при~котором повторный вызов
с~тем~же набором параметров даёт тот~же результат, что~и~первый,
без~побочных эффектов. В~контексте платежей это~означает:
если мерчант отправил запрос на~авторизацию, не~получил ответа
(timeout, сетевой сбой) и~повторил его с~тем~же idempotency key,
шлюз не~должен создавать вторую транзакцию, а~должен вернуть
результат первой.

Реализация требует, чтобы шлюз хранил маппинг
\codeword{idempotency_key} \textrightarrowglyph\ \codeword{transaction_id} в~хранилище
с~гарантированной консистентностью (обычно~--- реляционная БД
с~уникальным индексом по~ключу). TTL idempotency key~--- как правило,
от~24~часов до~7~дней: слишком короткий TTL не~защитит от~задержанных
retry, слишком длинный засоряет хранилище.

\begin{important}
  Идемпотентность на~уровне шлюза не~заменяет идемпотентность
  на~уровне эквайера. Если шлюз отправил авторизацию эквайеру,
  не~получил ответа и~отправил повторно~--- эквайер может
  обработать оба запроса. Шлюз должен либо использовать
  \de{11}~(STAN) + \de{37}~(RRN) для~дедупликации
  на~стороне эквайера, либо реализовать pattern
  \enquote{authorize-then-check}: при~timeout отправлять не~повтор,
  а~status inquiry (\mti{0120} или~проприетарный запрос),
  и~только если~статус~--- \enquote{не~найдено}~--- повторять
  авторизацию.
\end{important}

\subsection*{Обработка timeout}

Timeout~--- одна из~самых коварных ситуаций в~платёжном
процессинге, потому что отсутствие ответа~--- это~\emph{не}~отказ.
Если шлюз отправил авторизационный запрос эквайеру и~не~получил
ответа в~течение установленного порога (обычно 30--60 секунд),
возможны три~исхода: (а)~запрос не~дошёл и~никогда не~будет
обработан; (б)~запрос дошёл, эмитент одобрил транзакцию, но~ответ
потерялся на~обратном пути; (в)~запрос дошёл, эмитент отклонил
транзакцию, и~ответ тоже потерялся. Шлюз не~может различить
эти три случая в~момент timeout, и~именно это~делает ситуацию
опасной.

Наихудший из~трёх исходов~--- (б): деньги заблокированы на~карте
держателя, эмитент считает транзакцию одобренной, а~мерчант
не~знает об~этом и~может попытаться повторить платёж, создав
двойное списание. Правильная стратегия~--- после timeout
отправить reversal (\mti{0420}) по~тому~же STAN и~RRN:
если оригинальная авторизация прошла, reversal её~отменит;
если~не~прошла~--- reversal будет отклонён как~not found,
что~тоже информативно. Только после reversal шлюз может
безопасно вернуть мерчанту статус \enquote{неуспешно}
или~предложить повторить попытку.

\subsection*{Дедупликация}

Дедупликация~--- защита от~обработки одной и~той~же транзакции
дважды~--- работает на~нескольких уровнях. На~уровне API~---
через idempotency key. На~уровне внутреннего pipeline~--- через
уникальность комбинации STAN + RRN + amount + timestamp,
которая позволяет отловить дубли, проскочившие мимо idempotency
(например, при~cascade routing, когда два эквайера получают
запрос с~разными STAN, но~за~одну и~ту~же покупку).
На~уровне клиринга~--- через reconciliation с~эквайером
и~карточной сетью, описанную
в~главе~\ref{ch:reconciliation}.

\subsection*{Webhooks и гарантия доставки}

Результат транзакции возвращается мерчанту двумя путями:
синхронно~--- в~ответе на~API-запрос, и~асинхронно~---
через webhook (HTTP callback на~URL, указанный мерчантом).
Webhook необходим потому, что~синхронный ответ может
не~дойти (мерчант потерял соединение), а~также потому,
что~статус транзакции может измениться после первого
ответа (например, 3-D Secure challenge завершён,
или~авторизация отклонена на~стадии клиринга).

Гарантия доставки webhook строится по~принципу at-least-once:
шлюз отправляет POST-запрос на~URL мерчанта и~ожидает ответ
с~HTTP-кодом~2xx; если ответ не~получен, шлюз повторяет попытку
с~экспоненциальной задержкой (exponential backoff)~--- обычно
через 5~секунд, 30~секунд, 5~минут, 30~минут, 2~часа, 24~часа.
Каждый webhook подписывается HMAC (SHA-256)~--- мерчант
верифицирует подпись, чтобы убедиться, что~callback пришёл
от~шлюза, а~не~от~злоумышленника.

\begin{practice}
  At-least-once означает, что~мерчант может получить один
  и~тот~же webhook несколько раз. Обработчик webhook на~стороне
  мерчанта должен быть идемпотентным: проверять, не~был~ли
  уже~обработан webhook с~данным \codeword{event_id} (или~
  \codeword{transaction_id} + \codeword{status}), прежде чем
  менять состояние заказа. Игнорирование этого правила~---
  частая причина двойных зачислений в~CRM мерчанта.
\end{practice}

\subsection*{Наблюдаемость}

Платёжный шлюз, обрабатывающий тысячи транзакций в~минуту,
нуждается в~метриках, которые позволят быстро обнаружить
проблему, прежде чем она~затронет значительную долю трафика.
Ключевые метрики можно разделить на~четыре группы.

\textbf{Authorization rate}~--- процент одобренных транзакций
от~общего числа попыток, с~разбивкой по~эквайеру, BIN-диапазону,
стране эмитента, MCC и~сумме. Падение authorization rate
на~2~процентных пункта за~час~--- сигнал, требующий немедленного
расследования: причиной может быть как~сбой у~эквайера, так
и~массовая атака с~использованием скомпрометированных карт.

\textbf{Латентность}~--- p50, p95 и~p99 времени обработки
запроса, измеренного от~получения HTTPS-запроса мерчанта
до~отправки ответа. P99~выше~5~секунд~--- аномалия, типичная
для~проблем с~HSM, медленного ответа эквайера или~перегрузки
собственного routing engine.

\textbf{Error rate}~--- доля транзакций, завершившихся системной
ошибкой (timeout, network error, internal server error),
а~не~business-ответом (approved/declined). Error rate~$> 1\%$~---
инцидент; в~нормальном режиме этот показатель должен находиться
ниже~0,1\%.

\textbf{Webhook delivery rate}~--- процент webhook, доставленных
с~первой попытки. Низкий показатель говорит либо о~проблемах
на~стороне мерчантов, либо о~перегрузке собственной очереди
доставки.

Эти четыре группы метрик, агрегированные в~реальном времени
и~визуализированные на~дашборде, дают операционной команде
достаточный обзор для~того, чтобы обнаружить деградацию
в~течение минут, а~не~часов~--- что~в~платёжном бизнесе может
означать разницу между десятками и~тысячами затронутых
транзакций.


\section{От Stripe API к~\iso{8583}}
\label{sec:gw-api-to-iso}

Для~разработчика транзакция сначала выглядит как~JSON-объект:
\codeword{amount}, \codeword{currency}, \codeword{payment_method},
\codeword{description}. Для~банка-эквайера и~карточной сети
та~же операция уже выглядит как~сообщение
\iso{8583}~(см.~главу~\ref{ch:iso8583}). Одна из~ключевых
функций шлюза~--- перевести одно представление в~другое.

Рассмотрим, как~типичный API-запрос на~создание платежа
превращается в~авторизационное сообщение. Мерчант отправляет
запрос с~суммой~\codeword{1500}, валютой~\codeword{RUB},
номером карты и~сроком действия. Шлюз транслирует это~в~сообщение
\mti{0100}~(Authorization Request), заполняя десятки data
elements:

\begin{itemize}
  \item \codeword{amount}~(1500)~→ \de{4}~(Amount, Transaction)~---
        12-символьное числовое поле, выровненное нулями:
        \codeword{000000150000} (в~минорных единицах~--- копейках);
  \item \codeword{currency}~(\codeword{RUB})~→ \de{49}~(Currency Code,
        Transaction)~--- \codeword{643} (ISO~4217 numeric code рубля);
  \item \codeword{card.number}~→ \de{2}~(Primary Account Number)~---
        PAN, до~19~цифр;
  \item \codeword{card.exp_month}, \codeword{card.exp_year}~→
        \de{14}~(Date, Expiration)~--- \codeword{YYMM};
  \item внутренний ID терминала мерчанта~→ \de{41}~(Card Acceptor
        Terminal Identification)~--- 8-символьный код;
  \item MCC мерчанта~→ \de{18}~(Merchant Type)~--- 4-значный код.
\end{itemize}

Кроме того, шлюз генерирует поля, которых нет в~API-запросе
мерчанта, но~которые требуются протоколом: \de{11}~(STAN)~---
уникальный 6-значный номер транзакции, \de{7}~(Transmission Date
and Time), \de{12}~(Time, Local Transaction), \de{22}~(Point
of~Service Entry Mode)~--- способ ввода данных карты
(e-commerce~--- \codeword{81x}), и~\de{25}~(Point of Service
Condition Code).

Если мерчант использует сетевой токен вместо PAN, шлюз заполняет
\de{2} токенизированным PAN (DPAN), а~криптограмму от~TSP
помещает в~\de{55}~(EMV data) или~в~проприетарное поле,
зависящее от~конкретной карточной
сети~(см.~главу~\ref{ch:tokenization}).

Ответ эмитента проходит обратный путь: \de{39}~(Response Code)~---
двухсимвольный код~--- шлюз транслирует в~JSON-поле
\codeword{status} (\codeword{succeeded}, \codeword{declined},
\codeword{requires_action}) и~дополняет человекочитаемым
описанием. При~этом шлюз неизбежно \emph{теряет} часть
информации: \iso{8583} различает десятки причин отказа (\rc{05}~---
Do Not Honor, \rc{51}~--- Insufficient Funds, \rc{61}~--- Exceeds
Withdrawal Limit), тогда как~API шлюза обычно группирует их
в~несколько категорий. Разработчику, которому важно отличить
\enquote{недостаточно средств} от~\enquote{карта заблокирована},
стоит обращать внимание на~поле \codeword{decline_code}
(или~его~аналог), а~не~только на~\codeword{status}~---
подробнее об~интерпретации response codes см.
в~главе~\ref{ch:decline-management}.

\begin{practice}
  При~отладке платёжной интеграции полезно понимать, во~что
  превращается ваш JSON-запрос на~уровне \iso{8583}. Многие
  шлюзы предоставляют расширенный лог транзакции (raw response),
  в~котором видны значения ключевых data elements~--- DE~39,
  DE~44 (Additional Response Data), DE~55 (EMV data). Если~шлюз
  этого не~показывает, а~эквайер возвращает загадочный decline,
  запросите у~эквайера trace log по~STAN и~RRN: это~единственный
  способ понять, что~именно произошло на~уровне протокола.
\end{practice}

Трансляция работает не~только для~авторизации. Refund
(\codeword{POST} к~\endpoint{/v1/refunds}) превращается в~\mti{0400}
или~\mti{0220}~(в~зависимости от~того, прошёл~ли клиринг);
void~--- в~reversal \mti{0420}; capture (подтверждение
pre-authorization)~--- в~клиринговую запись, которая попадёт
в~batch-файл (TC33 для~\scheme{Visa}, IPM
для~\scheme{Mastercard}), описанный
в~главе~\ref{ch:reconciliation}. Каждая операция на~уровне
API~--- это~один или~несколько сообщений на~уровне протокола,
и~шлюз скрывает эту~сложность за~простым интерфейсом, позволяя
мерчанту работать с~платежами, не~погружаясь в~битовые карты
и~data elements.


\section*{Итоги главы}

\begin{itemize}
  \item Платёжный шлюз~--- это~не~один endpoint, а~конвейер.
        В~его ядре: API gateway, risk engine, routing engine,
        tokenization vault и~authorization proxy.
  \item Порядок вызова этих компонентов влияет и~на~конверсию,
        и~на~PCI scope системы.
  \item Smart routing оптимизирует три конфликтующие оси~---
        стоимость, approval rate и~латентность; cascade routing~---
        повторная отправка другому эквайеру при~soft decline~---
        эффективен, но~требует различения soft и~hard decline
        и~немедленного reversal при~дублях.
  \item Выбор интеграционной модели~--- hosted page, iFrame/SDK,
        server-to-server~--- определяет PCI scope мерчанта:
        от~SAQ~A (~30~контролей) до~SAQ~D / ROC (300+~контролей);
        PCI~DSS~4.0.1 ужесточил требования к~клиентским скриптам
        даже для~iFrame-интеграций.
  \item Timeout~--- не~decline: отсутствие ответа от~эквайера
        не~означает отказ, и~правильная стратегия~--- reversal
        + status check, а~не~повтор; webhook-доставка строится
        по~принципу at-least-once с~HMAC-подписью.
  \item JSON-запрос мерчанта транслируется шлюзом в~сообщение
        \iso{8583}: amount → \de{4}, currency → \de{49},
        PAN → \de{2}, response code \de{39} → \codeword{status};
        при~трансляции неизбежно теряется часть информации,
        и~для~точной диагностики отказов нужен доступ к~raw
        response.
\end{itemize}
