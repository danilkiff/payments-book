В~процессинге лежит авторизационный запрос на покупку на сумму
1\,250 рублей 50 копеек:

\begin{center}
  \small
  \texttt{30 31 30 30}\\[2pt]
  \texttt{72 3C 04 81 20 C0 80 00}\\[2pt]
  \texttt{31 36 32 32 30 30 31 32 31 32 33 34 35 36 30 38 39 32}\\[2pt]
  \texttt{\ldots}
\end{center}

Дамп превращается в~рассказ о~транзакции в~три шага: тип сообщения, карта присутствующих полей, значения по порядку.
Разберу MTI~0100 и~его ответ MTI~0110 -- тот же жизненный цикл из главы~\ref{ch:tx-lifecycle}, только в~байтах.

\section{Одно сообщение в~логах}\label{sec:iso8583-structure}

Грамматика ISO~8583 одинакова для всех сообщений. Три части, строго в~одном порядке:

\begin{enumerate}
  \item \textbf{MTI} (Message Type Indicator): четыре цифры,
    которые отвечают на вопрос, \emph{что это за сообщение}:
    авторизация, финансовое сообщение, сообщение отмены,
    сетевое
    управление.
  \item \textbf{Bitmap}: битовая карта, которая отвечает
    на вопрос, \emph{какие поля присутствуют дальше}.
  \item \textbf{Элементы данных} (data elements): сами значения: номер карты,
    сумма, STAN, код ответа, идентификатор терминала
    и десятки других признаков транзакции.
\end{enumerate}

ISO~8583 устроен иначе, чем самоописывающиеся форматы вроде JSON или XML.
Сначала -- тип операции, потом bitmap с~картой присутствия, потом значения подряд в~порядке возрастания номеров. Тег и~длина внутри каждого поля не передаются.

Канонический пример -- MTI~0100, запрос на авторизацию, и~его пара MTI~0110, ответ эмитента.
Байты в~запросе и~ответе различаются, грамматика общая. Кто читает один MTI~0100, читает и MTI~0110.

\section{MTI: где сообщение находится в~жизненном цикле}\label{sec:iso8583-mti}

MTI -- первые четыре цифры сообщения -- определяет тип операции: запрос на одобрение, финансовая запись, отмена, служебная проверка связи (echo-test).

Учебные обзоры раскладывают MTI по разрядам. На практике надёжнее смотреть в~список разрешённых значений из спецификации интерфейса партнёра.
Стандарт задаёт рамку, конкретный процессинг или сеть фиксирует собственный набор MTI и~правил их использования; даже публичные профили расходятся в~деталях~\cite{iso-8583-2023,fis-iso8583-guide-2023}.

\medskip
Пара MTI~0100/MTI~0110 -- двухстадийная авторизация: эмитент принял решение, деньги ещё не двинулись в~расчётном смысле.
MTI~0120 -- advice: ответ не нужен, доставка гарантирована; типичное применение -- deferred auth и уведомления stand-in processing.
Пара MTI~0200/MTI~0210 -- финансовое сообщение; во многих карточных контурах именно она реализует одностадийный сценарий.
Отмена идёт через MTI~0400/MTI~0410 в~одних интерфейсах и через MTI~0420/MTI~0430 в~других; пара определяется спецификацией интерфейса~\cite{fis-iso8583-guide-2023}.

\subsection*{MTI~0100: запрос на решение}

MTI~0100 -- запрос с~данными, достаточными для решения: кто платит, сколько, где и~каким способом считана карта.
Минимальный набор: PAN или Track~2, код обработки (processing code), сумма, дата и~время передачи, STAN, режим ввода карты, идентификаторы терминала и ТСП. Для чиповой операции добавляется DE~55~\cite{fis-iso8583-guide-2023}.

\subsection*{MTI~0110: ответ эмитента}

MTI~0110 -- решение эмитента. Ответ короче запроса: PAN и Track~2 не нужны, появляются DE~39 (response code) и, при одобрении, DE~38 (authorization ID response).
STAN, сумма, дата и идентификатор терминала повторяются ради сопоставления ответа с~запросом~\cite{fis-iso8583-guide-2023}.

\subsection*{Отмена (reversal): MTI~0400 и MTI~0420}

Одни интерфейсы оформляют отмену как MTI~0400/MTI~0410, другие как MTI~0420/MTI~0430; FIS/Worldpay разводит request и advice~\cite{fis-iso8583-guide-2023}.
Смысл общий: \enquote{считать предыдущее сообщение несостоявшимся} или \enquote{скорректировать его последствия}. Возврат денег клиенту -- отдельная операция.

Сообщение об отмене должно однозначно сослаться на исходную операцию. Тот же STAN, тот же временной контекст, в~ряде профилей -- специальные original data elements вроде DE~90. Официальные профили прямо описывают DE~90 как способ передать эмитенту значения исходной операции~\cite{fis-iso8583-guide-2023}.

\practicenote{%
  Одинаковый MTI~0100 в~двух разных IFS -- не одинаковые запросы. Один
  партнёр считает DE~32 обязательным, другой не принимает его вовсе;
  у~одного DE~22 кодирует бесконтактный сценарий кодом \texttt{071},
  у~другого -- \texttt{05F}. Код, работающий с~одним процессингом,
  молча падает на неожиданном поле у~другого. Диалектные различия
  обнаруживаются на тестовых транзакциях -- не в production.
}

\section{Bitmap: какие поля искать дальше}\label{sec:iso8583-bitmap}

После MTI идёт bitmap -- восемь или шестнадцать байт, в~которых каждый бит отвечает за присутствие одного элемента данных.
Сами значения bitmap не хранит, а~только показывает, какие поля искать дальше в~теле сообщения~\cite{fis-iso8583-guide-2023}.
\definitionnote{Bitmap}{Карта присутствия полей ISO~8583. Она не
  несёт сами значения DE, а~только сообщает, какие элементы данных
нужно читать дальше по сообщению.}

Bitmap -- не единственная возможная структура. Поздние форматы вроде TLV (Tag-Length-Value), как в EMV DE~55, делают сообщение самоописывающимся: каждое поле несёт собственный тег и~длину.
Bitmap проектировался в 1970-х -- 80-х, когда каждый байт в~телекоммуникационном канале имел цену. Сдвинуть бит для проверки присутствия поля дешевле, чем пройти по тегам. Выбор закрепился стандартом прежде, чем TLV-форматы с~сопоставимой производительностью стали широко доступны.

\begin{figure}[tbp]
  \centering
  \includefiguresvg[width=.75\linewidth]{assets/figures/ch05-iso8583/bitmap-to-fields}
  \caption{От bitmap к~полям данных.}\label{fig:bitmap-to-fields}
\end{figure}

В~нашем примере основная bitmap выглядит так:

\begin{center}
  \texttt{72 3C 04 81 20 C0 80 00}
\end{center}

Переведём байты в~двоичную форму:

\begin{center}
  \small
  \texttt{72 = 0111\,0010} \quad
  \texttt{3C = 0011\,1100} \quad
  \texttt{04 = 0000\,0100} \quad
  \texttt{81 = 1000\,0001}

  \texttt{20 = 0010\,0000} \quad
  \texttt{C0 = 1100\,0000} \quad
  \texttt{80 = 1000\,0000} \quad
  \texttt{00 = 0000\,0000}
\end{center}

Первый бит равен \texttt{0} -- дополнительной битовой карты нет, всё сообщение укладывается в~первые 64 элемента данных. Остальные единицы указывают на DE~2, DE~3, DE~4, DE~7, DE~11, DE~12, DE~13, DE~14, DE~22, DE~25, DE~32, DE~35, DE~41, DE~42 и DE~49.
Правдоподобный набор для обычной авторизации: идентификация карты, сумма, временной контекст, точка приёма и~способ ввода.

Bitmap отделяет структуру от содержания. Значения DE~4 и DE~41 ещё не прочитаны, но известно, что они есть и~идут в~порядке возрастания номеров.

\practicenote{%
  При разборе инцидента сначала смотрите на bitmap: он быстро
  показывает, есть ли в~сообщении DE~55, Track 2, DE~38 и DE~39. Часто
  этого хватает, чтобы снять половину вопросов ещё до полного чтения дампа.

}
\section{Элементы данных: пять групп вместо каталога}\label{sec:iso8583-data-elements}

Полный каталог элементов данных здесь не нужен. Важнее, какую работу поля выполняют внутри сообщения.
Для MTI~0100/MTI~0110 практически значимое раскладывается в~пять смысловых групп. Полный каталог -- в~сетевой спецификации и в IFS партнёра~\cite{fis-iso8583-guide-2023}.

\subsection*{Идентификация и маршрутизация}

DE~2 (PAN) -- чья карта. DE~35 (Track~2 Data) дублирует ответ в~более богатой форме: PAN, срок действия, сервисный код и дискреционные данные одной строкой.
DE~32 -- идентификатор эквайера или банка-инициатора. DE~37 (RRN) появляется в~ответе, в~запросах и~в~последующих сообщениях, когда операцию приходится искать в~отчётах, диспутах и операторских инструментах~\cite{fis-iso8583-guide-2023}.

Группа адресации отвечает на три вопроса: кому принадлежит карта, кто отправил запрос, по какому идентификатору ту же операцию найдут в~другом контексте.

\subsection*{Деньги и~время}

DE~3 (Processing Code) кодирует тип операции: покупка, возврат, снятие наличных, запрос баланса.
DE~4 хранит сумму с~фиксированным масштабом, который определяется экспонентом из DE~49 и~таблицей ISO~4217 (для рубля и~евро -- две минорные единицы, для иены -- ноль).
DE~7 фиксирует сетевое время передачи сообщения. DE~11 -- STAN (Systems Trace Audit Number), шестизначный номер трассировки, по которому процессинг связывает запрос с~ответом. DE~12 и DE~13 -- локальное время и~дата терминала. DE~49 -- код валюты.

Группа нужна для сверки и расследований. Сумма без валюты ничего не значит; STAN без временного контекста не уникален; локальное время без сетевого -- ненадёжный ориентир в~споре между терминалом, процессингом и эмитентом.

\subsection*{Контекст точки приёма}

DE~22 (POS Entry Mode) показывает, как получены данные карты: вручную, с~магнитной полосы, с~чипа, бесконтактно, в~интернет-операции. DE~25 (POS Condition Code) добавляет условия операции. DE~41 и DE~42 -- идентификаторы терминала и ТСП -- привязывают транзакцию к~точке приёма.

Эта группа отвечает эмитенту, \emph{где} и \emph{как} прошла операция. Одни и~те же PAN и~сумма получают разную оценку для чиповой покупки в~супермаркете и для ручного ввода реквизитов на сайте.

\subsection*{Результат авторизации}

DE~39 (Response Code) -- главное поле ответа: одобрена операция или нет. DE~38 (идентификатор авторизации) появляется при одобрении и связывает авторизацию с~последующим клирингом.

Группа -- принципиально \emph{ответная}: в~запросе её нет, потому что решение ещё не принято. Чтение MTI~0110 начинается с DE~39 -- именно в~нём сидит смысл ответа.

\subsection*{Чиповые данные}

DE~55 -- TLV-контейнер для EMV-данных: криптограмм, TVR, AID, ATC и~других тегов, которыми терминал доказывает эмитенту, что операция действительно пришла с~чипа или бесконтактного ядра~\cite{emvco-specs}.

В~каноническом примере DE~55 нет умышленно: проще без чипового хвоста разделить грамматику ISO~8583 и~детали EMV. В~реальной чиповой авторизации именно DE~55 делает сообщение длиннее и~сложнее.

Пять групп складываются в~связный ответ: кто платит, какая операция, где она происходит, какое решение вернул эмитент.

\section{Разбираем MTI~0100 и MTI~0110 до конца}\label{sec:iso8583-hex-dump}

MTI~0100 целиком:

\begin{center}
  \small
  \texttt{30 31 30 30}\\[2pt]
  \texttt{72 3C 04 81 20 C0 80 00}\\[2pt]
  \texttt{31 36 32 32 30 30 31 32 31 32 33 34 35 36 30 38 39 32}\\[2pt]
  \texttt{30 30 30 30 30 30}\\[2pt]
  \texttt{30 30 30 30 30 30 31 32 35 30 35 30}\\[2pt]
  \texttt{30 33 31 37 31 37 33 30 34 35}\\[2pt]
  \texttt{34 38 32 37 33 31}\\[2pt]
  \texttt{31 37 33 30 34 35}\\[2pt]
  \texttt{30 33 31 37}\\[2pt]
  \texttt{32 36 31 32}\\[2pt]
  \texttt{30 37 31}\\[2pt]
  \texttt{30 30}\\[2pt]
  \texttt{30 36 31 32 33 34 35 36}\\[2pt]
  \texttt{33 34 32 32 30 30 31 32 31 32 33 34 35 36 30 38 39 32}\\[2pt]
  \texttt{3D 32 36 31 32 32 30 31 31 32 33 34 35 36 37 38 39 30}\\[2pt]
  \texttt{54 45 52 4D 30 30 30 31}\\[2pt]
  \texttt{4D 45 52 43 48 41 4E 54 30 30 30 30 30 30 31}\\[2pt]
  \texttt{36 34 33}
\end{center}

\textbf{Шаг 1: MTI.} Первые четыре байта \texttt{30 31 30 30} в ASCII -- \texttt{0100}: запрос на авторизацию.

\textbf{Шаг 2: bitmap.} Следующие восемь байт \texttt{72 3C 04 81 20 C0 80 00} перечисляют поля: DE~2, DE~3, DE~4, DE~7, DE~11, DE~12, DE~13, DE~14, DE~22, DE~25, DE~32, DE~35, DE~41, DE~42, DE~49. Первый бит не установлен -- вторичной битовой карты нет.

\textbf{Шаг 3: первое переменное поле.} DE~2 идёт как LLVAR-поле. Два байта длины после bitmap, \texttt{31 36}, в ASCII -- \texttt{16}; дальше 16 цифр PAN: \texttt{2200121234560892}.

Дамп разбирается по логике одного конкретного профиля. Точные коды DE~22, DE~25, список обязательных полей и~пара сообщений отмены зависят от спецификации интерфейса. Глава даёт модель чтения, а~не универсальный справочник всех диалектов~\cite{fis-iso8583-guide-2023}.

\textbf{Шаг 4: остальные поля -- в~порядке возрастания номеров.}
Сразу после PAN: DE~3=\texttt{000000} и DE~4=\texttt{000000125050}. Код операции -- стандартная покупка, сумма -- 1\,250 рублей 50~копеек при DE~49=\texttt{643}.
Дальше DE~7=\texttt{0317173045}, DE~11=\texttt{482731}, DE~12=\texttt{173045}, DE~13=\texttt{0317}. Это временной каркас: сетевое время, STAN для связи запроса с~ответом, локальные дата и~время терминала.

Платёжный контекст уточняется следующими полями. DE~14=\texttt{2612} -- срок действия карты. DE~22=\texttt{071} в~этом профиле -- бесконтактный чиповый сценарий с~терминалом, способным принять PIN (третий разряд кодирует именно PIN entry capability терминала, а~не факт верификации PIN в~этой операции). DE~25=\texttt{00} -- обычные условия операции.
Поля маршрутизации и точки приёма: DE~32=\texttt{123456} -- эквайер или банк-инициатор; DE~35 -- Track~2 с PAN, сроком действия и сервисным кодом; DE~41=\texttt{TERM0001} и DE~42=\texttt{MERCHANT0000001} -- терминал и~ТСП. Последним приходит DE~49=\texttt{643} -- код рубля.

MTI~0100 прочитан целиком: \enquote{эквайер просит одобрить бесконтактную покупку на сумму 1\,250 рублей 50 копеек по карте с PAN \texttt{2200121234560892} в~терминале \texttt{TERM0001} у~ТСП \texttt{MERCHANT0000001}}.

Эмитент отвечает MTI~0110. Ответ оставляет только тот контекст, который достаточен для сопоставления с~запросом, и добавляет результат.
MTI меняется на MTI~0110, bitmap становится короче -- например \texttt{32 38 00 00 0E C0 80 00}, то есть DE~3, DE~4, DE~7, DE~11, DE~12, DE~13, DE~37, DE~38, DE~39, DE~41, DE~42, DE~49. PAN и Track~2 уже не нужны; к~эху сумм, времени и идентификаторов точки приёма добавляются поля результата.
DE~37 несёт RRN -- сквозной ссылочный номер для сопоставления в~клиринге. DE~38 приносит код авторизации, скажем \texttt{A1B2C3}. DE~39 фиксирует решение эмитента: \texttt{00} -- одобрение, \texttt{51} -- недостаток средств. Это типичный рисунок ответа в~одном из публично документированных профилей~\cite{fis-iso8583-guide-2023}.
Сообщение отмены -- MTI~0400 или MTI~0420 в~зависимости от IFS -- должно нести ссылочные данные на исходную авторизацию: тот же STAN, тот же временной контекст, в~ряде профилей -- специальные original data elements~\cite{fis-iso8583-guide-2023}.

\importantnote{%
  В~примере выше все числовые поля закодированы в ASCII:
  каждая цифра занимает один байт. Но в~реальных IFS цифры
  могут кодироваться упакованным числовым форматом (BCD, Binary Coded
  Decimal): две цифры в~одном байте, что вдвое сжимает числовые поля.
  ASCII удобнее при отладке в~логах; BCD типичен для продуктовых (\textit{production}) контуров,
  где каждый лишний байт увеличивал стоимость телеком-канала и~время
  обработки на железе 1980-х. Прежде чем читать дамп вручную, нужно
  узнать кодировку из IFS -- ошибка на этом этапе делает бессмысленным
  весь дальнейший разбор~\cite{fis-iso8583-guide-2023}.

}
\section{Где заканчивается ISO~8583 и~зачем нужна IFS}\label{sec:iso8583-implementations}

ISO~8583 не описывает сетевой протокол целиком. Стандарт определяет \emph{грамматику сообщения}; в~официальной аннотации ISO прямо оговорено, что метод расчёта в~предмет стандарта не входит.
Заметная часть реальной интеграции лежит вне этой грамматики~\cite{iso-8583-2023,fis-iso8583-guide-2023}.

\textbf{Транспорт и фрейминг.} Способ доставки -- по TCP, через VPN, с~каким заголовком и префиксом длины -- стандартом не предписан. Двух- или четырёхбайтовый префикс длины, отдельный application header, выбор между ASCII и EBCDIC -- соглашение конкретного процессинга.
Транспортный нейтралитет -- не упущение, а проектный выбор: стандарт 1987 года должен был одинаково работать на X.25, SNA и~любом будущем протоколе. Цена -- каждый IFS фиксирует транспорт сам, и~при смене партнёра ничего из этого не переносится.

\textbf{Криптография и ключи.} PIN block едет в DE~52, MAC -- в DE~64 или DE~128. Алгоритмы, формат PIN block, схема ключей и~порядок их обмена задаются сетью или спецификацией конкретного коммутатора, не самим ISO~8583.

\textbf{Состояние сессии.} Sign-on, echo test, cutover, управление ключами и повторная отправка опираются на MTI~0800/MTI~0810 и~на то, как конкретная система понимает таймауты, идемпотентность и повтор.

\textbf{Правила обработки.} Что считать дубликатом, когда отправлять сообщение отмены, как долго ждать ответ, какие поля обязательны для чиповой или интернет-операции -- прикладные правила интерфейса, а~не часть синтаксиса сообщения.

\importantnote{%
  Само наличие DE~64 или DE~128 не объясняет MAC. Для проверки нужны
  ключи, алгоритм, padding, состав строки и транспортная спецификация;
  это контур вокруг ISO~8583, а~не часть его базовой грамматики.

}
Прочитанного канонического запроса и~ответа для подключения нового процессинга мало. Стандарт задаёт общий язык; интеграция всегда идёт на одном из его диалектов.

IFS (Interface Specification) -- спецификация интерфейса конкретной сети, процессинга или банка-партнёра. Она фиксирует обязательные поля, кодировки, правила повторов и транспортную обвязку канала~\cite{fis-iso8583-guide-2023}.

Диалекты расходятся сразу на нескольких слоях. В~одном контуре всё кодируется в ASCII, в~другом рядом живут ASCII, BCD и~бинарные поля. Одному партнёру достаточно DE~35, другому нужен ещё DE~2, третьему -- набор частных полей под конкретный сценарий. Где-то дополнительные данные раскрываются через DE~48, DE~60, DE~61, DE~63 или DE~124, где-то собираются в~собственные подполя. То же -- с~кодами ответа, жизненным циклом сообщений отмены, cutover, транспортным заголовком, sign-on и~ключевым обменом.

Одной книги по ISO~8583 для интеграции мало. Стандарт описывает грамматику; подключение всегда упирается в IFS конкретной сети или процессинга. Только IFS говорит, какие поля обязательны для вашего канала, как кодируется сумма, где искать частные подполя и~каким должен быть ответ на ваш MTI~0800.

Детали диалектов Visa, Mastercard и НСПК здесь не разбираются. ISO~8583 задаёт \emph{грамматику}, карточная сеть -- \emph{диалект}, интеграция опирается на \emph{IFS}. Полный справочник элементов данных и~дампы MTI~0200, MTI~0420 и~сетевых служебных сообщений живут в~сетевой спецификации и в IFS; книга оставляет за собой устойчивую модель чтения.

\practicenote{%
  Новый процессинг подключают по конкретной IFS: именно там
  зафиксированы сопоставление полей, sign-on, таймауты, отмены и
  фактическое кодирование даже простых полей вроде суммы, даты и STAN.

}
