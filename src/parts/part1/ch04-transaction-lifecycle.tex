В~первой главе мы~увидели, что карточный платёж~--- операция

каждую фазу в~деталях~--- что~именно происходит на~каждом шаге,

Затем рассмотрим два архитектурных варианта (dual message и~single

\section{Авторизация}
\label{sec:tx-authorization}

Авторизация~--- единственная фаза карточного платежа, которая
происходит в~реальном времени и~видна покупателю: терминал
показывает \enquote{одобрено} или \enquote{отказано}, и~на~этом
взаимодействие покупателя с~платёжной системой заканчивается.
Всё остальное~--- клиринг, settlement, возможный диспут~--- происходит
за~кулисами, иногда через дни после покупки. Но~именно
в~эти несколько секунд авторизации закладывается фундамент, на~котором
держится вся дальнейшая обработка транзакции.

\subsection*{Путь авторизационного запроса}

Когда держатель карты прикладывает карту к~терминалу (или~вставляет
её, или~проводит магнитной полосой), терминал считывает данные
карты и~формирует авторизационный запрос~--- структурированное
сообщение в~формате \iso{8583} (подробно о~формате~---
в~главе~\ref{ch:iso8583}), содержащее, как минимум, следующие
данные:

\begin{itemize}
  \item PAN (\de{2}) и~данные Track~2 (\de{35}) или~эквивалент
        чиповых данных (\de{55});
  \item сумму и~валюту операции (\de{4}, \de{49});
  \item идентификатор терминала (\de{41}) и~мерчанта (\de{42});
  \item категорию мерчанта, MCC (\de{18});
  \item тип операции~--- покупка, снятие наличных, запрос баланса
        (определяется MTI и~processing code, \de{3}).
\end{itemize}

Сформированное сообщение отправляется эквайеру~--- банку или~процессору,
обслуживающему данный терминал. Эквайер выполняет первичные проверки:
валидность терминала, соответствие MCC, базовые антифрод-правила
на~стороне эквайера,~--- и~маршрутизирует запрос в~карточную
сеть~\sidecite{visa-core-rules}. Карточная сеть, в~свою очередь,
по~IIN/BIN определяет эмитента (или~его процессор) и~передаёт
запрос ему.

Весь этот путь~--- терминал, эквайер, карточная сеть, эмитент~---
и~обратно должен уложиться в~жёсткие временны\'{е} рамки. Карточные
сети устанавливают максимальное время ответа (response time):
если эмитент не~ответил в~отведённое время, сеть генерирует
таймаут, и~терминал получает отказ. Для~\scheme{Visa} и~\scheme{Mastercard}
типичный лимит на~ответ эмитента составляет порядка нескольких
секунд от~момента получения запроса
сетью~\todocheck{точные значения таймаутов авторизации
Visa~(VisaNet) и~Mastercard~(Banknet)~--- проверить по~TADG
и~Mastercard Authorization Manual}.

\subsection*{Решение эмитента}

Эмитент (или~его процессор), получив авторизационный запрос,
принимает решение~--- одобрить или~отклонить~--- на~основе
нескольких групп проверок, которые выполняются за~доли секунды:

\begin{enumerate}
  \item \textbf{Проверка карты.} Существует~ли карта с~таким PAN?
        Не~заблокирована~ли она? Не~истёк~ли срок действия?
        Совпадает~ли CVV или~криптограмма?
  \item \textbf{Проверка баланса.} Достаточно~ли средств на~счёте
        (для~дебетовой карты) или~доступного кредитного лимита
        (для~кредитной)?
  \item \textbf{Лимиты и~ограничения.} Не~превышен~ли дневной лимит
        на~операции? Не~заблокирована~ли карта для~данного типа
        операций (например, запрет на~интернет-покупки)? Не~ограничена~ли
        география использования?
  \item \textbf{Антифрод.} Соответствует~ли операция типичному
        поведению держателя? Нет~ли признаков компрометации карты?
        Не~входит~ли мерчант в~списки повышенного риска?
  \item \textbf{Аутентификация держателя.} Для~чиповых транзакций~---
        верификация криптограммы (ARQC); для~CNP~--- результат
        3-D~Secure, если он~был; для~магнитной полосы~--- CVV.
\end{enumerate}

По~результатам этих проверок эмитент формирует ответ с~кодом
(response code, \de{39}): \rc{00}~--- approved, \rc{05}~--- do~not
honor (общий отказ без детализации), \rc{51}~--- insufficient funds,
\rc{14}~--- invalid card number, и~так далее; полный список
кодов ответа определяется стандартом \iso{8583}, однако каждая
карточная сеть дополняет его своими специфическими
значениями~\todocite{таблица response codes: Visa TADG
и~Mastercard IPM Reference}.

Если транзакция одобрена, эмитент присваивает ей код авторизации
(authorization code, \de{38})~--- шестисимвольный идентификатор,
который привязывает одобрение к~конкретному запросу
и~будет использоваться на~этапе клиринга для~сопоставления
авторизации с~финансовой записью. Одновременно эмитент блокирует
(hold) запрошенную сумму на~счёте держателя: деньги ещё
не~списаны, но~уже недоступны для~других операций.

\subsection*{Hold и его срок жизни}

Hold~--- блокировка суммы на~счёте держателя~--- существует
не~бессрочно. Если эквайер не~представит клиринговую запись
в~течение определённого времени, hold снимается, и~средства
снова становятся доступными. Правила схем устанавливают
максимальные сроки: для~стандартных покупок типичный срок
составляет от~семи до~тридцати дней в~зависимости от~категории
мерчанта (MCC), а~для~некоторых категорий~--- гостиницы,
аренда автомобилей~--- он~может быть
длиннее~\todocite{сроки hold по категориям MCC~---
Visa Core Rules, Mastercard Rules}.

\begin{practice}
Если клиент жалуется, что \enquote{деньги списали, но~покупка
не~прошла}, чаще всего речь идёт не~о~списании, а~о~hold:
авторизация была одобрена и~средства заблокированы, но~по~какой-то
причине мерчант не~завершил операцию (например, отменил заказ
на~своей стороне, не~отправив reversal). Деньги вернутся
автоматически по~истечении hold, но~клиенту это не~очевидно,
и~он~видит в~мобильном банке уменьшенный доступный баланс.
\end{practice}


\section{Клиринг}
\label{sec:tx-clearing}

Если авторизация~--- это обещание заплатить, то~клиринг~--- это
предъявление счёта: формальное требование эквайера к~эмитенту
исполнить данное обещание. Клиринг происходит не~в~реальном
времени, а~пакетами (batches), и~его механика принципиально
отличается от~авторизации~--- здесь нет мгновенного ответа,
нет покупателя, ожидающего результат, и~нет жёстких таймаутов
в~несколько секунд.

\subsection*{Формирование клирингового файла}

В~конце операционного дня (или~несколько раз в~день, в~зависимости
от~настроек) эквайер формирует клиринговый файл (clearing file)~---
пакет записей обо~всех транзакциях, прошедших авторизацию за~период.
Каждая запись содержит реквизиты транзакции: PAN, сумму (которая
может отличаться от~авторизованной~--- об~этом ниже), валюту, дату
и~время, идентификатор мерчанта, MCC, код авторизации и~другие
данные, необходимые для~расчёта~\sidecite{mc-rules}.

Формат клирингового файла зависит от~карточной сети. \scheme{Visa}
использует формат TC (Transaction Clearing)~--- часть системы
VisaNet; \scheme{Mastercard}~--- формат IPM (Integrated Processing
Message); НСПК~--- собственный формат для~внутрироссийских
операций. Несмотря на~различия в~форматах, логика одна и~та~же:
набор записей с~реквизитами транзакций, которые сеть должна
обработать и~по~которым нужно произвести
расчёт~\todocite{форматы клиринговых файлов~---
Visa TC, Mastercard IPM}.

\subsection*{Сопоставление с авторизацией}

Получив клиринговый файл, карточная сеть сопоставляет каждую
клиринговую запись с~соответствующей авторизацией~--- по~комбинации
PAN, суммы, кода авторизации и~даты. Это сопоставление
(matching)~--- нетривиальная задача, потому что сумма в~клиринге
может отличаться от~авторизованной. Типичный пример~--- ресторан:
авторизация проходит на~сумму заказа, а~в~клиринг попадает сумма
с~чаевыми, которые клиент добавил позже. Правила схем допускают
такое расхождение в~определённых пределах, зависящих от~MCC:
для~ресторанов и~гостиниц допуск шире, для~обычного ретейла~---
уже~\todocite{допустимые расхождения авторизация/клиринг
по~MCC~--- Visa Core Rules}.

Если клиринговая запись не~находит соответствующей авторизации~---
например, потому что авторизация была отменена reversal-ом,
или~потому что hold уже истёк,~--- запись всё~равно может быть
обработана: это так называемая late presentment, и~карточные
сети допускают её в~определённых временны\'{х} рамках, хотя
поздняя отправка может повлечь штрафные комиссии
для~эквайера~\sidecite{visa-core-rules}.

\subsection*{Расчёт interchange и комиссий}

На~этапе клиринга карточная сеть рассчитывает interchange
для~каждой транзакции~--- ту~самую межбанковскую комиссию, которую
эквайер платит эмитенту и~которую мы~подробно разобрали
в~главе~\ref{ch:ecosystem}. Ставка interchange зависит
от~множества параметров, зафиксированных в~клиринговой записи:
типа карты (дебетовая, кредитная, корпоративная), MCC,
способа приёма (чип, магнитная полоса, CNP), географии
(внутренняя или~трансграничная операция) и~ряда
дополнительных признаков.

Одновременно сеть начисляет собственные комиссии (scheme fees,
assessments): плату за~обработку, за~трансграничную маршрутизацию,
за~использование специальных сервисов. По~итогам обработки
всех клиринговых записей за~период сеть формирует чистые расчётные
позиции (net settlement positions) для~каждого банка-участника~---
итоговые суммы, которые один банк должен другому с~учётом
взаимозачёта встречных потоков.


\section{Settlement}
\label{sec:tx-settlement}

Settlement~--- фактическое перемещение денег между банками~---
завершает жизненный цикл транзакции. До~этого момента деньги
не~покидали счёт держателя карты: авторизация лишь заблокировала
сумму, клиринг~--- сформировал требование; теперь, на~этапе
settlement, происходит реальное межбанковское
перемещение~\sidecite{fz-161}.

\subsection*{Нетто-расчёт}

Карточные сети используют нетто-расчёт (net settlement):
вместо того чтобы проводить отдельную межбанковскую операцию
для~каждой транзакции, сеть суммирует все встречные обязательства
между парами банков за~расчётный период и~переводит только
разницу. Если за~день банк~А как эквайер должен банку~Б как
эмитенту миллион рублей interchange, а~банк~Б как эквайер должен
банку~А как эмитенту восемьсот тысяч,~--- фактически переводятся
только двести тысяч от~А к~Б. Этот механизм многократно сокращает
число и~объём межбанковских проводок.

Расчёт производится через расчётный банк (settlement bank),
назначенный карточной сетью. Для~внутрироссийских транзакций
расчёт по~всем карточным сетям проходит через НСПК;
для~международных транзакций \scheme{Visa} и~\scheme{Mastercard}
используют собственных расчётных агентов~--- как правило, крупные
международные банки в~соответствующих валютах.

\subsection*{Когда мерчант получает деньги}

С~точки зрения мерчанта, settlement~--- это момент, когда деньги
за~проданный товар или~оказанную услугу поступают на~его расчётный
счёт. Однако между settlement-ом в~карточной сети и~зачислением
на~счёт мерчанта может быть ещё один шаг: эквайер, получив средства
от~сети, перечисляет их мерчанту в~соответствии с~условиями
договора эквайринга~--- ежедневно, через день или~по~иному графику,
за~вычетом MSC.

Типичный срок от~момента покупки до~зачисления денег мерчанту~---
от~одного до~трёх рабочих
дней~\todocheck{типичные сроки settlement и~зачисления
мерчанту~--- проверить по~договорам эквайринга и~правилам НСПК
для~внутрироссийских операций}. Для~мерчанта, работающего
с~большим оборотом или~узкой маржой, каждый дополнительный день~---
это замороженные оборотные средства, поэтому скорость settlement
часто становится предметом переговоров с~эквайером.

\section{Dual message и single message}
\label{sec:tx-dual-single}

До~сих пор мы~описывали схему, в~которой авторизация и~клиринг~---
два отдельных сообщения, разнесённых во~времени: сначала терминал
спрашивает \enquote{можно?} (авторизация), потом эквайер говорит
\enquote{вот счёт} (клиринг). Эта схема называется dual message
(двухсообщенческая), и~она является стандартом для~большинства
карточных покупок в~торговых точках~\sidecite{visa-core-rules}.

Однако существует и~другая схема~--- single message
(односообщенческая), в~которой авторизация и~финансовая запись
совмещены в~одном сообщении: запрос одновременно спрашивает
разрешение \emph{и}~фиксирует финансовое обязательство. Ответ
эмитента в~single message означает не~только \enquote{одобрено},
но~и~\enquote{деньги можно считать списанными}~--- отдельного
клирингового шага не~требуется.

\subsection*{Где используется каждая схема}

Dual message доминирует в~POS-транзакциях (покупки в~магазинах,
ресторанах, e-commerce), потому что допускает расхождение между
авторизованной и~финальной суммой (чаевые, довзвешивание товара,
частичная отмена) и~позволяет эквайеру группировать транзакции
в~пакеты для~оптимизации обработки.

Single message~--- классическая модель для~банкоматов (ATM):
когда держатель карты снимает наличные, нет смысла разделять
авторизацию и~финансовую запись, потому что наличные выданы
физически и~отменить операцию уже невозможно. По~той~же логике
single message используется в~некоторых дебетовых сетях (PIN debit
в~США), где транзакция всегда проходит на~точную сумму и~не~предполагает
последующей корректировки.

На~уровне \iso{8583} различие выражается в~MTI (Message Type
Indicator): авторизационный запрос dual message имеет MTI
\mti{0100}, а~финансовый запрос single message~--- \mti{0200};
соответствующие ответы~--- \mti{0110} и~\mti{0210}. Подробно
о~структуре MTI мы~поговорим в~следующей
главе~(\ref{ch:iso8583}).

\begin{important}
Выбор между dual и~single message~--- не~решение мерчанта
или~разработчика, а~свойство канала и~правил карточной сети.
POS-терминал, работающий по~протоколу \scheme{Visa} или~\scheme{Mastercard},
отправляет dual message; банкомат, работающий в~сети ATM,~---
single message. Попытка отправить финансовое сообщение (\mti{0200})
туда, где ожидается авторизационное (\mti{0100}), приведёт
к~отказу сети.
\end{important}

\subsection*{Гибридные варианты}

На~практике граница между dual и~single message размывается.
\scheme{Visa} предлагает схему Visa~DPS (Dual Processing Service),
при~которой одобренная авторизация автоматически трактуется
и~как клиринговая запись, если эквайер не~отправил отдельный
клиринг в~течение определённого
срока~\todocite{Visa DPS / auto-clearing~--- TADG}. \scheme{Mastercard}
имеет аналогичный механизм DMSA (Dual Message Single Authorization),
при~котором авторизация и~клиринг обрабатываются единой
инфраструктурой, хотя формально остаются двумя
сообщениями~\todocite{Mastercard DMSA~--- IPM Reference}.
Эти гибриды упрощают жизнь эквайерам, но~добавляют сложности
процессинговым системам, которым приходится учитывать оба сценария.


\section{Три способа отменить платёж}
\label{sec:tx-cancellation}

Отмена карточного платежа~--- одна из~тех областей, где разрыв
между авторизацией и~settlement создаёт наибольшую сложность.
В~отличие от~банковского перевода, который либо прошёл, либо нет,
карточная транзакция в~каждый момент времени находится в~одной
из~нескольких фаз, и~способ отмены зависит от~того, в~какой фазе
транзакция находится прямо сейчас.

\subsection*{Reversal: отмена авторизации}

Reversal~--- сообщение, которое отменяет авторизацию \emph{до~того},
как транзакция попала в~клиринг. С~точки зрения \iso{8583}
reversal~--- это отдельный тип сообщения (MTI \mti{0400}),
отправляемый эквайером в~карточную сеть с~просьбой аннулировать
ранее одобренную авторизацию.

Reversal возникает в~нескольких сценариях: покупатель передумал
у~кассы, терминал не~получил ответ на~авторизацию (таймаут)
и~отправляет reversal на~всякий случай, кассир ошибся с~суммой
и~отменяет операцию. При~успешном reversal эмитент снимает hold
со~счёта держателя, и~средства снова становятся доступными~---
обычно мгновенно или~в~течение нескольких минут.

Reversal~--- самый \enquote{чистый} способ отмены: он~работает
в~реальном времени, не~порождает финансовой записи и~не~требует
возврата денег, потому что деньги ещё не~перемещались.
Но~reversal возможен только до~клиринга; после того как транзакция
попала в~клиринговый файл, отменить авторизацию уже
нельзя~\sidecite{mc-rules}.

\subsection*{Void: отмена до расчёта}

Void (аннулирование) применяется, когда транзакция уже попала
или~вот-вот попадёт в~клиринговый файл, но~расчёт ещё
не~произведён. Эквайер удаляет (или~помечает как отменённую)
клиринговую запись до~отправки файла в~карточную сеть, или~отправляет
корректирующую запись, обнуляющую исходную транзакцию.

С~точки зрения держателя карты void и~reversal выглядят
одинаково: hold снимается, деньги возвращаются на~счёт.
Но~с~технической стороны void может обрабатываться медленнее,
чем reversal, потому что он~привязан к~циклу клиринга:
если клиринговый файл уже отправлен, корректировка попадёт
только в~следующий файл.

\begin{practice}
В~API большинства PSP и~платёжных шлюзов вы~встретите метод
\texttt{cancel} или~\texttt{void}, который на~самом деле
абстрагирует оба механизма: если транзакция ещё не~ушла
в~клиринг, PSP отправляет reversal; если ушла, но~settlement
не~произошёл,~--- void. Для~разработчика, интегрирующего платежи,
это удобная абстракция, но~важно понимать, что~внутри она
покрывает разные процессы с~разной скоростью возврата средств
клиенту.
\end{practice}

\subsection*{Refund: возврат после расчёта}

Refund (возврат)~--- единственный способ вернуть деньги после
того, как settlement завершён и~средства фактически перемещены
между банками. В~отличие от~reversal и~void, которые отменяют
исходную транзакцию, refund~--- это \emph{новая} транзакция
в~обратном направлении: эквайер инициирует перевод от~мерчанта
к~держателю карты через ту~же карточную сеть.

Refund проходит тот~же цикл, что и~обычная покупка,~---
авторизация (не~всегда), клиринг, settlement,~--- но~с~обратным
знаком: теперь interchange платит эмитент эквайеру
(или, точнее, interchange по~исходной транзакции может быть
частично возвращён~--- правила зависят от~сети и~типа
операции)~\todocite{правила возврата interchange при~refund~---
Visa Core Rules, Mastercard Rules}.

Для~держателя карты refund~--- самый медленный вариант возврата:
средства могут поступить на~счёт через несколько рабочих дней,
потому что refund должен пройти полный цикл клиринга и~settlement.
Для~мерчанта refund~--- самый дорогой вариант: помимо возврата
суммы покупки, мерчант может потерять часть или~весь interchange
по~исходной транзакции, а~эквайер может взять дополнительную
комиссию за~обработку возврата.

\begin{important}
Распространённая ошибка при~проектировании платёжной интеграции~---
использовать refund там, где достаточно reversal или~void.
Если клиент отменяет заказ через минуту после оплаты, а~система
мерчанта отправляет refund вместо reversal, клиент будет ждать
возврата несколько дней вместо нескольких минут, мерчант потеряет
interchange, а~эквайер зафиксирует лишнюю транзакцию. Правило
простое: отменяй как можно раньше~--- reversal лучше void,
void лучше refund.
\end{important}


\section{Граничные случаи}
\label{sec:tx-edge-cases}

Схема \enquote{авторизация на~точную сумму, клиринг на~ту~же
сумму, settlement через день}~--- идеальный случай, который
покрывает основную массу розничных покупок. Но~платёжная индустрия
полна сценариев, не~укладывающихся в~эту модель, и~каждый из~них
требует своего механизма. Эти граничные случаи~--- не~экзотика:
любой разработчик, интегрирующий приём платежей, рано или~поздно
столкнётся с~каждым из~них.

\subsection*{Pre-authorization}

Pre-authorization (предавторизация, pre-auth)~--- авторизация,
при~которой финальная сумма заранее неизвестна. Классические
примеры~--- гостиница и~аренда автомобиля: в~момент заселения
или~получения автомобиля мерчант авторизует оценочную сумму
(ожидаемую стоимость проживания или~депозит за~автомобиль),
а~финальная сумма определяется при~выезде или~возврате
и~может быть как меньше, так и~больше авторизованной.

С~точки зрения протокола pre-auth~--- обычная авторизация
с~особым processing code, указывающим, что сумма предварительная.
Эмитент устанавливает hold, но~знает, что клиринг может прийти
на~другую сумму. Правила схем разрешают эквайеру отправить
клиринг на~сумму, превышающую авторизованную, в~определённых
пределах (обычно 15--20\% для~гостиниц), без необходимости
запрашивать дополнительную
авторизацию~\todocheck{допустимое превышение pre-auth
для~гостиниц и~аренды~--- проверить по~Visa и~Mastercard rules}.

\begin{practice}
Если вы~проектируете систему бронирования или~маркетплейс услуг
с~переменной стоимостью, pre-auth~--- ваш основной инструмент.
Но~учтите два нюанса: во-первых, hold на~карте клиента будет
стоять на~полную авторизованную сумму, даже если итоговая покажется
меньше; во-вторых, если финальная сумма окажется значительно
больше авторизованной, вам потребуется либо инкрементальная
авторизация, либо новая транзакция на~разницу.
\end{practice}

\subsection*{Incremental authorization}

Incremental authorization (инкрементальная авторизация) позволяет
увеличить сумму ранее одобренной авторизации без~отмены исходной.
Мерчант отправляет дополнительный авторизационный запрос, ссылающийся
на~исходный (через код авторизации или~специальный идентификатор),
и~эмитент увеличивает hold на~запрошенную
сумму~\sidecite{visa-core-rules}.

Типичный сценарий~--- гостиница, где гость продлевает пребывание
или~заказывает дополнительные услуги: вместо того чтобы
авторизовать и~отменять повторно, мерчант наращивает сумму
инкрементально. Не~все карточные сети и~не~все эмитенты
поддерживают incremental authorization~--- при~проектировании
интеграции необходимо проверять поддержку конкретной сети
и~иметь fallback-сценарий (например, новую авторизацию
на~дополнительную сумму).

\subsection*{Partial authorization}

Partial authorization (частичная авторизация)~--- сценарий,
в~котором эмитент одобряет транзакцию не~на~полную запрошенную
сумму, а~лишь на~часть. Например, держатель карты пытается
оплатить покупку на~тысячу рублей, но~на~карте доступно
только семьсот~--- вместо полного отказа (\rc{51}, insufficient
funds) эмитент может одобрить операцию на~семьсот рублей,
а~мерчант предлагает клиенту доплатить оставшиеся триста
другим способом (наличными, другой картой).

Partial authorization требует поддержки на~всех уровнях:
терминал должен уметь обработать частичное одобрение и~предложить
клиенту доплату, POS-система~--- скорректировать
чек, а~кассир~--- понять, что происходит. На~практике partial
authorization чаще всего встречается в~сценариях с~предоплаченными
(prepaid) картами и~подарочными картами (gift cards), где~остаток
заранее известен и~ограничен.

\begin{important}
Если ваша система не~поддерживает partial authorization, убедитесь,
что терминал или~платёжный шлюз не~отправляет соответствующий флаг
в~авторизационном запросе (\de{60} или~специфичное поле сети).
Получить частичное одобрение и~не~обработать его корректно~---
верный путь к~расхождениям в~учёте: мерчант отпустит товар
на~полную сумму, а~спишется только часть.
\end{important}

\subsection*{Deferred authorization}

Deferred authorization (отложенная авторизация)~--- механизм,
при~котором транзакция совершается без~онлайн-авторизации,
а~авторизационный запрос отправляется позже, когда связь
восстановлена. Классические сценарии~--- оплата на~борту самолёта
(терминал работает офлайн, авторизации отправляются после
посадки), паркинг (сумма определяется по~времени стоянки),
платные дороги и~общественный транспорт.

В~режиме deferred authorization терминал собирает данные карты,
фиксирует операцию локально и~позже формирует авторизационный
запрос с~соответствующим флагом, указывающим, что транзакция
уже совершена. Эмитент может одобрить или~отклонить этот запрос,
но~товар или~услуга уже предоставлены~--- что создаёт кредитный
риск для~эквайера и~мерчанта. Именно поэтому deferred authorization
разрешена только для~определённых MCC и~при~соблюдении лимитов
на~сумму отдельной операции, устанавливаемых карточными
сетями~\todocite{правила deferred authorization~--- MCC,
лимиты сумм~--- Visa, Mastercard rules}.

\subsection*{Force post}

Force post (принудительная отправка)~--- механизм, при~котором
мерчант отправляет транзакцию в~клиринг без~предварительной
онлайн-авторизации, используя код авторизации, полученный
другим путём~--- как правило, по~телефону (voice authorization).

Сценарий типичен для~ситуаций, когда онлайн-авторизация невозможна
(терминал не~работает, связь пропала), но~мерчанту нужно провести
платёж: он~звонит в~авторизационный центр эквайера (или~эмитента),
сообщает данные карты, получает код авторизации голосом
и~впоследствии отправляет транзакцию в~клиринг с~этим кодом.
Карточная сеть обрабатывает такую транзакцию как имеющую
авторизацию, хотя онлайн-запроса через сеть не~было.

Force post несёт повышенные риски: голосовая авторизация
не~включает проверку криптограммы или~CVV (кроме случаев, когда
оператор запрашивает CVV2 устно), и~ответственность за~последствия
ложится на~мерчанта. С~развитием устойчивых каналов связи
и~офлайн-режимов терминалов force post становится всё более
редким механизмом, но~полностью не~исчезает~--- особенно
в~авиаиндустрии и~в~гостиничном бизнесе, где он~по-прежнему
используется как запасной вариант.

\section*{Итоги главы}

\begin{itemize}
  \item Карточная транзакция проходит три фазы: авторизация
        (мгновенное одобрение, hold на~счёте), клиринг (пакетная
        отправка финансовых записей, расчёт interchange) и~settlement
        (фактическое перемещение денег между банками через
        нетто-расчёт).
  \item Dual message (авторизация и~клиринг~--- два отдельных
        сообщения)~--- стандарт для~POS-покупок; single message
        (авторизация и~финансовая запись совмещены)~--- стандарт
        для~банкоматов и~PIN-дебетовых сетей.
  \item Три способа отмены зависят от~фазы транзакции: reversal
        отменяет авторизацию до~клиринга (мгновенно, бесплатно),
        void убирает запись из~клиринга до~settlement (часы),
        refund создаёт обратную транзакцию после settlement
        (дни, с~потерей interchange). Правило: отменяй как можно
        раньше.
  \item Граничные случаи~--- pre-auth, incremental auth, partial
        auth, deferred auth, force~post~--- нарушают идеальную
        схему \enquote{авторизация = клиринг = settlement
        на~одну сумму} и~требуют явной поддержки в~коде;
        игнорирование этих сценариев~--- частая причина расхождений
        в~учёте и~потерь.
\end{itemize}
