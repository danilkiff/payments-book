\chapter{Reconciliation: где теряются деньги}
\label{ch:reconciliation}

\begin{kaobox}[title=Что вы узнаете из этой главы]
  Сверка~--- это процесс, без~которого деньги теряются незаметно, а~проблема
  обнаруживается через месяц или~квартал. Из~этой главы вы~узнаете, из~каких
  трёх слоёв состоит reconciliation и~почему каждый нужен; как~устроены
  клиринговые файлы TC33 и~IPM; что~такое net settlement и~почему итоговый
  перевод не~равен сумме транзакций; как~обрабатывать исключения~---
  mismatches, unmatched records, late presentment; и~где мерчант теряет
  деньги при~мультивалютном settlement.
\end{kaobox}

Представим обычную жалобу финансовой команды: вчера авторизаций
было на~миллион, в~клиринге сегодня другая сумма, а~на~settlement-счёт
пришло третье число. Reconciliation начинается ровно там, где одна
и~та~же покупка начинает жить в~нескольких реестрах сразу и~перестаёт
совпадать сама с~собой без~дополнительной работы.


\section{Три разрыва, которые нужно закрыть}
\label{sec:recon-gaps}

Деньги в~карточной системе движутся в~три шага~--- авторизация, клиринг,
settlement~--- и~каждый из~них порождает собственный информационный поток.
Авторизационный запрос фиксирует намерение провести транзакцию;
клиринговая запись~--- факт завершённой операции; банковская выписка~---
факт реального движения средств. Три потока не~обязаны совпадать идеально,
и~именно там, где они~расходятся, живёт reconciliation.

Первый разрыв~--- между авторизацией и~клиринговой записью. Авторизованная
сумма и~сумма к~клирингу могут отличаться~--- это~законно и~предусмотрено
правилами схем~\sidecite{visa-core-rules}. Ресторан авторизует счёт на~1\,000~₽,
клиент добавляет чаевые~--- к~клирингу идёт 1\,150~₽. Отель делает
pre-authorization на~сумму ожидаемого проживания, при~выезде итоговая сумма
больше. Авиакомпания авторизует без~суммы (partial shipment authorization)
и~уточняет при~клиринге. Сверка на~этом уровне проверяет: есть~ли
к~каждой клиринговой записи соответствующая авторизация и~не~выходит~ли
расхождение за~допустимые пределы.

Второй разрыв~--- между клиринговой записью и~settlement. Клиринговая
запись~--- это~одна транзакция в~файле; settlement~--- итоговая позиция
за~день по~всем транзакциям, с~учётом interchange, сборов и~возможных
chargebacks. Сумма settlement никогда не~равна сумме клиринговых
транзакций: из~неё вычитается interchange в~пользу эмитента, сборы схемы,
сборы эквайера, и~прибавляются доходные позиции (например, interchange
по~транзакциям, где мерчант выступает эмитентом). Сверка на~этом уровне
reconciles клиринговый файл с~итоговой settlement-позицией.

Третий разрыв~--- между settlement-позицией и~банковской выпиской.
Settlement-позиция~--- это~число в~системе схемы или~эквайера, тогда как~реальное
движение денег~--- это~дебет или~кредит по~корреспондентскому счёту.
Они~должны совпадать, но~тайминг, валюта пересчёта и~банковские комиссии
могут создавать расхождения. Сверка на~этом уровне сопоставляет
settlement-файл эквайера с~банковской выпиской по~settlement-счёту.

Для~мерчанта, подключённого через PSP, часть этих уровней абстрагирована:
PSP предоставляет единую выплату с~уже рассчитанными комиссиями.
Но~для~прямого эквайера, банка или~крупного процессора все три уровня
остаются собственной задачей~--- и~именно здесь сосредоточена основная
часть ручной работы в~финансовых операциях.


Итак, reconciliation закрывает не~один разрыв, а~три разных
бухгалтерских и~операционных разрыва между намерением,
финансовой записью и~реальным движением денег. Теперь можно
перейти к~первому из~конкретных источников правды~--- клиринговым
файлам, начиная с~TC33.

\section{TC33: клиринговый файл Visa}
\label{sec:recon-tc33}

TC33~--- Transaction Clearing~33~--- это~формат клирингового файла,
по~которому эквайеры передают завершённые транзакции в~BASE~II~\sidecite{visa-core-rules}.
Число~33 в~названии~--- исторический идентификатор типа записи в~файле;
оно~не~несёт особого смысла для~современного разработчика, но~остаётся
неотъемлемой частью терминологии.

Файл TC33 имеет иерархическую структуру: \textbf{Header Record}~---
метаданные файла (дата, идентификатор эквайера, порядковый номер файла);
\textbf{Batch Records}~--- группировка транзакций по~мерчантам или~терминалам;
\textbf{Transaction Records}~--- данные каждой отдельной транзакции;
\textbf{Summary Records}~--- итоги по~batch; \textbf{Trailer Record}~---
контрольные суммы по~всему файлу~\sidecite{visa-core-rules}. Контрольные
суммы в~trailer~--- не~формальность: они~позволяют получателю файла
немедленно проверить целостность, не~разбирая все записи.

Ключевые поля Transaction Record, важные для~сверки: PAN (или~токен),
сумма в~валюте транзакции, сумма в~валюте расчётов, MCC, дата транзакции,
дата обработки, STAN (System Trace Audit Number~--- тот~же, что~использовался
в~авторизации), Terminal~ID, Merchant~ID, Response Code при~авторизации,
IRD~(Interchange Reimbursement Detail~--- код interchange-программы,
определяющий ставку). Поле IRD~--- ключ к~расчёту interchange-сборов:
именно оно~определяет, в~какой строке interchange-таблицы искать ставку
для~данной транзакции.

Один из~источников путаницы~--- дата транзакции и~дата обработки
в~TC33 могут отличаться на~несколько дней. Транзакция, совершённая
в~пятницу вечером, может попасть в~клиринговый файл только в~понедельник~---
особенно если POS-терминал в~выходные не~отправлял batch. Для~сверки
это~означает, что~авторизация и~клиринговая запись для~одной транзакции
будут в~разных daily-срезах~--- matcher engine должен уметь соединять
их~по~STAN и~Terminal~ID, а~не~только по~дате.

Полная структура TC33~--- поля, форматы, допустимые значения~--- вынесена
в~онлайн-приложение к~книге.


\section{IPM: клиринговый файл Mastercard}
\label{sec:recon-ipm}

Mastercard использует формат IPM~--- Integrated Product Messages~---
для~передачи клиринговых данных в~Banknet~\sidecite{mc-rules}. IPM~--- более
гибкий формат по~сравнению с~TC33: он~основан на~ISO~8583-подобной
структуре с~bitmap-навигацией, что~делает его~легче расширяемым, но~немного
сложнее в~парсинге для~тех, кто привык к~фиксированному формату TC33.

Ключевая концепция IPM~--- Private Data Subelement~(PDS). Помимо стандартных
полей ISO~8583, каждая запись содержит набор PDS-блоков, в~которых
Mastercard передаёт собственную специфику: данные о~программе interchange,
дополнительные MCC-квалификаторы, параметры транзакции, которых нет
в~стандартном ISO~8583. Для~правильного парсинга IPM нужно не~только
разбирать стандартный bitmap, но~и~уметь извлекать PDS-структуры~---
это~отдельный уровень навигации внутри записи.

GCMS~(Global Clearing Management System)~--- это~серверная часть
Mastercard, обрабатывающая IPM-файлы. Эквайер передаёт IPM в~GCMS,
тот~маршрутизирует данные эмитентам, рассчитывает interchange,
формирует settlement-позицию. Схема взаимодействия эквайер~→~GCMS~→~эмитент
параллельна BASE~I/BASE~II у~Visa: авторизация~--- через Banknet в~реальном
времени, клиринг~--- пакетным файлом через GCMS~\sidecite{mc-rules}.

Принципиальных различий между TC33 и~IPM для~целей reconciliation два.
Первое~--- именование полей и~структура bitmap: парсеры для~этих форматов~---
разные компоненты. Второе~--- тайминг settlement: у~Mastercard может быть
иное расписание расчётных окон по~сравнению с~Visa, что~важно для~сверки
на~уровне банковской выписки~\sidecite{mc-rules}. Операционным аналитикам,
работающим с~обеими схемами, приходится либо поддерживать два отдельных
парсера, либо строить унифицированное промежуточное представление~---
в~большинстве промышленных reconciliation-систем выбирают второй путь.


\section{Net settlement vs~gross settlement}
\label{sec:recon-net-vs-gross}

Когда суммируешь все транзакции за~день и~сравниваешь с~суммой, поступившей
на~settlement-счёт, числа не~совпадают~--- и~это~нормально. Понимание
модели settlement~--- ключ к~тому, чтобы это~несовпадение не~выглядело
потерями.

При~\textbf{net settlement} схема зачисляет на~счёт эквайера разницу
между суммой транзакций к~получению (дебет эмитента) и~суммой транзакций
к~выплате (interchange, сборы, возвраты). Итоговый перевод один~---
нетированная позиция за~расчётный период. Это~стандартная модель для~карточных
схем: \scheme{Visa} и~\scheme{Mastercard} рассчитывают net settlement
позицию, и~Центральный банк (или~расчётный банк схемы) выполняет один
перевод эквайеру~\sidecite{visa-core-rules}\sidecite{mc-rules}.

При~\textbf{gross settlement} каждая транзакция или~группа транзакций
перечисляется отдельно, без~нетирования. Эта~модель характерна для~систем
вроде СБП: каждый перевод финален и~независим, нет~механизма объединить
все в~одну нетированную позицию~\sidecite{nspk-sbp}.

Для~финансового учёта важно понимать, как~именно ваш эквайер реализует
settlement. Некоторые эквайеры предоставляют gross-выплаты на~уровне
мерчанта, нетируя схемные сборы на~своей стороне,~--- тогда мерчант
получает сумму за~вычетом interchange и~эквайерского processing fee.
Другие~--- net~с~детальной расшифровкой: мерчант получает один перевод
и~подробный settlement-отчёт, в~котором каждая строка~--- отдельная
транзакция или~fee. Без~этого понимания сверка превращается в~поиск
несуществующих расхождений.

\begin{practice}
  Типичная ошибка при~первичной настройке reconciliation~--- сравнивать
  сумму транзакций из~авторизационного лога с~суммой bank statement.
  Это~неправильные источники для~сравнения: авторизации включают pre-auth,
  которые не~завершились клирингом; bank statement отражает net settlement
  после всех вычетов. Правильные пары для~сравнения: авторизации~vs~TC33/IPM
  (проверяем, всё~ли прошло в~клиринг), TC33/IPM~vs~settlement позиция
  (проверяем расчёт interchange), settlement позиция~vs~bank statement
  (проверяем реальное движение денег).
\end{practice}


\section{Exception handling}
\label{sec:recon-exceptions}

В~идеальном мире каждая авторизация имеет клиринговую запись, каждая
клиринговая запись~--- соответствующую авторизацию, и~все суммы
совпадают в~пределах допустимых дельт. В~реальности исключения составляют
от~0,1\%~до~нескольких процентов объёма~--- и~их~обработка~--- это~отдельный
операционный процесс.

Первый тип исключений~--- amount mismatch: клиринговая сумма отличается
от~авторизованной более чем~на~допустимый лимит. Для~ресторанного tips-сценария допустимое
превышение регулируется правилами схем (обычно 20\%~или~фиксированный
порог)~\sidecite{visa-core-rules}. Превышение сверх порога~--- основание
для~disp~12.5 (Incorrect Amount). Для~reconciliation: такие записи
помечаются как~исключения и~требуют ручной верификации или~автоматического
правила одобрения.

Второй~--- unmatched клиринговая запись, то~есть клиринг без~авторизации.
Она~возникает в~нескольких сценариях: force post (мерчант принудительно
провёл транзакцию без~авторизации или~с~ответом \enquote{одобрить
  офлайн}), голос-авторизация по~телефону~--- авторизация существовала,
но~не~была записана в~систему, или~технический сбой~--- авторизация
прошла через резервный канал, не~отражённый в~основной системе.
Unmatched клиринговая запись~--- потенциальный fraud-сигнал и~требует
расследования.

Зеркальная ситуация~--- unmatched авторизация, когда авторизация остаётся без~клиринга. Мерчант
авторизовал транзакцию, но~не~представил её~к~клирингу в~срок~--- либо
передумал, либо технический сбой. В~результате у~держателя карты висит
hold на~сумму авторизации, деньги не~списались, и~он~может жаловаться
или~не~понимать свой баланс. Для~мерчанта~--- упущенная выручка; для~reconciliation~---
повисший открытый авторизационный записи, требующий закрытия (reversal
или~истечение авторизационного окна).

Наконец, late presentment~--- мерчант представил транзакцию к~клирингу
после установленного срока, как~правило позже 30~дней от~даты
авторизации. Схема может отклонить late presentment или~принять с~понижением
interchange-категории; эмитент вправе инициировать chargeback 12.1. Для~reconciliation~---
такие записи требуют либо поглощения убытка, либо работы с~эквайером
по~исключению~\sidecite{visa-dispute-rules}.

\section{DCC и~мультивалютный settlement}
\label{sec:recon-dcc}

Когда транзакция проводится в~одной валюте, а~settlement происходит
в~другой, возникает вопрос: кто~конвертирует и~по~какому курсу. Этот
вопрос определяет, кто~именно потеряет на~курсовой разнице.

По~умолчанию конверсию выполняет схема: \scheme{Visa}
и~\scheme{Mastercard} конвертируют сумму транзакции в~settlement currency
по~собственному обменному курсу. Курс схемы обычно близок к~межбанковскому
с~добавлением небольшой маржи; для~мерчанта это~прозрачно~--- он~видит
сумму в~settlement currency в~TC33/IPM~\sidecite{visa-core-rules}.

Альтернативный механизм~--- DCC (Dynamic Currency Conversion). Это~сервис, позволяющий
мерчанту (или~эквайеру) предложить держателю карты оплатить в~его~домашней
валюте прямо на~кассе: сумма конвертируется в~валюту карты по~курсу
мерчанта, а~держатель видит итоговую сумму в~своих деньгах до~оплаты.
Мерчант при~этом зарабатывает на~марже обменного курса~\sidecite{visa-core-rules}\sidecite{mc-rules}.

Для~reconciliation DCC создаёт дополнительный уровень сложности: в~клиринговом
файле присутствуют два поля~--- сумма в~валюте транзакции (валюта карты
при~DCC) и~сумма в~settlement currency. При~DCC валюта транзакции и~settlement
currency совпадают~--- это~указание на~то, что~конверсия уже произошла.
Reconciliation-система должна уметь идентифицировать DCC-транзакции
и~корректно их~обрабатывать, не~пытаясь повторно конвертировать.

Для~мерчантов с~мультивалютным потоком (принимают оплату в~нескольких
валютах, settlement в~одной или~нескольких) вопрос курса конверсии
напрямую влияет на~P\&L. Если settlement currency фиксирована, а~транзакции~---
в~разных валютах, мерчант несёт FX-риск: курс схемы на~дату settlement
может отличаться от~курса на~дату транзакции. Управление этим риском~---
либо через multi-currency settlement (отдельный счёт в~каждой значимой
валюте, без~конверсии схемой), либо через хеджирование~--- выходит
за~рамки этой книги, но~reconciliation должна это~разделение
поддерживать.

Подробнее о~FX-риске при~разных моделях settlement~--- в~главе~\ref{ch:multicurrency}.


\section{Автоматизация сверки}
\label{sec:recon-automation}

Ручная сверка в~Excel была стандартом пятнадцать лет назад. Сегодня
она~встречается только у~микромерчантов с~десятками транзакций в~день:
при~тысячах и~десятках тысяч транзакций ручная работа означает либо
вечные задержки, либо постоянный штат финансовых аналитиков, либо
оба последствия одновременно.

Автоматизация reconciliation строится вокруг \textbf{matcher engine}~---
компонента, который сопоставляет авторизации с~клиринговыми записями
по~набору ключей (STAN, Terminal~ID, сумма, дата с~допуском). Matcher
работает в~несколько проходов: строгое совпадение по~всем ключам → нечёткое
по~подмножеству ключей → fuzzy по~сумме с~допуском → остаток попадает
в~exception queue. Exception queue~--- это~список записей, не~прошедших
автоматическую сверку; по~нему работают аналитики.

Строить собственный matcher engine имеет смысл при~наличии прямого
подключения к~схемам и~специфических требований: нестандартные правила
нетирования, интеграция с~собственной ERP, сложный мультивалютный поток.
В~этом случае matcher engine~--- несложный компонент, но~вокруг него
выстраивается операционная инфраструктура: хранилище, audit trail,
workflow для~исключений, уведомления.

Покупать готовый reconciliation SaaS разумно
для~мерчантов и~PSP без~прямого подключения к~схемам: сервис получает
данные напрямую от~вашего PSP или~эквайера и~предоставляет уже
reconciled~отчёт. Критерии выбора: поддерживаемые форматы (TC33, IPM,
форматы вашего эквайера), объём транзакций в~тарифном плане, SLA
на~доступность данных, наличие API для~интеграции с~вашей ERP.

Отдельная тема~--- observability reconciliation engine, то~есть метрики, без~которых
неясно, работает~ли сервис корректно: доля matched-записей за~день
(падение~--- сигнал проблемы с~парсером или~форматом файла); время
до~полной сверки дня (SLA: день~X должен быть reconciled к~утру дня~X+1);
объём exception queue (стабильно высокий~--- сигнал системной
проблемы, а~не~операционной нагрузки).

\begin{practice}
  Наиболее частая техническая проблема reconciliation~--- неправильная
  обработка дней перехода между финансовыми датами. Settlement date
  у~схем~--- не~calendar date, а~processing date, которая зависит от~времени
  отсечки (cutoff) для~данного эквайера. Транзакции, прошедшие после
  cutoff в~18:00 в~пятницу, попадут в~settlement следующего рабочего дня~---
  часто понедельника. Если ваша система жёстко привязывает транзакции
  к~calendar date, вы~будете иметь регулярные расхождения в~конце
  недели и~перед праздниками.
\end{practice}


\section*{Итоги главы}

\begin{itemize}
  \item Reconciliation закрывает три разрыва: авторизация~vs~клиринг,
        клиринг~vs~settlement-позиция, settlement-позиция~vs~банковская
        выписка; сравнивать несмежные уровни~--- ошибка.
  \item TC33~(Visa) и~IPM~(Mastercard) несут разные форматы, но~схожие
        данные; поле IRD в~TC33 и~PDS в~IPM~--- ключи к~расчёту
        interchange; полные структуры~--- в~онлайн-приложении.
  \item Net settlement~--- итоговая нетированная позиция за~период;
        сумма bank statement не~равна сумме транзакций из-за~interchange,
        сборов и~возвратов.
  \item Основные исключения: amount mismatch (суммы вышли за~допустимый
        delta), unmatched clearing (force post или~fraud), unmatched
        authorization (не~представлена к~клирингу), late presentment
        (штрафные последствия по~правилам схем).
  \item DCC создаёт двойное поле суммы в~клиринге: reconciliation
        должна идентифицировать DCC-транзакции и~не~конвертировать
        повторно; мультивалютный settlement требует отдельного учёта
        FX-риска.
  \item Matcher engine + exception queue + observability~--- три
        компонента любой reconciliation-системы, независимо от~того,
        строится~ли она самостоятельно или~покупается как~SaaS.
\end{itemize}
