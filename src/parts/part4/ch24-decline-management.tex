\chapter{Decline management}
\label{ch:decline-management}

\begin{kaobox}[title=Что вы узнаете из этой главы]
  Каждый отказ авторизации~--- это потерянная выручка. Из~этой главы вы~узнаете,
  чем мягкий отказ отличается от~жёсткого и~почему это~различие определяет
  всю логику повторных попыток; что~на~самом деле означает response code~05
  и~почему один и~тот~же код скрывает десятки причин; как~правильно
  выстроить retry logic, не~нарушив правила схем; почему сетевой токен
  добавляет~2--3~процентных пункта к~authorization rate; и~как~acquirer
  routing становится инструментом роста конверсии.
\end{kaobox}

Каждый decline~--- это решение, принятое в~условиях неполной
информации. Мерчант видит только код ответа и~потерянный checkout,
эмитент~--- внутренние правила и~риск-профиль, а~платёжная команда
должна решить, стоит~ли повторять попытку, менять канал
или~остановиться.


\section{Анатомия отказа}
\label{sec:decline-anatomy}

Не~все отказы одинаково полезны. Это~звучит как~каламбур, но~за~ним стоит
принципиальное разделение, от~которого зависит вся операционная логика
работы с~declined транзакциями. Повторная попытка при~неправильно
классифицированном отказе~--- не~просто бессмысленная трата ресурсов:
она~может нарушить правила схем и~привлечь штрафы.

Мягкий отказ (soft decline) означает, что~транзакция отклонена
по~временно́й или~устранимой причине. Эмитент фактически говорит:
\enquote{не~сейчас, но~попробуй ещё}. Response codes, которые традиционно
относят к~мягким: \rc{51}~--- Insufficient~Funds (недостаточно средств),
\rc{05}~--- Do~Not~Honor в~ряде контекстов, \rc{61}~--- Exceeds Withdrawal
Amount Limit, \rc{62}~--- Restricted Card (временное ограничение),
\rc{65}~--- Exceeds Withdrawal Frequency Limit. Для~всех~них повторная
попытка через некоторое время или~через другой канал имеет ненулевые
шансы на~успех.

Зеркальная ситуация~--- жёсткий отказ (hard decline): транзакция отклонена
по~постоянной причине, и~повторная попытка с~теми~же реквизитами
бессмысленна. Response codes жёсткого отказа: \rc{04}~--- Pick~Up
(карта заблокирована, предписано изъять), \rc{14}~--- Invalid Account Number
(PAN не~существует), \rc{41}~--- Lost~Card, \rc{43}~--- Stolen~Card,
\rc{54}~--- Expired~Card. Автоматически повторять транзакцию при~этих
кодах~--- значит попусту нагружать сети и~раздражать клиента, которому
нужна другая карта.

Между ними находится промежуточный класс~--- коды, требующие участия человека
или~смены канала. \rc{01}~--- Refer to Card~Issuer: эмитент хочет
верифицировать транзакцию через иной канал; обычно это~значит, что~клиенту
следует позвонить в~банк. \rc{91}~--- Issuer Unavailable: технический
сбой на~стороне эмитента, не~связанный с~конкретной картой; повторная
попытка через несколько минут или~часов разумна. \rc{96}~---
System Malfunction: схожая техническая ситуация. Разумная стратегия
для~промежуточного класса~--- один немедленный retry, затем логика
с~задержкой~\sidecite{visa-core-rules}\sidecite{mc-rules}.

Итак, первая задача decline management~--- не~\enquote{отправить
retry}, а~правильно классифицировать отказ по~типу обратимости
и~допустимого действия. Только после этого имеет смысл читать
сами response codes и~строить логику вокруг них.

\section{Что реально означают response codes}
\label{sec:decline-response-codes}

Response code~--- это~ответ эмитента, но~не~его~объяснение. За~одним
кодом могут стоять десятки различных причин, и~понять истинную из~них
без~дополнительных данных невозможно. Рассмотрим наиболее распространённые
коды с~точки зрения того, что~за~ними реально происходит.

\rc{05} Do~Not~Honor~--- самый распространённый и~при~этом
самый неинформативный код. Он~означает буквально \enquote{не~проводи},
и~больше ничего. Реальные причины, которые эмитент может скрывать
за~этим кодом: сработал правила антифрода (transaction scoring), сработал
дневной лимит по~сумме или~числу операций, транзакция не~соответствует
поведенческому профилю клиента (необычная страна, необычная категория
мерчанта, необычное время), карта временно заблокирована клиентом
через мобильное приложение, или~у~эмитента внутренняя политика,
запрещающая данный MCC. Из~всех этих причин некоторые~--- временные
и~устранимые (антифрод может переобучиться, лимит~--- сброситься), а~другие~---
постоянные (запрет категории). Без~доступа к~системам эмитента отличить
их~невозможно, поэтому \rc{05} обычно трактуется как~мягкий отказ
с~ограниченным числом допустимых retry.

Столь~же обманчив \rc{51} Insufficient~Funds, который вводит в~заблуждение своим буквальным
смыслом. Для~дебетовых карт он~действительно означает отсутствие средств
на~счёте. Но~для~кредитных карт~--- это~превышение кредитного лимита
или~временное превышение лимита авторизаций, которое никак не~связано
с~реальным балансом. Кроме того, ряд эмитентов использует \rc{51}
как~универсальный код мягкого отказа вместо более специфичных: так~короче
объяснять клиенту \enquote{денег не~хватает}, даже если реальная причина~---
временный hold или~pending-транзакция, ещё~не~списанная со~счёта.

Код \rc{54} Expired~Card встречается чаще, чем~следовало~бы. Карта
может быть физически перевыпущена с~новой датой истечения, но~если
Account Updater (VAU/ABU) ещё~не~обновил реквизиты у~мерчанта, мерчант
продолжает отправлять старую дату~\sidecite{visa-core-rules}. Кроме того,
некоторые эмитенты возвращают \rc{54} при~попытке использовать карту
после того, как~клиент заблокировал её~в~приложении~--- формально карта
не~истекла, но~эмитент выбирает этот код как~\enquote{карта недействительна}.

Совершенно иную природу имеет \rc{91} Issuer~Unavailable~--- технический код, не~связанный
с~состоянием карты или~счёта. Эмитент недоступен: его~авторизационный
хост не~отвечает в~установленный таймаут, дата-центр переключается на~резерв,
или~схема не~может маршрутизировать запрос. Транзакцию следует повторить:
через несколько минут ситуация, скорее всего, нормализуется. Мерчантам
с~высокой долей \rc{91} среди отказов стоит проверить, не~работают~ли
они преимущественно с~одним эквайером, который маршрутизирует через
один банк-эмитент с~нестабильной доступностью.

\begin{practice}
  Хороший мониторинг decline должен сегментировать response codes не~по~двум
  (hard/soft), а~по~трём-четырём группам: жёсткие (не~повторять никогда),
  мягкие финансовые (повторять с~задержкой), мягкие поведенческие
  (повторять через другой канал~--- токен вместо PAN, другой эквайер),
  технические (повторять немедленно). Aggregate dashboard с~одним числом
  \enquote{decline rate} скрывает эту структуру: 15\%~decline, половина из~которых~---
  технические коды, означает совершенно другую проблему, чем~15\%~decline
  с~половиной hard-кодов.
\end{practice}


\section{Retry logic: тайминг, частота, каналы}
\label{sec:decline-retry}

Правила retry~--- одна из~областей, где интересы мерчанта (максимизировать
конверсию) и~правила схем (не~перегружать авторизационные сети) расходятся.
Понимание ограничений схем~--- не~академическая задача: нарушение правил
retry влечёт штрафы.

Немедленный retry оправдан только при~технических сбоях: \rc{91},
\rc{96}, timeout без~ответа. В~этих случаях одна немедленная повторная
попытка разумна; если и~она не~прошла~--- переходи к~отложенной логике.
Немедленный retry при~\rc{05} или~\rc{51}~--- бессмысленен и~может
нарушить ограничения схем.

Для~мягких финансовых кодов (\rc{51}, \rc{61}) применяется отложенный retry,
который должен давать эмитенту время на~изменение ситуации. Практически разумный
минимум~--- 24~часа до~первой повторной попытки, затем 48--72~часа между
последующими. Для~подписочных MIT-транзакций схемные правила определяют
конкретный retry protocol: нарушение этого протокола классифицируется
как~превышение допустимого числа попыток~\sidecite{visa-retry-rules}\sidecite{mc-retry-rules}.

Отдельно регулируется предельное число попыток. \scheme{Visa} ввела Retry Transaction
Rules в~октябре 2019~года, установив лимит на~число повторных попыток
для~declined транзакций~\sidecite{visa-retry-rules}. Для~жёстких
отказов~--- ноль: повторять запрещено вообще. Для~мягких~--- ограниченное
число попыток в~течение 30~дней, точные значения которого зависят от~response
code~\sidecite{visa-retry-rules}\todocite{актуальные лимиты retry по~каждому response code Visa и~Mastercard~--- см.\ онлайн-приложение}.
\scheme{Mastercard} ввела аналогичные ограничения~--- Excessive Attempts
Rules~--- с~собственными порогами~\sidecite{mc-retry-rules}. Превышение
лимитов~--- прямой путь в~мониторинговую программу.

Наконец, эффективный инструмент~--- смена канала при~retry, не~связанная
с~тайм-слотами. Если транзакция упала с~кодом, свидетельствующим о~поведенческом
антифроде (\rc{05} при~подозрительном паттерне), повторная попытка с~теми~же
реквизитами пройдёт тот~же фильтр с~тем~же результатом. Но~та~же транзакция
через другой канал может помочь: через сетевой токен вместо PAN~---
токенизированный flow имеет иной trust level у~эмитента; через другого
эквайера~--- маршрутизация влияет на~то, какой авторизационный профиль
видит эмитент. Эти~приёмы рассмотрены в~следующих секциях.

\section{Network token vs PAN: +2--3 п.п. к~authorization rate}
\label{sec:decline-tokens}

Когда мерчант отправляет авторизационный запрос с~реальным PAN клиента,
эмитент видит транзакцию без~дополнительного контекста о~том, кто инициирует
запрос и~прошла~ли карта через дополнительную верификацию. Когда мерчант
использует сетевой токен (network token от~VTS или~MDES), к~авторизационному
запросу добавляется информация об~уровне верификации токена~---
Cryptogram и~Token Assurance Level~--- которая сигнализирует эмитенту:
этот запрос прошёл через инфраструктуру схемы и~привязан к~конкретному
мерчанту и~домену~\sidecite{visa-vts}\sidecite{mc-mdes}.

Практический эффект~--- устойчивый прирост authorization rate на~2--3
процентных пункта для~мерчантов, перешедших с~PAN на~сетевые
токены~\sidecite{visa-vts}\todocite{данные Visa и~Mastercard о~приросте authorization rate при~переходе на~сетевые токены}.
Механизм объяснения прост: эмитент получает более богатый контекст транзакции
и~может применить более точную оценку риска вместо консервативного отказа
при~неопределённости. Токенизированный запрос~--- это доверенный запрос
от~известного домена.

Дополнительный бонус сетевого токена~--- автоматическое обновление
реквизитов при~перевыпуске карты. Когда эмитент перевыпускает карту
с~новым PAN или~новой датой истечения, токен остаётся действующим:
схема обновляет маппинг токена на~новый PAN прозрачно для~мерчанта.
Это~устраняет целый класс \rc{54} Expired~Card и~технического churn
в~подписках~--- тот~же эффект, который дают Account Updater сервисы~(VAU/ABU),
но~без~необходимости отдельного batch-запроса.

Почему не~все мерчанты переходят на~токены? Основных препятствий три.
Первое~--- техническая интеграция: прямое подключение к~VTS или~MDES
требует сертификации и~операционных ресурсов; для~мерчантов без~прямого
процессинга переход зависит от~готовности их~PSP. Второе~--- PCI~scope:
мерчант на~PAN до~сих пор мог работать в~SAQ~A, и~миграция на~токены
через PSP не~меняет это~положение, а~вот самостоятельное управление
токенами добавляет к~инфраструктуре новый компонент. Третье~---
инерция: интеграция \enquote{работает}, и~до~конкретной боли (высокий
decline rate или~churn) стимула переключаться мало.

Подробнее об~архитектуре сетевых токенов~--- в~главе~\ref{ch:tokenization}.


\section{Acquirer routing как~инструмент конверсии}
\label{sec:decline-routing}

Один и~тот~же PAN, одна и~та~же карта, одна и~та~же сумма~--- но~authorization
rate может отличаться на~несколько процентных пунктов в~зависимости
от~того, через какого эквайера проходит транзакция. Это~не~случайность
и~не~ошибка: это~следствие нескольких факторов, поддающихся управлению.

Первый фактор~--- BIN-аффинность: у~некоторых BIN-диапазонов выше исторический
approval rate при~маршрутизации через конкретного эквайера или~конкретный
регион. Причины могут быть разными: эмитент настроил дополнительное
доверие к~определённому acquirer~BIN, или~исторически между этим эквайером
и~эмитентом не~было инцидентов с~высоким fraud rate. Данные о~BIN-аффинности
можно собрать из~собственной статистики, накапливая пары
(BIN, эквайер)~→~approval~rate.

Второй приём~--- cascade routing. Классический паттерн: первая попытка через
основного эквайера; если получен soft decline~--- вторая попытка через
резервного. Резервный эквайер может иметь другой маршрут до~эмитента,
другой профиль доверия, другой географический endpoint сети~\scheme{Visa}
или~\scheme{Mastercard}. На~практике cascade даёт от~0,5~до~2~п.п.
прироста approval rate~--- цифра скромная в~абсолютных единицах, но~значимая
при~больших объёмах~\sidecite{visa-core-rules}.

При~этом важно помнить об~ограничениях схем: routing с~целью уклонения
от~interchange-ставок запрещён правилами схем. Если мерчант маршрутизирует
транзакции через эквайера в~другой юрисдикции, чтобы применить более
низкий interchange~--- это~нарушение, и~схема вправе потребовать
корректировки~\sidecite{visa-core-rules}\sidecite{mc-rules}. Легитимная
цель routing~--- оптимизация approval rate и~latency, а~не~структуры
interchange-сборов.

Все эти приёмы объединяет smart routing~--- движок, который сочетает BIN-аффинность, cascade и~cost-модель:
для~каждой транзакции он~выбирает эквайера, максимизируя
ожидаемый approval rate с~учётом стоимости. Такие движки реализуют
крупные PSP и~платёжные оркестраторы~--- и~именно это~является одним
из~главных их~преимуществ для~мерчанта по~сравнению с~прямым подключением
к~одному эквайеру.


\section{Authorization rate benchmarks по~вертикалям}
\label{sec:decline-benchmarks}

Authorization rate~--- метрика, которую сложно интерпретировать без~контекста:
85\%~--- это~хорошо или~плохо? Ответ полностью зависит от~вертикали,
географии, типа карты и~модели оплаты.

Для~корректного расчёта AR нужно чётко определить знаменатель. Если включать
в~него все транзакции~--- авторизационные попытки включая технические таймауты
и~заведомо мусорные запросы,~--- получится одна цифра. Если брать только
транзакции с~валидным PAN и~ненулевой суммой~--- другая. Индустриальный
стандарт: AR~= одобренные / (одобренные~+ отклонённые), где~отклонённые~---
это~только response codes с~ответом эмитента, не~технические ошибки
и~не~таймауты. Именно в~этой метрике публикуют benchmarks~\scheme{Visa}
и~\scheme{Mastercard}.

Для~e-commerce retail (товары широкого потребления, средний чек
30--150~USD) типичный AR~--- 85--92\%~в~зависимости от~географии и~типа
карты~\todocite{данные Visa/Mastercard о~средних AR по~вертикалям; industry benchmarks}.
Подписки демонстрируют более высокий AR при~первоначальной
CIT-транзакции (с~3DS)~--- 88--94\%~--- но~могут иметь заметный decline
при~MIT, особенно при~неправильно выстроенном retry~protocol или~без~Account
Updater. В~travel-вертикали (авиабилеты, отели) AR исторически ниже~---
80--87\%~--- отчасти потому, что~суммы велики, транзакция нестандартная
по~паттерну (редкая, высокобилетная), и~эмитенты применяют более консервативный
scoring~\todocite{benchmarks AR для~travel вертикали}. Ещё~ниже показатели
в~gaming и~gambling~--- категориях с~повышенным fraud risk, где часть
эмитентов применяет специальные правила по~MCC~7993/7994/7995~--- AR
в~диапазоне 70--82\%~\todocite{AR benchmarks для~gaming и~gambling}.

Для~российского рынка данные публично доступны в~ограниченном виде.
Транзакции в~рублях через российских эмитентов исторически имеют более
высокий AR, чем~трансграничные, поскольку нет~FX-риска и~эмитент лучше
знает поведенческий паттерн клиента. СБП-транзакции не~знают понятия AR
в~карточном смысле: платёж либо исполнен мгновенно, либо ошибка~---
промежуточного состояния \enquote{одобрено, но~может не~пройти} нет.

\begin{practice}
  Измеряйте AR раздельно по~сегментам: новые клиенты vs~вернувшиеся,
  карты с~3DS vs~без, PAN vs~токен, своя страна vs~трансграничные. Агрегированный
  AR~--- почти бесполезная метрика для~оптимизации. Сегментация немедленно
  показывает, где сосредоточены отказы: как~правило, 60--70\%~всех decline
  приходится на~10--15\%~транзакций с~конкретными характеристиками
  (нестандартная география, нестандартный MCC, новая карта, отсутствие
  3DS), и~именно на~них следует направить усилия.
\end{practice}


\section*{Итоги главы}

\begin{itemize}
  \item Мягкий отказ (soft decline) поддаётся повторной попытке,
        жёсткий (hard decline)~--- нет; промежуточные технические коды
        (\rc{91}, \rc{96}) требуют немедленного retry. Правильная
        классификация~--- основа retry logic.
  \item \rc{05} Do~Not~Honor скрывает десятки причин: антифрод,
        лимиты, поведенческий паттерн, временная блокировка; \rc{51}
        для~кредитных карт~--- не~всегда реальная нехватка средств;
        \rc{54}~--- не~всегда истёкший срок карты.
  \item Правила Visa и~Mastercard ограничивают число допустимых
        retry-попыток для~мягких отказов: нарушение~--- прямой путь
        в~мониторинговую программу; актуальные лимиты~--- в~онлайн-приложении.
  \item Сетевой токен добавляет~2--3~п.п. к~authorization rate
        за~счёт дополнительного контекста доверия для~эмитента
        и~автоматически решает проблему перевыпуска карты без~Account
        Updater.
  \item Acquirer routing~--- BIN-аффинность, cascade, smart routing~---
        повышает AR без~изменения логики самой транзакции; routing
        с~целью уклонения от~interchange запрещён правилами схем.
  \item Authorization rate имеет смысл только в~разрезе сегментов:
        вертикаль, тип карты, 3DS/no-3DS, PAN/токен; агрегированный
        AR скрывает реальные проблемы.
\end{itemize}
