\addchap{Карта первоисточников}\label{ch:primary-sources}
Когда в~платёжной команде звучит фраза \enquote{надо проверить
в~правилах}, спор часто разгорается вокруг источника истины. Один человек
открывает свод правил сети, другой ищет ответ
в~технической спецификации, третий пересказывает заметку поставщика.
Одну и~ту же тему действительно могут описывать три класса
документов: публичные правила для участников, технические
спецификации и закрытые оперативные уведомления.

Отсюда первая задача навигации: прежде чем читать,
понять, какой документ в~этой теме главный, а~какой~--- только
комментарий, пересказ или вспомогательный портал интеграции. Эта
глава помогает быстрее выбрать
правильную точку входа в~первоисточники.

Ниже разобран каждый корпус документов: с~чего начинать
и~как не перепутать правило участника с~протоколом
или вспомогательным порталом.

\section*{EMVCo}\label{sec:ps-emvco}

EMVCo нужен там, где спор идёт не о~договорных обязанностях,
а~о~поведении платёжной техники. Если вопрос звучит как
\enquote{что именно должны сделать терминал, карта, токен
или мобильный кошелёк}, искать ответ нужно прежде всего здесь.
Ни правила эквайера, ни правила карточной сети эти спецификации
не заменят.

\subsection*{Главные документы}

Если речь идёт о~контактной EMV-транзакции, точкой входа остаются
\textbf{EMV Chip Specifications}, Books 1--4
(см. главу~\ref{ch:emv})~\cite{emvco-specs}. Для бесконтактной оплаты
нужен корпус
\textbf{EMV Contactless Specifications}, где описаны NFC
и~логика бесконтактного ядра
(см. главу~\ref{ch:contactless})~\cite{emvco-contactless-chip}.
Вопросы токенизации закрывает
\textbf{EMV Payment Tokenisation Specification}: роли TSP,
ограничения домена токена и~его жизненный цикл
(см. главу~\ref{ch:tokenization})~\cite{emvco-payment-tokenisation}.
Для 3DS основной документ~--- \textbf{EMV 3-D Secure Protocol and Core Functions
Specification}, где описаны сообщения AReq/ARes, сценарий
дополнительной проверки и~общая логика протокола
(см. главу~\ref{ch:3ds})~\cite{emvco-3ds}.

\subsection*{Как входить}

EMVCo тяжелее всего читать подряд. Здесь выгоднее идти
от вопроса к~разделу. Если нужно понять жизненный цикл контактной
операции, полезнее всего Book 3. Если важно поведение терминала,
кассира и интерфейса приёма, чаще открывают Book 4. Если вопрос
относится к~бесконтактному входу в~транзакцию, нужен соответствующий
том Contactless. EMVCo редко читают от корки до корки~--- этот корпус удобнее
как справочник для точечной проверки спорного места.

\section*{PCI SSC}\label{sec:ps-pci}

Стандарты PCI нужны тогда, когда команда спорит о~периметре PCI
и~наборе обязательных контролей. Как ограничить себя SAQ~A, как
контролировать JavaScript на странице оплаты, как защищать
PIN block и~где проводить точку дешифрования: на все эти вопросы
отвечать нужно документами PCI.

\subsection*{Главные документы}

\textbf{PCI DSS} остаётся главным документом по контролям среды,
где появляются данные держателя карты
(см. главу~\ref{ch:pci-dss})~\cite{pci-dss-4}. \textbf{SAQ} нужны как
сокращённые
режимы оценки для разных моделей интеграции~\cite{pci-saq}.
\textbf{PCI PIN Security Requirements} регулируют работу
с PIN и HSM~\cite{pci-pin-security}. \textbf{PCI P2PE} нужен тогда, когда вопрос
упирается в~шифрование от терминала до точки дешифрования~\cite{pci-p2pe}.

\subsection*{Как входить}

Здесь практический порядок чтения один: сначала
определить периметр PCI, потом понять, какой путь оценки
применим, и~затем разбирать отдельные требования. Ошибка
в~определении периметра обычно обходится дороже, чем неверное
чтение одного подпункта.

\section*{Visa}\label{sec:ps-visa}

У Visa легко перепутать две вещи: правила
для участников и техническую документацию VisaNet. Первое
отвечает на вопрос \enquote{что разрешено и кто обязан}, второе --
на вопрос \enquote{как это устроено в~сообщениях, терминалах
и сервисах сети}.

\subsection*{Главные документы}

\textbf{Visa Core Rules and Visa Product and Service Rules}
нужны, когда речь идёт об обязанностях эмитента, эквайера
и~мерчанта~\cite{visa-core-rules}. \textbf{Dispute Management
Guidelines for Visa Merchants}
дают публичную точку входа в~диспутную логику; при расхождении
между ними приоритет остаётся за Core
Rules~\cite{visa-dispute-guidelines-merchants}.
\textbf{Visa Technical Acceptance Device Guide (TADG)}
описывает приём карт на терминалах~\cite{visa-tadg}. Для авторизации и клиринга
в VisaNet основной ориентир -- материалы \textbf{BASE I / BASE II};
публичные обзоры по ним у Visa есть~\cite{visa-visanet-booklet},
но рабочая техническая
детализация обычно доступна только участникам сети.
\textbf{VTS documentation} описывает Visa Token Service~\cite{visa-vts}.

\subsection*{В~каком порядке читать}

Если спор касается ролей участников или допустимости сценария,
начинать нужно с Core Rules. Если вопрос в~поведении терминала
или в~техническом поле на приёме, нужен TADG или закрытая
техническая спецификация через участника сети. Портал
для разработчиков~\cite{visa-developer-portal} полезен
при работе с~API; нормативные вопросы решаются по Core Rules.

\section*{Mastercard}\label{sec:ps-mastercard}

У Mastercard композиция похожа, но названия документов
другие: отдельно живут правила и диспутная политика, отдельно идут
технические материалы Banknet/IPM и сервисная документация
для разработчиков.

\subsection*{Главные документы}

\textbf{Mastercard Rules} задают обязанности участников~\cite{mc-rules}.
\textbf{Mastercard Chargeback Guide} нужен для логики
диспутов~\cite{mc-chargeback-guide}. \textbf{Transaction Processing Rules}
и связанные технические материалы Banknet/IPM~--- точка
входа в~авторизацию и клиринг по Banknet/IPM~\cite{mc-tpr,mc-switching-services}.
\textbf{MDES} описывает токенизацию Mastercard~\cite{mc-mdes-token-connect}.
\textbf{M/Chip Requirements} нужны, когда важно понять
Mastercard-специфику EMV~\cite{mc-mchip}.

\subsection*{В~каком порядке читать}

Если вопрос договорный или операционный, обычно начинают
с Rules. Если нужен протокол или клиринговая модель, переходят
к~материалам Banknet и IPM. Портал для разработчиков Mastercard
полезен при работе с~открытыми API; закрытые технические
документы получают через участника сети.

\section*{НСПК и~Мир}\label{sec:ps-nspk}

В~местной платёжной системе трудность другая: распределение ролей
и продуктов частично видно в~публичных правилах, а значительная
часть технических деталей скрыта в~документах для участников.
Поэтому здесь нужно заранее определить, ищете вы правовое
поле, продуктовую модель или конкретный формат сообщения.

\subsection*{Главные документы}

\textbf{Правила платёжной системы Мир} задают роли и обязанности
участников~\cite{nspk-mir-rules}. \textbf{Спецификация бесконтактных
платежей Мир}
описывает приём~\cite{nspk-mir-contactless}.
\textbf{Программное обеспечение НСПК} охватывает мобильные решения
и терминальные технологии~\cite{nspk-tsp}; подробные спецификации
форматов сообщений и процедуры подключения НСПК предоставляет
участникам через закрытый портал поддержки. Для
СБП публичные точки входа~--- сайт СБП~\cite{sbp-main} и нормативные
документы Банка России по подключению~\cite{cbr-sbp-actions};
подробные технические требования и тестовые материалы НСПК выдаёт
через портал поддержки участникам и банкам-кандидатам.

\subsection*{Как входить}

Если вопрос касается роли участника, продуктовой схемы или общего
режима работы, обычно хватает публичных правил. Если же спор
упирается в~формат сообщения, сертификацию или детали
внутрироссийского процессинга, понадобится
закрытая документация через участника системы или портал поддержки
НСПК.

\section*{Банк России}\label{sec:ps-cbr}

Документы Банка России нужны тогда, когда требуется понять рамки допустимого поведения.
Если вопрос звучит как \enquote{обязаны ли мы},
\enquote{имеем ли мы право} или \enquote{какой статус нужен},
переходить следует к~нормативным актам ЦБ и федеральным законам.

\subsection*{Главные документы}

161-ФЗ задаёт базовые роли платёжного рынка~\cite{fz-161}.
762-П описывает правила перевода денежных средств~\cite{cbr-762p}.
821-П и 851-П фиксируют текущие требования по защите
информации~\cite{cbr-821p,cbr-851p}. 266-П удерживает карточную
специфику~\cite{cbr-266p}; на смену готовится новый акт
Банка России (опубликован проект Указания), точные реквизиты
которого нужно сверить перед публикацией книги.
Для цифрового рубля, СБП и~других специальных тем набор документов
будет своим, но логика поиска не меняется: начинать нужно
с~нормативного текста, а~пересказ использовать только
как вспомогательный источник. Для цифрового рубля отправная точка~---
340-ФЗ~\cite{fz-340}; требования к~защите информации для участников
платформы установлены 833-П~\cite{cbr-833p}.

\subsection*{Как входить}

Нормативные акты полезно читать связкой: закон задаёт роли
и~пределы модели, положение описывает операционную логику,
письма и разъяснения показывают регуляторскую интерпретацию
спорных мест. Для практической работы удобнее пользоваться правовыми
системами, где доступны история изменений и перекрёстные ссылки.

\section*{ISO}\label{sec:ps-iso}

ISO нужен тогда, когда требуется обобщённый язык, более широкий,
чем диалект конкретной сети. Это уровень стандартов, на котором платёжная
индустрия договаривается о~структуре сообщений и терминов;
в~реальной системе его конкретизирует сетевой диалект.

\subsection*{Главные документы}

ISO~8583 остаётся базовым стандартом по структуре финансовых
сообщений~\cite{iso-8583-2023}, хотя на практике его
конкретизируют сетевые
реализации. ISO~20022~\cite{iso-20022} задаёт современную модель финансовых
сообщений и особенно важен для трансграничных систем, SWIFT MX
и~многих систем мгновенных платежей
(см. раздел~\ref{sec:mc-correspondent}). ISO~4217~\cite{iso-4217},
ISO~3166--1~\cite{iso-3166-1}
и ISO~9564--1~\cite{iso-9564}: базовые документы по кодам валют,
кодам стран и~форматам PIN block.

\subsection*{Как входить}

Если задача прикладная и привязана к~конкретной сети, сначала
читают сетевой диалект, а~потом возвращаются к ISO
как к~верхнему уровню терминов и~модели. Если же задача
архитектурная, межсистемная или проектная (новый обмен),
ISO~20022 полезно открывать сразу
как исходную модель сообщения.

\section*{Итоги}
\begin{itemize}
  \item Для каждой темы есть свой главный документ: EMVCo~--- для
    поведения карты, терминала, токена и 3DS, PCI SSC~---
    для контролей, карточные сети~--- для правил участников
    и протоколов своей сети, ЦБ~--- для правового поля, ISO~---
    для языка стандартов.
  \item Открытые и~закрытые документы стоит различать заранее:
    это экономит время и~снимает иллюзию,
    что любой вопрос можно решить одним публичным PDF.
  \item Начинать чтение лучше с~документа, который отвечает
    на ваш тип вопроса: правила участников, протокол,
    набор контролей или норма регулятора.
  \item Порталы для разработчиков полезны для API-интеграции,
    но редко служат главным источником по сетевым правилам.
  \item Хорошая навигация начинается с~карты источников:
    у~кого по этой теме главный документ, где он доступен
    и~в~каком порядке читать.
\end{itemize}
